联网了却访问不了外网？网络分层、代理与 SSH 跳板详解

2026-07-07
9.5k- 37m
课程笔记-学习Linux网络运维

一个非常常见的困惑是：机器明明已经联网了，能访问校内网站、能 ping 通、能上镜像站，可一旦要访问 GitHub、某些境外 API 或克隆一个仓库，就卡住、超时、报错。这篇笔记要把这件事彻底讲清楚：为什么”联网”不等于”什么都能访问”，以及当你有一台可以出网的跳板机时，该如何正确地配置代理和 SSH 跳板。

为了便于阅读，文中所有 IP、主机名、用户名、仓库地址都是示意占位符（采用 RFC 文档保留地址段 192.0.2.x / 198.51.100.x / 203.0.113.x），替换掉了真实环境里的具体信息。把它们换成你自己的实际值即可。

所以你遇到的问题不是”这台机器完全没联网”，而是”这台机器接入了网络，但不同目标、不同端口、不同协议的可达性不同”。最准确的提问方式应该是”这台机器能不能访问 github.com:443“”能不能访问 github.com:22“，而不是笼统地问”有没有网”。

4. IP、网关、DNS、路由分别是什么

IP 是机器在网络里的地址。机器 A 是 192.0.2.10，跳板机 B 是 198.51.100.20。

DNS 把域名翻译成 IP。例如 nslookup github.com 可能得到 github.com -> 20.205.243.166。但 DNS 成功只说明”知道地址”，不说明”能走到这个地址”。

路由与出口策略决定数据包往哪里走。出口设备上的 ACL（访问控制列表）决定哪些 IP 可以出去、哪些目标可以访问、哪些端口可以访问、哪些协议被允许。所以即使 DNS 和网关都正常，也可能出现”校内某站 443 可访问、github.com:443 不可访问、github.com:22 可访问”这种看似矛盾的现象。这正是因为限制是按目标 IP、目标端口、协议分别施加的。

5. 为什么校内能通，GitHub HTTPS 不通

机器 A 可以访问 https://www.<学校>.edu.cn、清华镜像、中科大镜像、npm 镜像，这些资源要么在校内，要么在教育网/国内网络环境中，路径更近、策略更宽松。但访问 https://github.com 超时，说明 github.com:443 这条路径或这个端口没有打通。这不是 DNS 问题，也不是完全没网，而是对特定目标和端口的访问受限。

6. ping、curl、git clone 分别说明什么

这三个工具测试的是完全不同的东西，弄混它们是排错时最常见的误区。

ping 使用 ICMP 协议，它只说明 ICMP 报文能不能来回，不能说明 HTTPS 443 是否通、SSH 22 是否通、git clone 是否能用。所以”ping 通不等于网站可访问，ping 不通也不等于网站不可访问”。

ICMP 全称 Internet Control Message Protocol，主要用于网络控制和诊断（目标是否可达、路由是否有问题、TTL 超时、路径 MTU 发现等）。ping 最常用的是其中的 Echo Request（”你在吗？”）和 Echo Reply（”我在。”）。它和 HTTPS（TCP 443 -> TLS -> HTTP）、SSH（TCP 22 -> SSH）完全不是一套东西，这里没有 TCP 端口的概念。这也解释了一个常见疑惑：为什么开了 SOCKS 代理，ping 还是不通。因为 ssh -D 创建的 SOCKS 代理只代理 TCP 连接，而 ping 用的是 ICMP，SOCKS 没有办法代理它。执行 ping github.com 时，真实路径仍然是本机直连 GitHub 的 ICMP，而不是走 127.0.0.1:1080。

proxy_on 设置的是 ALL_PROXY、HTTP_PROXY、HTTPS_PROXY 这些环境变量，是给支持代理的应用（比如 curl）看的，所以会影响 curl；但 ping 不读取这些 HTTP/SOCKS 代理环境变量，它直接走系统网络栈发 ICMP，所以不受影响。结论是：不要用 ping 来判断 SOCKS 代理是否成功。要测代理后的 HTTPS，应该用 curl -I --socks5-hostname 127.0.0.1:1080 https://github.com；要知道跳板机能不能 ping 某目标，应该直接在跳板机上跑 ssh <跳板机> 'ping -c 4 github.com'。

curl -I https://github.com --connect-timeout 10 -m 20 更接近真实 HTTPS 访问，它依次测试 DNS -> TCP 443 -> TLS -> HTTP。如果返回 HTTP 状态码，通常说明已经到达 HTTP 层；如果连接超时，往往卡在 TCP 连接阶段。

git clone 有两种地址，走的是完全不同的网络路径：git clone https://github.com/owner/repo.git 走 github.com:443 的 HTTPS；git clone git@github.com:owner/repo.git 走 github.com:22 的 SSH。两者不是同一条路径，一个通不代表另一个也通。

7. 核心：TCP、TLS、HTTP、HTTPS、SSH 与端口

这一节是理解整件事的关键。很多困惑都来自一句话：”我不是已经连上网了吗，为什么还是访问不了某个网站？” 原因是，一次网络访问不是一个动作，而是很多层动作叠在一起，任何一层失败，最终看起来都像”访问不了”。

7.1 分层模型

一次访问可以想成从下往上的几层：应用层（HTTP、SSH、Git 协议）、加密层（TLS）、传输层（TCP/UDP）、网络层（IP/路由/网关）、链路层（网卡/网线/交换机/Wi-Fi）。以访问 https://open.bigmodel.cn/api/anthropic 为例，过程大致是：DNS 解析域名、IP 路由决定从哪个网关出去、TCP 连接到 :443、TLS 在 TCP 连接上建立加密通道、HTTP 在 TLS 通道里发送请求、服务器返回响应。简化成一句话就是 DNS -> IP 路由 -> TCP 443 -> TLS -> HTTP。而 GitHub SSH 是 DNS -> IP 路由 -> TCP 22 -> SSH -> Git。所以 HTTPS 和 SSH 从 TCP 端口开始就已经分叉了。

7.2 端口是什么

IP 地址定位到”哪台机器”，端口定位到”这台机器上的哪个服务”。可以类比成一栋楼的地址（IP）和楼里的房间号（端口），进房间后说的语言就是协议。同一个域名可以有多个端口：github.com:22 提供 SSH 服务，github.com:443 提供 HTTPS 服务。所以”能访问 github.com:22“不等于”能访问 github.com:443“，就像同一栋楼的 22 号房间开着，不代表 443 号房间也开着。

7.3 TCP 与连接失败的几种表现

TCP 是传输层协议，负责在两台机器之间建立可靠连接（建立连接、保证顺序、丢包重传、控制速度、关闭连接）。访问 HTTPS 网站时第一步不是 TLS 也不是 HTTP，而是先连 TCP：本机随机端口 -> 目标服务器:443。如果 TCP 连接失败，后面的 TLS 和 HTTP 根本不会发生。常见的 TCP 连接失败有四种表现：Connection timed out（请求发出去了但没人回应，可能被丢弃、路由不通或防火墙静默丢包）、Connection refused（对方可达但目标端口没有服务或被明确拒绝）、No route to host（本机或中间路由不知道怎么到目标）、Network is unreachable（本机没有可用网络路径）。本次访问 GitHub HTTPS 的典型错误 Failed to connect to github.com port 443, Connection timed out，说明问题大概率发生在 TCP 443 连接阶段，还没进入 TLS。

7.4 TLS 与 HTTP

TLS（Transport Layer Security）运行在 TCP 之上、HTTP 之下，顺序是 TCP -> TLS -> HTTP。它负责验证服务器身份（通过证书确认你连到的确实是目标网站而不是中间人）、协商加密参数、建立加密通道、保证数据不被篡改。浏览器地址栏的小锁，本质上就是 TLS 证书和加密通道正常。TLS 失败的常见错误是 certificate verify failed、SSL certificate problem、TLS handshake timeout，这些和 TCP 超时不一样。如果你看到的是 Failed to connect to ... port 443，通常说明 TCP 都没连上，还没到 TLS 证书验证。

HTTP 是应用层协议，负责表达”我要什么资源”和”服务器返回什么”。看到 403、404、500，一般不是网络不通，而是已经到了 HTTP 应用层，服务器给了一个业务响应。而 HTTPS 可以简单理解为 HTTPS = HTTP + TLS：先用 TCP 连到 443，再用 TLS 建立加密通道，最后在加密通道里传 HTTP。

7.5 SSH 与 HTTPS clone 的区别

SSH 是另一套应用层协议，通常使用 TCP 22，可以用于远程登录、GitHub SSH 仓库访问、端口转发、跳板机。GitHub SSH clone 的链路是 DNS -> IP 路由 -> TCP 22 -> SSH 握手 -> SSH 公钥认证 -> Git 数据传输，这和 HTTPS clone（TCP 443 -> TLS -> HTTP -> Git over HTTPS）完全不同。

把本次现象放进分层模型里：机器 A 直接访问 github.com:443 超时，说明 DNS 可能成功但 TCP 443 不通，TLS 和 HTTP 都没有机会发生；跳板机 B 访问 github.com:22 成功，说明 TCP 22、SSH 握手、GitHub 公钥认证、Git 数据传输全通；B 访问 github.com:443 超时，说明 B 到 GitHub 的 TCP 443 不通或被策略阻断；B 访问 open.bigmodel.cn:443 返回 HTTP/2 200，说明 B 到该目标的 TCP 443、TLS、HTTP 都通。因此当前最准确的网络状态是：不是”有网/没网”的问题，而是不同目标、不同端口、不同协议的可达性不同。

7.6 一个排错判断表

用一句话总结这一节：IP 决定去哪台机器，端口决定找哪个服务，TCP 负责连接，TLS 负责加密和证书，HTTP 负责网页/API 语义，HTTPS 是 HTTP 跑在 TLS 里面，SSH 是另一套独立协议。

8. 为什么 GitHub SSH 22 能通，HTTPS 443 不通

这一节回答两个问题：为什么同一个 github.com，22 能通而 443 不通？这个”通不通”到底是谁决定的？

先说结论：不是 GitHub 这个域名整体”通”或”不通”，而是从某个源 IP 到某个目标 IP 的某个端口是否被允许、是否有路由、是否能建立连接。网络设备看到的不是一句”我要访问 GitHub”，而是这样一组信息：

所以 198.51.100.20 -> 20.205.243.166:22 和 198.51.100.20 -> 20.205.243.166:443 在网络设备眼里是两条不同流量，可以被不同规则处理。

决策者从近到远有这么几层。第一是本机防火墙（iptables/nftables/ufw/firewalld），不过在本次场景里 B 能访问某些 HTTPS 网站，所以不太像本机完全禁止 443。第二是校园网/机构网出口防火墙，这是最可能的决策点：出口设备（防火墙、ACL、NAT 网关、安全审计、流量管控、代理网关）可以按源 IP、目标 IP、目标端口、协议、目标国家、域名分类、应用识别结果等条件放行或阻断，例如”允许 B 访问 github.com:22、不允许 B 访问 github.com:443、允许 B 访问 open.bigmodel.cn:443“。这类策略不一定是”故意针对你”，可能是机构统一策略、出口安全策略或历史配置。第三是教育网/运营商/上级出口路由：到不同目标走的路径可能不同，某条链路可能丢包、绕路、对某些端口处理异常，所以即使没有明确写”禁止 GitHub 443”的规则，也可能表现为 GitHub 443 超时而 22 可用。第四是目标侧（GitHub 自身），但本次不是 GitHub 没开 443，因为全球其他网络可以访问。第五是中间安全设备的应用识别：现代防火墙不只看端口，还能基于 SNI、证书信息、目标 IP 段、流量特征识别应用，对 TCP 22 上的 SSH 和 TCP 443 上的 TLS 做不同处理。

为什么机构网络会按端口做策略？因为端口通常代表服务类型：22 是 SSH（远程登录、Git SSH、运维），443 是 HTTPS（网页、API、下载、几乎所有现代应用流量）。443 的流量范围非常广，安全设备更关注它；而 22 可能被当作”开发/运维用途”放行，或刚好这条路径没被阻断。也可能反过来，有些机构禁止 22 只允许 443。所以不能说”22 一定比 443 更容易通”，只能说在这次实测中 GitHub 22 通、443 不通。

要从现象推断策略，而不是直接看到策略。普通用户通常看不到出口防火墙的具体规则，能看到的是测试结果：ping 通、git SSH 通、curl HTTPS 超时、curl 某个国内 API 通。综合所有测试，最合理的解释是：校园/机构网络对不同源 IP、不同目标、不同端口有不同出口策略或不同路由路径。具体是哪一台设备造成的，需要网管从网络设备上确认。

9. 固定 IP、动态 IP 与校园网网关认证

机器 A 的 IP 192.0.2.10 更像机构固定/服务器网段，而普通校园网动态地址常见是 10.x.x.x。如果在固定 IP 网段上运行校园网认证工具提示”当前 IP 禁止登录网关”，说明这个 IP 段不允许通过普通动态 IP 认证网关登录。所以这类机器的外网权限通常不是靠个人账号登录获得，而是由网管在出口策略上配置。

10. 中转访问的理论基础：VPN、代理、SOCKS、SSH 跳板

这一节是”借另一台机器访问外网”的理论基础。一句话概括：VPN、代理、SOCKS、SSH 跳板，本质上都是把原本”本机直接访问目标”的路径，改成”本机先访问一个中转点，再由中转点访问目标”，但它们工作的层级、影响范围、使用方式并不一样。

10.1 直连与中转

没有 VPN、没有代理时，访问一个网站是 A -> 校园网出口 -> 互联网 -> 目标。如果出口不允许这条流量或路由不可达，就会失败。如果有一台中转机 B 能访问目标，就可以变成 A -> B -> 目标。这个模型成立的前提是 A -> B 通 且 B -> 目标 通，只要这两段都通，即使 A -> 目标 不通 也能通过 B 间接访问。

10.2 VPN 的原理

VPN（Virtual Private Network）的”虚拟”是指它不是重新拉一根真实网线，而是在现有网络上建立一条逻辑上的加密隧道。建立 VPN 后机器上通常会多出一个虚拟网卡（tun0/wg0/utun/ppp0），系统会把一部分或全部流量交给它。VPN 会把原始数据包”包起来”：原本是 A -> github.com，封装后外层看起来是 A -> VPN 服务器，里面包着”我要访问 github.com”的数据。校园网在外层主要看到 A 正在连接 VPN 服务器，而不是直接看到 A 在连接 github.com；VPN 服务器收到后拆开封装，再代表 A 去访问目标。

VPN 能访问原本访问不了的网站，关键不是它有魔法，而是换了一个出口。常见情况是 A -> github.com 不通，但 A -> 某 VPN 服务器 通 且 VPN 服务器 -> github.com 通，于是整体变成 A -> VPN 服务器 -> github.com。目标网站看到的访问来源通常是 VPN 服务器的公网 IP，而不是 A 的校园网出口 IP。不过 VPN 也不是”无条件访问一切”：如果 A -> VPN 服务器 本身不通，VPN 连不上；如果 VPN 服务器 -> 目标 不通，连上 VPN 也访问不了。它只是把出口换成 VPN 服务器。

10.3 一个容易困惑的点：为什么能连上国外 VPN 服务器

很多人会困惑：既然某些国外网站不能直接访问，为什么还能直接连上国外的 VPN 服务器？关键在于，”不能访问国外网站”不是指所有国外 IP、所有端口、所有协议都完全不能访问。真实网络限制通常更细：某些域名不通、某些 IP 段不通、某些端口不通、某些协议不通、某些线路质量差、某些流量特征被识别。所以 A -> github.com:443 不通 和 A -> 某个国外 VPN 服务器:443 通 可以同时成立，这不矛盾，因为网络设备处理的是具体的源 IP、目标 IP、目标端口、协议、SNI、流量特征、路由路径，而不是一个笼统的”国外”标签。

具体来说，不是所有国外 IP 都被阻断，VPN 服务商通常选择相对容易连通的地区和线路；VPN 服务器常运行在 443、8443、80、22 这类常见端口上，而 443 因正常网站大量使用通常被广泛放行；VPN 服务器用的是自己的地址（如 vpn.example.com），而不是被限制网站的域名；VPN 的第一跳只是连 VPN 服务器，不是直接连目标网站；VPN 流量通常加密封装，中间网络不一定能直接看到隧道内部访问的是什么。所以”直连目标网站失败”不代表”连接所有国外服务器都会失败”，VPN 利用的正是这个差异。当然 VPN 服务器本身也可能被限制（IP 被封、端口被阻断、协议特征被识别、线路质量差），这就是为什么 VPN 服务商要准备很多节点让用户切换，以及为什么有些 VPN 也会连不上。

回到当前两台机器：跳板机 B 在这里扮演的角色有点像”校内代理出口”，但它不是完整 VPN。它能做到 A -> B -> open.bigmodel.cn:443（因为 B 到该目标 443 通）和 A -> B -> github.com:22（因为 B 到 GitHub 22 通），但不能很好地做到 A -> B -> github.com:443（因为 B 到 GitHub 443 不通）。真正的 VPN 节点通常要求”它到大多数目标都比较通”，而 B 只是”某些目标通、某些目标不通”。

10.4 VPN 与代理的区别

VPN 通常在系统网络层工作，创建虚拟网卡、修改系统路由，让很多程序自动走 VPN，像是给整台机器换了一条出口。代理通常在应用层或传输层附近工作，只开一个本地端口，程序必须主动配置使用它，更像是给某些程序指定一个中转窗口。例如 VPN 开启后 curl、浏览器、git、Python、系统更新都可能自动走 VPN；而 SOCKS 代理开启后，只有配置了代理的 curl、浏览器、Python 才会走代理，ssh 默认不会自动读 HTTP_PROXY/HTTPS_PROXY。

10.5 HTTP 代理与 SOCKS 代理

HTTP 代理主要服务 HTTP/HTTPS 客户端。对 HTTP 请求是直接转发；对 HTTPS 请求，常见方式是 CONNECT：客户端先请求代理 CONNECT open.bigmodel.cn:443，代理建立到目标的 TCP 连接，之后客户端和目标在这条连接里进行 TLS。常见环境变量是 HTTP_PROXY=http://127.0.0.1:7890 和 HTTPS_PROXY=http://127.0.0.1:7890，注意变量虽然叫 HTTPS_PROXY，但代理本身可能仍是 HTTP 代理，只是用于转发 HTTPS 目标。

SOCKS 代理比 HTTP 代理更底层、更通用，它不太关心上层是 HTTP、HTTPS 还是别的 TCP 协议。程序告诉 SOCKS 代理”请帮我连接某地址:某端口”，代理负责建立连接并转发数据。ssh -D 创建的就是 SOCKS 代理。

socks5:// 和 socks5h:// 的区别在 DNS 解析位置：前者可能由本机先解析域名再把 IP 交给代理；后者（h 表示 hostname）把域名解析交给代理出口端处理。在网络受限时推荐 socks5h://，因为本机解析出来的 IP 不一定适合代理出口端，让代理端自己解析可能拿到更合适的 CDN/线路。

10.6 SSH 跳板与 -D/-J 的根本区别

SSH 跳板（ssh -J）不是通用代理，而是 SSH 自己的中转机制。ssh -J <跳板机> -T git@github.com 的意思是：A 先 SSH 到 B，B 再帮 A 打开到 github.com:22 的连接，A 和 GitHub 的 SSH 会话通过 B 中转。它非常适合 SSH 登录、GitHub SSH clone、scp、rsync over ssh，但它不等于通用 HTTPS 代理。

ssh -D 和 ssh -J 的根本区别在于：ssh -D 创建一个本地 SOCKS 代理端口，其他程序必须配置使用这个端口，适合 curl、API、浏览器、部分 SDK，路径是 应用程序 -> 127.0.0.1:1080 -> B -> 目标；ssh -J 告诉 SSH 自己通过跳板机连接另一个 SSH 目标，主要服务 SSH 协议本身，适合 git@github.com、ssh、scp、rsync，路径是 SSH 客户端 -> B -> SSH 目标。所以结论是：**普通 HTTPS/API 用 ssh -D 加代理环境变量，GitHub SSH 用 ssh -J 或 ~/.ssh/config 的 ProxyJump**。

10.7 为什么”开了代理”不代表所有命令都走代理

如果是 VPN，系统路由可能被接管，很多程序自然走 VPN。但 ssh -D 只是本机开了一个端口 127.0.0.1:1080，这只是一个”可用的代理入口”，程序必须知道并使用它。curl 可以通过环境变量知道要走代理，但 ssh 默认不会因为你设了 HTTPS_PROXY 就自动走 SOCKS。所以”代理入口存在”不等于”所有程序自动使用代理”，这是 VPN 和普通代理最重要的区别之一。

11. 两类需求，两套方案

需求一：GitHub SSH 仓库（如 git clone git@github.com:<你的用户名>/repo.git），走 SSH 协议、github.com:22，推荐用 ~/.ssh/config 加 ProxyJump。

这两类不要混淆，是后面两节配置的核心。

这里有几个关键点要理解清楚。Host jumphost 定义了一个别名，HostName 是真实地址，User 是用户名，IdentityFile 指定私钥；所以 ssh jumphost 实际就是以指定用户名和私钥连接跳板机。ProxyJump jumphost 告诉 SSH：连接这个目标时先经过 jumphost，再由它连接最终目标。-J 是命令行写法（临时测试用），ProxyJump 是配置文件写法（长期生效），两者表达的是同一件事。

GitHub 认证用的是哪台机器的 key，这是一个容易误解的点。即使路径是 A -> B -> github.com，GitHub 认证使用的仍然是 A 上的 GitHub 私钥 ~/.ssh/id_ed25519_github，不是 B 上的 key。B 的作用只是帮 A 打开到 github.com:22 的网络通道，真正和 GitHub 做 SSH 公钥认证的是 A 本机的 SSH 客户端。所以你需要 A 上存在 GitHub 私钥、且 GitHub 账号里添加了对应公钥，而不是只在 B 上配置。

关于私钥，机器上通常有两类：~/.ssh/id_ed25519 用来登录跳板机，~/.ssh/id_ed25519_github 用来登录 GitHub。IdentitiesOnly yes 表示只使用指定的 IdentityFile，不要再尝试 ssh-agent 或默认路径里的其他 key，避免 SSH 乱试多把 key 导致 GitHub 先看到错误 key 而认证失败（Permission denied (publickey)）。ProxyJump 是比旧的 ProxyCommand ssh -W %h:%p <跳板机> 更现代、更好读的写法（%h 是最终目标主机名、%p 是端口），现在一般优先用 ProxyJump。

14. 命令参数逐项详解

这一节拆解常用命令的每个参数，重点不是背命令，而是理解”这个参数控制谁、影响的是 SSH 自己还是 curl/Python 这些程序、是创建代理还是让程序使用代理”。

ssh -N -D 127.0.0.1:1080 <跳板机用户名>@198.51.100.20 里，ssh 启动 OpenSSH 客户端；<跳板机用户名>@198.51.100.20 是远程登录目标；-D 127.0.0.1:1080 是动态端口转发（Dynamic port forwarding），在本机开启一个 SOCKS 代理入口，监听 127.0.0.1 的 1080 端口；-N 是不执行远程命令、不打开远程 shell，只建立连接并保持端口转发。这条命令做的事情是：本机开一个 SOCKS 代理 127.0.0.1:1080，背后出口是跳板机。执行后看起来”卡住”是正常的，因为它要一直运行保持隧道，关掉它代理入口就没了。

-D 里的 127.0.0.1:1080 中，127.0.0.1 是本机回环地址（只允许本机自己访问），1080 是监听端口（SOCKS 常用 1080，也可换成 7890、1081 等未占用端口）。所以这条命令的意思是”在本机创建 127.0.0.1:1080 的代理入口”，不是”去连接远程的 127.0.0.1:1080“。如果写成 0.0.0.0:1080 则会监听所有网卡、局域网其他机器也能访问，通常不安全，推荐用 127.0.0.1。

-fN 是两个参数合写：-f 是认证后转入后台运行（fork to background），-N 是只做转发。所以 ssh -fN -D ... 等价于在后台建立 SOCKS 隧道，终端立刻返回提示符，适合不想单独占一个终端时使用。检查是否启动可以用 ss -lntp | grep 1080（看到 LISTEN 127.0.0.1:1080 即说明代理入口存在），停止后台隧道可以用 pkill -f "ssh -fN -D 127.0.0.1:1080"。

-J 是命令行写法的 ProxyJump，如 ssh -J <跳板机用户名>@198.51.100.20 -T git@github.com：-J 指定先连跳板机再中转，-T 不分配伪终端（测试 GitHub SSH 认证时常用），git@github.com 是最终目标。-i 指定本次认证用哪把私钥。-o IdentitiesOnly=yes 表示只使用指定的 IdentityFile，不尝试 ssh-agent 或默认路径里的其他 key。

-L 和 -D 的区别：-D 是动态转发，创建 SOCKS 代理，程序可以通过它访问很多不同目标；-L 是本地固定端口转发（如 ssh -N -L 127.0.0.1:9999:example.com:443 <跳板机>），只能转发到一个固定目标。-D 更像通用代理。

注意 bash 默认不读 ~/.zshrc 而是读 ~/.bashrc，所以这些函数在 bash 里可能不可用，除非同样写入 ~/.bashrc。

**为什么不直接在 ~/.zshrc 里永久 export HTTPS_PROXY**？因为如果每次打开终端都自动设代理，但此时 1080 隧道没开，那么 curl、npm、pip、某些 SDK 都会去连一个没人监听的端口而卡住。所以更安全的方式是：需要代理时先 socks_bg 再 proxy_on，不需要时 proxy_off。记住 proxy_on 不会自动启动隧道，只有 socks_on/socks_bg 才会启动隧道，正确顺序是先开隧道再开代理。

18. 小结

回到开头那个困惑，现在的答案应该很清晰了。”联网了却访问不了外网”不是矛盾，是因为”联网”是分层的，而网络设备对”哪个源 IP、到哪个目标、走哪个端口、用什么协议”是分别施加策略的。github.com:22 和 github.com:443 在网络设备眼里是两条不同的流量，可以被分别放行或阻断，所以才会出现 SSH 能通而 HTTPS 不通。

当你手里有一台部分目标可达的跳板机时，正确的做法是按需求分两套方案：SSH 类访问（GitHub SSH 仓库、scp、rsync）用 ~/.ssh/config 的 ProxyJump，它只服务 SSH 协议，认证用的还是本机的私钥；HTTPS/API 类访问用 ssh -N -D 开一个本地 SOCKS 代理，再通过 proxy_on 设置环境变量让 curl、Python、SDK 这些支持代理的程序走隧道。两者不要混淆，因为 SSH 不读 HTTPS_PROXY，而 SOCKS 代理也不会自动被所有程序使用。

更宏观地看，VPN、代理、SOCKS、SSH 跳板本质上都是”把直连改成中转”，区别只是工作层级和影响范围：VPN 在系统网络层接管路由，像给整机换出口；代理只开一个本地端口，由程序主动选用，像给某些程序开中转窗口。理解了分层模型（DNS -> IP 路由 -> TCP -> TLS -> HTTP）和这套中转思路，遇到任何”某某目标不通”的问题，都能有条理地拆解、定位、绕过。

最后提醒一点安全常识：ssh -D 监听 127.0.0.1 只允许本机使用是安全的；如果改成 0.0.0.0 会让局域网其他机器也能用你的代理，通常不要这样做。私钥（id_ed25519、id_ed25519_github）要妥善保管，~/.ssh/config 里的跳板机信息属于内部网络资产，分享配置时记得脱敏。
