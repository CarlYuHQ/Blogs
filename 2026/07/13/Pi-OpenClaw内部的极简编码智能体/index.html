Pi：OpenClaw 内部的极简编码智能体

2026-07-13
2.8k- 10m
课程笔记-学习Agent大模型翻译

本文翻译整理自 Armin Ronacher（Flask、Jinja2 作者）的博客文章 Pi: The Minimal Agent Within OpenClaw（2026 年 1 月 31 日），并补充了 Pi 官方文档 和相关技术资料的内容。

OpenClaw 与 Pi 的关系

OpenClaw（早期也叫 ClawdBot 或 MoltBot）是近期备受关注的智能体项目，本质上是一个连接到通讯渠道、能自主运行代码的智能体。但很多人不知道的是：OpenClaw 引擎盖下跑的，是一个叫 Pi 的小型编码智能体。

Pi 的作者是 Mario Zechner（GitHub: badlogic）。与 OpenClaw 作者 Peter 追求”科幻加一点疯狂”的风格不同，Mario 非常务实。尽管风格迥异，OpenClaw 和 Pi 遵循同一个核心理念：LLM 极其擅长写代码和运行代码，那就拥抱这件事。

Mario 对 Pi 的核心哲学有一句话概括：

“An autonomous agent is just an LLM + tools + a loop.”（一个自主智能体，不过是一个 LLM 加工具加一个循环。）

这不是空谈。Pi 在 Terminal-Bench 2.0 上使用 Claude Opus 4.5 进入了排行榜前列，与 Codex、Cursor、Windsurf 等拥有复杂工具链的智能体竞争，而 Pi 的系统提示词加工具定义加起来不到 1000 token。

Pi 是什么

Pi 是一个极简的终端编码助手。根据官方文档的定义，它的核心设计保持轻量，同时通过 TypeScript 扩展、Skill（技能）、Prompt 模板、主题和 Pi 软件包进行扩展。

Pi 吸引人的核心有两点：

内核极小。 Pi 拥有极短的系统提示词，只有四个工具：Read（读）、Write（写）、Edit（编辑）、Bash（执行命令）。Mario 的核心论点是：前沿模型已经被 RL 训练得足够理解”编码 Agent”是什么，不需要一万 token 的系统提示词来教它怎么当 Agent。

扩展系统。 扩展不仅能添加功能，还能将状态持久化到会话（session）中。Pi 本身的工程质量极高：不闪烁、不吃内存、不无故崩溃，非常可靠。

安装与上手

Pi 通过 npm 安装（也可以用 curl 一键脚本）：

安装后在项目目录运行 Pi，使用 /login 进行 Provider 认证（或设置 ANTHROPIC_API_KEY 等环境变量）。配置和会话存储在 ~/.pi/agent/ 目录下。

扩展能力一览

根据官方文档，Pi 的扩展体系涵盖以下几个方面：

扩展类型 说明
Extensions TypeScript 模块：注册工具、斜杠命令、事件处理和自定义 TUI 组件
Skills 可复用的按需能力（Agent Skills），类似 Nanobot 的 SKILL.md
Prompt Templates 可通过斜杠命令展开的可复用提示模板
Themes 内置和自定义终端主题
Pi Packages 打包和分享扩展、Skill、Prompt 和主题的分发机制

此外，Pi 还支持编程式使用：通过 SDK 在 Node.js 应用中嵌入 Pi，通过 RPC Mode 进行进程集成，通过 JSON Event Stream Mode 输出结构化事件，以及用 TUI Components 构建自定义终端 UI。

Pi 里没有什么

要理解 Pi 有什么，更重要的是理解它没有什么。

最明显的缺失是 MCP 支持。 Pi 不内置 MCP。虽然可以为其写 MCP 扩展，但也可以像 OpenClaw 那样用 mcporter（一个把 MCP 调用暴露为 CLI 接口或 TypeScript 绑定的工具）来间接使用 MCP 能力。

这不是偷懒，而是 Pi 的核心哲学决定的：如果想让智能体做一件它还不会做的事，不应该去下载一个扩展或技能，而应该让智能体自己扩展自己。 Pi 庆祝的是”写代码加运行代码”这个理念，需要什么功能，就让它现场写一个。

Pi 完全支持下载别人的扩展，但更鼓励的做法是：把智能体指向一个已有的扩展，然后说”照着那个做一个，但改成我喜欢的样子”。

这跟 Nanobot 的 Skills 系统（从 ClawHub 技能市场下载）形成了有趣的对比。Nanobot 走的是”标准化技能包，按需加载”路线；Pi 走的是”让智能体自己写自己需要的工具”路线。前者生态复用性强，后者灵活度极高。

为智能体构建智能体：架构设计

Pi 的设计哲学是”软件像粘土一样可塑”。这对底层架构有特殊要求，某些约束被内建到了核心设计中。

跨模型可移植的会话。 Pi 的底层 AI SDK 被设计成一个会话可以包含来自多个不同模型供应商的消息。它意识到会话在不同模型供应商之间的可移植性有限，所以不会过度依赖某个供应商的专有特性。

自定义消息层。 除了模型消息，Pi 在会话文件（JSONL 格式）中维护自定义消息。扩展可以用它存状态，系统本身也可以用它存”完全不发给 AI 或只发一部分”的信息。

热重载。 因为扩展状态可以持久化到磁盘，Pi 内置了热重载机制：智能体可以写代码、重载、测试、循环，直到扩展真正可用。它还自带文档和示例，智能体可以用这些资料来扩展自己。

会话是树形结构。 Pi 的会话不是线性列表，而是一棵树。可以分支、在会话内导航，这打开了各种有趣的可能性：比如在主会话不浪费上下文的前提下，开一个”支线任务”去修复一个坏掉的工具。修好后，可以把会话倒回分支前的状态，Pi 会把支线上发生的事总结出来。

树形会话是 Pi 最独特的设计之一。想象在写代码时发现一个工具坏了。传统做法是停下主线任务去修工具，修完回来时上下文已经被污染了。Pi 的做法是开一个分支去修工具，修完后合并回来，主线上下文干干净净。这就像 Git 的分支模型，但用在 AI 对话上。

这一切为什么重要？以 MCP 为例：大多数模型供应商要求工具在会话开始时就被加载进系统上下文。这意味着几乎不可能在不破坏缓存或让 AI 困惑的情况下，中途完全重新加载工具的定义。Pi 的自定义消息和树形会话绕过了这个限制。

上下文之外的工具

Pi 的扩展可以向 LLM 注册一个可调用的工具，但大部分时候，添加给智能体的东西要么是技能（skill），要么是 TUI 扩展。

除了斜杠命令，Pi 扩展可以直接在终端里渲染自定义的 TUI 组件：加载动画、进度条、交互式文件选择器、数据表格、预览面板。TUI 灵活到 Mario 证明了可以在里面跑 Doom，虽然不实用，但能跑 Doom 就能做出有用的仪表盘或调试界面。

实战扩展介绍

以下是几个有代表性的扩展，展示了 Pi 的可能性。虽然可以直接使用，但整个理念是：把智能体指向其中一个，然后随心所欲地改造。

/answer：优雅地回答智能体的问题。 鼓励智能体用自然散文提问而不是结构化对话框。/answer 读取智能体的最后一条回复，提取所有问题，重新格式化成一个漂亮的输入框。

/todos：待办管理。 调出 .pi/todos 目录下所有的 markdown 文件。智能体和用户都能编辑它们，会话可以认领任务来标记”进行中”。

/review：智能体先自审，人类再审。 随着越来越多代码由智能体写出，在人类审查之前先让智能体自审一轮是合理的。因为 Pi 的会话是树形的，可以开一个干净的审查分支，拿到发现后把修复合并回主会话。

/control：多智能体实验。 允许一个 Pi 智能体向另一个发送 prompt，一个不需要复杂编排的简单多智能体系统。

/files：会话文件管理。 列出会话中所有被修改或引用的文件，可以在 Finder 中打开、在 VS Code 里 diff、Quick Look 预览。

其他人也做了扩展，比如 Nico 的子智能体扩展和交互式 shell（让 Pi 能在可观测的 TUI overlay 中自主运行交互式 CLI）。

软件构建软件

以上扩展的共同特点是：没有一个需要手写，都是智能体按照需求自己创建的。告诉 Pi 做一个扩展，它就做了。没有 MCP，没有社区技能市场。

Pi 用户会大量使用技能，但这些技能通常是 Pi 手工定制的，而非从市场下载。比如用一个基于 CDP（Chrome DevTools Protocol）的技能完全替代所有浏览器自动化的 CLI 和 MCP，不是因为别的方案不好，而是因为这样做简单自然，智能体自己维护自己的功能。不需要的技能随时可以删掉。

uv 是一个用 Rust 编写的极速 Python 包管理器（由 Astral 开发），比 pip 快 10 到 100 倍。有 Pi 用户让智能体自动用 uv 替代 pip，体现了”智能体优化自己的工作流”这一理念：它不只是帮你写代码，还帮你把开发工具链变得更高效。

使用 Pi 这样的极简智能体，核心魅力在于：它让”用软件构建更多软件”从口号变成了日常操作。把这种理念推到极致：去掉 UI、去掉输出、连接到聊天工具，就是 OpenClaw 做的事。

Pi 与其他编码智能体的定位差异

在当前 Agent 框架生态中，大多数项目在做加法：更多工具、更长提示词、更复杂的规划链、更多子智能体。Pi 走了相反的路。

对比维度 Claude Code Codex Pi
系统提示词 较长，详细规定行为 较长，含工具链描述 极短，不到 1000 token
内置工具 十余个专用工具 多个工具 + MCP 4 个（Read/Write/Edit/Bash）
扩展方式 MCP / 自定义工具 MCP / API TypeScript 扩展 + Skills
会话结构 线性 线性 树形（可分支）
核心理念 工具丰富、提示精细 工具链集成 让智能体自己扩展自己

Pi 的反直觉之处在于：更少的工具和更短的提示词，反而可能让智能体更强大。因为前沿模型已经通过 RL 训练学会了如何使用工具，过度的系统提示词反而可能限制模型的自主判断空间。
