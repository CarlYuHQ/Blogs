浅谈 LangChain 与 LlamaIndex

2026-07-05- 2026-07-13
5.7k- 21m
课程笔记-学习大模型LLMRAGLangChainLlamaIndex

当你第一次想用大语言模型去回答”自己私人文档里的问题”时，几乎不可避免地会撞上两个名字：LangChain 与 LlamaIndex。它们是当下 LLM 应用开发领域最主流的两个开源框架，也都把 RAG（检索增强生成）作为核心能力。但只要稍微深入就会困惑——既然两者都能做 RAG，到底该学哪个、用哪个？网上大量对比文章往往停在”LangChain 适合编排、LlamaIndex 适合数据”这种口号式结论上，难以指导真实选型。

本文想跳出速查表式的罗列，用叙述的方式把这两个框架的来龙去脉、设计哲学、核心抽象与生态版图讲透，最后再用同一份文档、同一个问题，分别用两个框架跑通一个 RAG 应用，让你在代码层面直观感受它们的差异。读完之后，你应当能根据自己的任务特点，做出有依据的判断，而不是被”哪个更火”牵着走。

LangChain：从 prompt 管理到 agent 工程平台

它是怎么长出来的

要理解 LangChain 今天为什么是这个样子，得回到它的起点。LangChain 由 Harrison Chase 在 2022 年 10 月创建——请注意这个时间点，它比 ChatGPT 的发布（2022 年 11 月 30 日）还要早一个多月。在那样一个”文本进、文本出”的蛮荒年代，开发者面对的最大痛苦不是模型不够强，而是没有统一的接口：每家模型的 SDK 风格各异，提示词散落在各处，把”先检索再生成”这种多步流程串起来要写大量胶水代码。LangChain 最初的定位就是一个 prompt 管理与 LLM 调用编排工具，名字本身就来自 “Language” 与 “Chains”——前者指语言模型，后者指把多个步骤串成的”链”。

随后的两年里，它的定位经历了三次明显的跃迁。第一阶段是 2022 年底到 2023 年初，它快速吸收了 ReAct 论文的思想，加入了第一批基于”让 LLM 生成 JSON 再解析”的通用 agent，成为一个真正意义上的 LLM 应用编排框架。第二阶段是 2023 年中到 2024 年，随着 OpenAI function calling 的发布和模型生态的爆发，LangChain 把集成数量做到了七百以上，并相继孵化出 LangSmith（可观测性平台）和 LangGraph（基于图的有状态编排运行时），公司本身也从开源项目变成了一个定位为”让智能体无处不在”的商业实体。第三阶段是 2025 年 10 月的 v1.0 大重构——官方彻底废弃了旧版的 chains 与 agents，只保留一个建在 LangGraph 之上的高层抽象 create_agent，自我定位进一步收敛为 “agent engineering platform”（智能体工程平台）。

理解这条演进线非常关键：LangChain 今天的形态不是某个天才的一锤定音，而是被”模型能力快速进化”和”应用复杂度持续上升”两只手共同塑形的结果。它从”管 prompt”出发，一路被推向了”编排复杂 agent”。

设计哲学：一切皆 Runnable，用 LCEL 组合

LangChain 最核心的设计决策，是把框架里的每一个组件——无论是聊天模型、嵌入模型、检索器、提示模板、输出解析器还是工具——都统一实现为一个叫做 Runnable 的接口。这个接口定义在 langchain-core 里，它要求每个组件都提供 invoke、batch、stream 以及对应的异步版本 ainvoke、astream 等一套统一的方法。这意味着无论你换的是模型还是检索器，调用方式始终一致；也意味着框架可以在运行时对任意组件做统一的批处理、流式和并行优化。

在这个统一接口之上，LangChain 提供了一套叫做 LCEL（LangChain Expression Language） 的声明式组合语法。它的核心是 Python 的 | 运算符：把两个 Runnable 用 | 连起来，就得到了一个 RunnableSequence，等价于 a.pipe(b)。在最经典的 RAG 场景里，你可以把”检索器”和”组装提示词、调用模型、解析输出”这一串步骤，用一行 retriever | prompt | llm | parser 优雅地表达出来。LCEL 还会自动把字典识别为并行执行、把普通函数包装成 Runnable，于是组合复杂流水线时几乎不需要样板代码。

LCEL 的真正卖点不是语法好看，而是它承诺的三项运行时优化。其一，优化的并行执行：当你用字典并行跑多个 Runnable，或用 batch API 让多个输入同时流过同一条链时，框架会自动并行以降低延迟。其二，天然支持异步：任何 LCEL 链都能用异步 API 跑，这对服务端高并发至关重要。其三，首字延迟最小化的流式：LCEL 链被流式执行时，框架会自动优化以尽快吐出第一个 token。不过官方也坦率地划了边界——LCEL 适合较简单的、本质上有向无环的编排；一旦任务需要循环、复杂分支、人在回路或多 agent 协作，就应该切换到 LangGraph。这条边界，恰恰是 LangChain 与 LlamaIndex 取舍分歧的起点之一。

包结构与生态版图

LangChain 并不是单一的包，而是一组分工明确的多包体系。最底层是 langchain-core，它定义了 Runnable、ChatModel、BaseRetriever、PromptTemplate、Document、VectorStore、Embeddings 等所有抽象基类，刻意不包含任何第三方集成、依赖极轻，是整个生态的地基。地基之上是主包 langchain，提供 chains、agents 和各种跨集成的检索策略这种”通用认知架构”。围绕主包，大量的具体集成被拆分成独立的 partner 包，如 langchain-openai、langchain-anthropic、langchain-pinecone、langchain-chroma 等；而那些由社区维护、面更广的第三方集成，则统一收在 langchain-community 里。更激进的实验性功能（CSV agent、SQL agent 等）被隔离在 langchain-experimental 中，明确告诉使用者”稳定性不保证”。这种分层让用户可以按需引入，避免一个臃肿的全家桶。

在框架之上，LangChain 还生长出了一个完整的平台生态。LangGraph 是其中最关键的一块——它是一个受 Pregel 和 Apache Beam 启发、对外接口借鉴 NetworkX 的低层编排运行时，把应用建模为”节点 + 边 + 共享状态”的有向图。与 LCEL 那种有向无环的流水线不同，LangGraph 原生支持循环、条件分支、人在回路、断点持久化以及短期与长期记忆，专门用于构建长时运行、有状态的复杂 agent。事实上 v1.0 之后，连高层抽象 create_agent 都是建在 LangGraph 之上的。LangSmith 则是一个闭源的商业化开发者平台，主打可观测性（trace、监控、告警）、评估（离线与在线打分）以及 prompt 工程的版本管理与协作；任何 LangChain 应用只要设置环境变量就能自动埋点。LangServe 负责把一条 Runnable 一键部署为 REST API，而更新的 LangGraph Platform 则面向有状态工作流的托管部署。这套”框架 + 运行时 + 套件”的三层结构，构成了 LangChain 今天的完整版图。

当然，LangChain 也并非没有争议。它最受诟病的一点是抽象过重：旧版的 LLMChain、ConversationalRetrievalChain 之类把提示词等关键细节藏得很深，官方自己在 LCEL 文档里都承认”这些旧链隐藏了重要细节”，而这正是 v1.0 大重构的根本动因。与之相伴的是版本变动频繁——从 0.0.x 一路到 1.0，几乎每一版都废弃一大批 API，社区维护成本很高，初学者也常在新旧两套文档之间迷路。理解这些批评，恰恰能帮我们看清它擅长什么、不擅长什么。

LlamaIndex：为 LLM 连接数据的数据框架

另一条起跑线

如果说 LangChain 的起点是”管 prompt、串步骤”，那么 LlamaIndex 的起点就是一个更具体也更朴素的痛点：LLM 不懂你的私有数据。预训练模型再强，也不知道你公司的内部 wiki、你的科研笔记、或者一份刚发布的财报里写了什么。LlamaIndex 由 Jerry Liu 在 2022 年底创建，最初的名字叫做 GPT Index，从名字就能看出它的初衷——为 GPT 这类模型建一个”索引”，让它们能查到私有数据。2023 年初项目更名为 LlamaIndex，并逐渐发展成”为 LLM 应用提供数据接入与检索能力”的完整框架，公司也定型为 run-llama。

这个出身决定了 LlamaIndex 的基因：它不是从”编排通用 LLM 任务”出发的，而是从”如何让 LLM 高效、准确地利用外部数据”出发的。换句话说，RAG 不是它的一个功能，而是它的主场。这种定位差异，会在后面几乎每一个设计选择中体现出来。

设计哲学：端到端的数据流水线

LlamaIndex 的核心抽象是一整条数据流水线，而不是可以任意拼接的小积木。它把整个 RAG 过程拆成”数据接入 → 文档处理 → 索引构建 → 查询编排 → 响应合成”几个阶段，并为每个阶段提供了高度封装的组件。

这条流水线的起点是数据接入层（Data Ingestion）。LlamaIndex 通过 LlamaHub 这个集中仓库，提供了两百多种数据连接器，覆盖 PDF、Notion、Google Drive、网页、数据库、Slack 等几乎所有常见的数据源；针对最棘手的复杂 PDF 与表格解析，它还提供了专门的 LlamaParse。被接入的原始内容会统一建模为 Document 与更细粒度的 Node——后者还携带元数据与上下游关系，是后续检索的基本单元。

流水线的中间是索引层（Indices），这也是 LlamaIndex 区别于 LangChain 最鲜明的特色。它不只提供向量索引，而是提供了一整套针对不同检索场景的索引结构：VectorStoreIndex 适合语义相似度检索；SummaryIndex（前身是 List Index）适合顺序遍历多个节点后综合；TreeIndex 适合从粗到细地钻取长文档；KeywordTableIndex 适合关键词驱动的精确定位；KnowledgeGraphIndex 则把实体关系组织成图，适合多跳推理。把”该用哪种索引”这件事显式地交给开发者，是 LlamaIndex 对 RAG 复杂性的正面回应。

流水线的终点是查询层（Query Engine）。一个索引可以 as_query_engine() 一行变成查询引擎，内部自动串起 retriever（召回）、node postprocessor（过滤）、reranker（重排）和 response synthesizer（响应合成）。再往上是更高阶的 chat engine、router（按查询动态分发到不同引擎或工具）以及 agent。这种设计的核心理念是：默认值要足够合理，让新手三行代码就能跑通——from_documents 建索引、as_query_engine 拿引擎、.query() 提问，于是开发者可以把精力放在数据本身，而不是反复调样板代码。

生态与边界

围绕这条流水线，LlamaIndex 也长出了自己的生态。LlamaHub 是数据连接器与工具的中央仓库，对应 LangChain 的集成体系，但更聚焦于”数据进”这一侧。LlamaCloud 是托管的企业级 RAG 平台，把索引管理、增量更新、权限控制等都做成了云服务。LlamaParse 则是一个针对复杂文档（尤其含表格、公式、多栏排版的 PDF）的高质量解析器，是 LlamaIndex 商业化的拳头产品之一。此外，LlamaIndex Workflow 提供了事件驱动的工作流引擎，让复杂的多步 RAG 流程可以被组织成可观测、可恢复的事件图。

需要客观指出的是，LlamaIndex 的高层抽象既是优点也是负担。它让你开箱即用，但当默认行为不符合需求时，要定制就得往下钻到 retriever、postprocessor 这一层，理解成本并不低；它的 agent 与工作流能力虽然齐全，但相比 LangGraph 那种专门为复杂有状态 agent 设计的低层运行时，深度和生态规模仍有差距。另外，由于迭代很快，文档偶尔会滞后于代码，照着官方文档跑也可能会撞上报错——这是所有快速演进框架的通病，LlamaIndex 也不例外。

深度对比：同样的 RAG，不同的取舍

讲完两个框架各自的来路，我们终于可以把它们放在一起比较了。需要先破除一个常见误区：LangChain 和 LlamaIndex 不是二选一的对立关系。它们解决的核心问题有重叠（都能做 RAG），但出发点和擅长领域并不相同。

从设计哲学上看，LangChain 是”积木式组合”思维——它把组件做小、做统一，再用 LCEL 这种声明式语法让你像搭乐高一样拼出流水线，强调的是可控性与可组合性。LlamaIndex 是”流水线式封装”思维——它把整条 RAG 链路做成高度封装的端到端组件，强调的是开箱即用与默认合理。这两种取向没有高下之分，只是把”灵活性”和”易用性”放在了天平的不同端。

从抽象层级上看，LangChain 的抽象相对更底层也更厚：你拿到的是 prompt、retriever、llm 这些可以单独摆弄的零件，代价是要写更多组合代码。LlamaIndex 的抽象更高层：一个 VectorStoreIndex 把分块、嵌入、入库全包了，代价是默认流程被锁死得更多。一个直接后果是——当你想做”非标准”的事情（比如自定义检索逻辑、改写查询、插入业务规则）时，LangChain 通常更顺手；而当你想最快地”把一份文档变成可问答的知识库”时，LlamaIndex 往往更省心。

从RAG 的纵深来看，两者各有王牌。LangChain 的强项是高级检索算子极其丰富：self-querying retriever 能把自然语言拆成”语义查询 + 元数据过滤”，multi-query retriever 能用多个变体查询提升召回，contextual compression retriever 能在召回后做抽取或过滤，ensemble retriever 能把 BM25 与向量检索融合成混合检索。这些算子都统一在 Runnable 接口下，可以用 LCEL 任意串联。LlamaIndex 的强项则是索引类型的多样性和查询引擎的精细分工：树索引、图索引、关键词索引让它能应对单靠向量相似度搞不定的检索场景，而 sub-question query engine、recursive retriever、sentence window retrieval 等模式则针对复杂问答做了专门优化。简单说，LangChain 在”召回之后怎么加工”上更强，LlamaIndex 在”召回本身怎么做”上更全。

从生态与可观测性上看，LangChain 背靠 LangSmith + LangGraph + LangServe 的完整闭环，从原型、调试、评估到部署有连贯的工具链，尤其 LangGraph 在复杂有状态 agent 上几乎没有对手。LlamaIndex 则用 LlamaHub + LlamaCloud + LlamaParse 在”数据接入与文档解析”上构筑了深护城河，LlamaParse 对复杂 PDF 的解析质量是业界公认的第一梯队。可观测性方面，LangChain 有原生的 LangSmith，LlamaIndex 则更多依赖第三方（如 Arize、Langfuse）的集成。

把这些差异汇总成一张选型对照表，会更清晰：

一条实用的选型经验是：如果你的核心是”把一堆文档快速变成高质量问答系统”，优先试 LlamaIndex；如果你的核心是”构建一个有多步推理、工具调用、人在回路的复杂 agent”，优先试 LangChain（特别是 LangGraph）。而在真实项目里，两者组合使用也很常见——比如用 LlamaParse 做文档解析、用 LlamaIndex 建索引，再把检索结果喂进 LangGraph 编排的 agent 流程。

实战：用两个框架实现同一个 RAG

说了这么多，不如让代码自己说话。下面我们用同一份 Markdown 文档、同一个问题”文中举了哪些例子？”，分别用 LangChain 和 LlamaIndex 跑通一个最小 RAG。两段代码完成的是完全一样的事情，但风格上的差异一眼可见。

LangChain 版：把每一步显式串起来

LangChain 的写法是把”加载、分块、嵌入、入库、检索、组装、生成”每个环节都显式写出来，最后用提示模板把上下文拼好交给 LLM。这种写法啰嗦但透明——每一步都看得见、改得动。

注意这段代码里几个值得留意的细节。其一，RecursiveCharacterTextSplitter 不带参数时的默认行为是按段落、行、空格、字符的顺序递归切分，尽量保留语义完整性，默认块大小 4000、重叠 200。其二，用 "\n\n" 双换行连接多个检索块，是为了让 LLM 把它们识别为相互独立的上下文片段。其三，提示模板里明确写了”如果上下文信息不足请直接告知”，这是抑制幻觉的关键约束——RAG 的可信度，一半取决于检索质量，另一半取决于你愿不愿意让模型承认”不知道”。

LlamaIndex 版：三行核心代码搞定

同样的功能，LlamaIndex 的写法显著更短——分块、嵌入、入库这些步骤被 VectorStoreIndex.from_documents 一行包办，开发者完全不需要关心中间细节。

把两段代码并排看，差异就具象化了。LangChain 版像是”手动挡”——你掌控离合、换挡、油门的每一步，代价是动作多；LlamaIndex 版像是”自动挡”——踩油门就走，代价是变速箱的逻辑被封装在内部。当默认逻辑正好合你意时，LlamaIndex 让你起飞；当你想精细控制分块策略、检索算子、提示结构时，LangChain 给你那个旋钮。

跑通之后，模型会基于这份本地文档，准确回答出文中举了哪些强化学习的例子（羚羊试错、股票交易、雅达利游戏、餐馆选择、挖油等等）。这就是 RAG 在工作——模型没有凭记忆瞎编，而是真的”读”了你喂给它的文档。

结语：工具是手段，理解才是目的

回到开头那个问题：到底该学 LangChain 还是 LlamaIndex？如果你已经读到这里，答案其实已经浮现——取决于你要解决的核心问题。如果你要做的是数据密集型的知识库问答，LlamaIndex 的开箱即用和丰富的索引结构会让你事半功倍；如果你要构建一个有多步推理、工具调用、人在回路的复杂智能体，LangChain 加 LangGraph 几乎是当前最完整的工程化路径。而理解了”加载—分块—嵌入—检索—生成”这条底层骨架之后，换任何框架都只是 API 翻译。

更值得记住的是，两个框架本身都在飞速演化。LangChain 在 2025 年用 v1.0 完成了一次伤筋动骨的重构，把旧的链与 agent 全部迁移到 LangGraph 之上；LlamaIndex 也在不断扩展它的 Workflow 与 agent 能力。具体的 API 会变，但它们背后的两条思路——“积木式组合”与”流水线式封装”——是 LLM 应用工程里会长期存在的两种基本范式。吃透这两种范式，远比记住某一版 API 重要。

最后给一个朴素的建议：别在选型上纠结太久。先用一个框架把最小可用的 RAG 跑通，让真实的数据和真实的反馈告诉你瓶颈在哪——是检索召回不准、是上下文太长被稀释、还是答案格式不可控。这些具体问题，才会真正指引你该往哪个方向深入，或者该不该切换工具。框架是脚手架，把 RAG 这件事做成、做对，才是目的本身。
