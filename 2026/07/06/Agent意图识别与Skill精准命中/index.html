Agent 意图识别怎么做？Skill 如何精准命中？

2026-07-06- 2026-07-13
5.7k- 19m
课程笔记-学习Agent大模型LLM

本文围绕两个 Agent 工程的核心问题展开：”Agent 意图识别怎么做？”与”Agent 如何精准命中 Skill？”。由于无法直接观看原短视频，文中内容综合自牛客、53AI、阿里云、清华大学 Skill-RAG 研究等公开技术资料，整理成一份系统的技术解答。这两个问题看似独立，实则前后呼应：意图识别回答”用户想干什么”，Skill 命中回答”我该用哪个能力去干”，两者共同构成了 Agent 推理层的设计骨架。

引言：为什么这两个问题总是一起出现

这两道问题之所以经常被放在一起讨论，是因为它们指向同一个核心架构难题：如何让一个拥有成百上千种能力的通用大模型，在每一次具体对话中，只激活正确的那一条路径。

大模型本身是”什么都懂一点”的概率机器，它的强大与它的不可控是一体两面的。一个真正可用的 Agent 系统，必须在大模型和真实业务之间，架起一层”推理前处理层”。这一层要完成两件事：先听懂用户到底想要什么（意图识别），再决定调用哪个具体能力去满足这个想要（Skill 命中）。前者是分类问题，后者是检索与决策问题。把这两件事讲透，基本上就理解了 Agent “怎么不犯傻”的全部秘密。

问题一：Agent 意图识别怎么做？

先把概念锚定准确

很多人一上来就说”用大模型分类”，这其实跳过了一个关键的前置问题：在 Agent 语境下，”意图识别”到底识别的是什么？

意图识别（Intent Recognition）是自然语言理解（NLU）的核心环节，它的本质是一个分类任务（Classification Task）：给定用户的自然语言输入，把它归入预定义的若干意图类别中的一个。比如在一个企业 IT 助手场景里，用户说”我电脑连不上网了”，意图识别要判断这是”故障排查”，而不是”购买咨询”，也不是”闲聊”。判断对了，后续才能把控制权分发给正确的下游工作流或工具；判断错了，后面再强的检索和生成都是在错误的方向上狂奔。

这里要特别区分两个容易混淆的概念：意图识别解决的是”去哪里“的问题（路由分发），而槽位填充（Slot Filling）解决的是”带什么参数去“的问题（实体提取）。一个成熟的 Agent，往往是”意图识别 + 槽位填充”联动工作：先判断意图，再从同一句话里抽取执行该意图所需的参数。把这两个概念分清，是理解后续一切设计的前提。

技术方案的三代演进

要回答”怎么做”，关键是理解不同技术路线的取舍。意图识别的技术方案大致经历了三代演进，每一代都有它仍然适用的场景。

最早的是规则与关键词匹配。它用正则、关键词词典或意图模板去做硬匹配，优点是完全可控、零成本、延迟极低，缺点是完全无法处理用户的表达多样性：用户说”网断了””连不上 Wi-Fi””上不了网”是同一个意图，规则很难穷举。这套方案今天并没有消失，在意图数量很少（比如只有三五个）、且对确定性要求极高的场景里，它依然是最稳妥的选择。

第二代是传统 NLU 模型，以 BERT 等预训练语言模型为代表。做法是收集大量标注数据，微调一个意图分类器，同时配合序列标注做槽位填充。这条路线在 2023 年之前的对话系统里是主流，准确率高、推理快、成本低（可以用很小的模型）。它的局限在于需要大量高质量标注数据，且新增一个意图就要重新标注、重新训练，扩展性差。今天在企业级客服、智能家居这类意图相对固定、数据量充足的场景里，它仍是性价比很高的选择。

第三代是基于大模型的意图识别，也是当下 Agent 场景最主流的方案。它的核心思路是不再训练专门的分类器，而是用一个通用大模型，通过精心设计的 Prompt 来完成分类。这带来一个根本性的好处：新增意图几乎零成本。只要在 Prompt 的意图定义里加一段描述，模型立刻就能识别新意图，无需重新训练。而且大模型天然能处理口语化、带错别字、跨意图边界的模糊输入，鲁棒性强得多。

一个被广泛采用的工程实践是大小模型分工：意图识别这种”快但不需要太深推理”的任务，交给一个便宜快的小模型（比如 Qwen-Flash 这类）；而核心推理、代码生成这种”需要深度思考”的任务，才交给贵但聪明的大模型（如 Qwen-Max）。这样既控制了成本和延迟，又不损失关键环节的质量。

别忘了兜底策略

把识别做得很准还不够，真实系统必须考虑”识别不出来怎么办”。一个成熟的系统必须有兜底机制，否则一旦遇到训练分布外的输入就会彻底崩盘。

兜底的核心思路有三层。第一层是设计一个”未知意图“分类，作为所有低置信度输入的收口。第二层是澄清话术，当置信度不高或者多个意图概率接近时，主动反问用户，比如”您是想查询订单，还是联系客服？”。第三层是转人工，对于反复澄清仍无法确定的、或属于高风险类别的请求，果断交给人类处理。这三层共同构成了一个”识别失败也不会让用户体验断裂”的安全网。在医疗、法律、金融这类高风险场景里，兜底机制往往比识别准确率本身更重要。

意图识别的进阶：查询重写与实体提取

意图识别之后，往往还有两个常被忽视但极其重要的环节。

第一个是查询重写（Query Rewriting）。用户的原始输入往往是模糊的、缺失上下文的、甚至带有误导性的，直接拿去检索或调用工具效果会很差。查询重写负责把用户的”原始需求”转化为机器可理解的”精准指令”。常见的重写策略有三种：同义扩展解决专有名词匹配问题（”我想买个本子” → “笔记本电脑 Laptop 价格 型号”）；指代消解解决多轮对话中的上下文缺失（”它多少钱？”结合上文重写成”iPhone 15 的价格是多少？”）；后退提示（Step-Back Prompting）解决过于具体导致检索失败的问题（”为什么我的 Python 代码报了 KeyError” → “Python 中 KeyError 的常见原因及解决方法”）。更高级的还有 HyDE（假设性文档嵌入）：让 LLM 先生成一个”假设答案”，再用这个假设答案去检索真实文档，因为”答案和答案”在向量空间里比”问题和答案”更接近。

第二个是实体提取与元数据过滤。在结构化数据查询场景（商品检索、简历筛选）里，光知道意图还不够，还要从同一句话里抽出结构化条件。比如用户说”帮我找几个北京的、三年经验以上的 Java 工程师”，意图是”简历筛选”，但真正可执行的前提是提取出 location=北京、years_of_experience>=3、skill=Java 这些字段，用来构造向量库的 filter 或 SQL 的 where 子句。所以一个完整的推理前处理层，往往是由 Router（路由）+ Rewriter（重写）+ Extractor（提取） 三个组件协同构成的。这一层”认知中间件”，才是区分一个简单 Chatbot 和真正智能业务助理的分水岭。

问题二：Agent 如何精准命中 Skill？

先厘清 Skill 到底是什么

回答”如何精准命中”之前，必须先把”Skill 是什么”讲清楚，因为这里的很多概念是混乱的。我们要快速回顾一下 Agent 能力扩展的三阶段演进，这样才能理解今天”命中 Skill”这件事的技术背景。

第一阶段是 Function Calling（2023 年 OpenAI 发布）。它让大模型从”只会生成文本”变成”会使用工具”：开发者告诉模型有哪些工具可用，模型在判断需要时输出一个结构化的 JSON 调用请求，由外部程序执行后把结果返回给模型。它的局限是每个应用都要自己定义和实现工具，工具之间没有统一标准，无法复用，而且所有工具定义都要塞进 System Prompt，token 消耗大。

第二阶段是 MCP（Model Context Protocol，2024 年 Anthropic 发布）。可以把它理解为 AI 工具的 RPC 协议标准，解决的核心问题是让不同开发者写的工具，AI 都能听得懂、用得上。它把”每个应用自己定义工具”升级为”统一协议标准”，把”静态配置”升级为”动态发现”，实现了”一次开发，处处可用”。

第三阶段就是 Agent Skills（2025 年 10 月 Anthropic 发布）。Skills 是”组织好的指令、脚本和资源的集合，Agent 可以动态发现并加载它们以更好地完成特定任务”。它的核心创新是按需加载，而非全量加载。传统做法是把所有能力塞进一个巨大的 System Prompt，不管用不用得上；Skills 则把能力拆分成独立单元，由模型自己判断什么时候加载哪个。可以说，Skills 的本质是提示词工程的进化：以前是”一个超大 Prompt 包打天下”，现在是”很多个独立 Skill 按需激活”。

精准命中的真正难点：清华揭示的三大痛点

一种常见的回答是”靠大模型自己判断该用哪个 Skill”。这话没错，但只对了一半。当 Skill 数量很少（十几个以内）时，模型确实能胜任；可一旦 Skill 规模上去，问题就来了。清华大学团队的研究精确地刻画了当前 Agent 在技能调用上的三大致命痛点。

第一个痛点是上下文过载。现有的做法是把所有可用技能的摘要塞进 Agent 的上下文窗口，让模型自己判断该用哪个。当技能只有十几个时这没问题，但当 Agent 拥有成百上千甚至上万个技能时，有限的上下文窗口会被技能描述完全塞满，根本没有多余空间去处理任务本身。截至 2026 年 4 月，仅 SkillsMP 一个平台就托管了超过一百万个技能，这个矛盾已经非常尖锐。

第二个痛点是选择准确率骤降。当候选技能从十几个扩展到成千上万个，模型无法从海量技能中精准匹配到最合适的那个，错误率呈指数级上升。给它的选择越多，它反而越不会选。

第三个痛点最隐蔽也最关键，叫技能加载幻觉。模型不管任务是否真的需要外部帮助，都以差不多的概率去加载技能，就算没有匹配的技能也瞎调用，只有最前沿的大模型才有微弱的区分能力。换句话说，模型不仅”选不准”，还”乱选”，在不该用工具的时候也硬用。

理解了这三大痛点，就能明白为什么”靠大模型自己判断”不是一个可扩展的答案。

命中机制：从两阶段加载到 Skill-RAG

那么业界是怎么解决这个问题的？有两套互补的机制。

第一套是 Skills 本身设计的两阶段加载机制，这是 Claude Agent Skills 设计中很精妙的部分，专门为了不导致 token 激增。它的做法是把加载分成两个阶段：启动时只加载每个 Skill 的元数据（name + description），这部分极省 token（大约每个 Skill 30 到 100 个 token）；只有当模型判断某个 Skill 与当前任务相关时，才把那个 Skill 的完整内容（SKILL.md 及其附带的脚本、资源）注入上下文。比如你装了 Obsidian 相关的 Skills，平时只占一两百 token 的描述；只有当你说”帮我创建一个 Obsidian 笔记”时，对应 Skill 的完整内容才会被加载。这就是”按需加载”的具体落地。

这里有一个常被追问的细节：谁来判断是否需要加载某个 Skill？答案是完全由大模型决定，客户端不做语义判断。客户端只负责收集元数据、打包发送、执行工具调用；它不做关键词匹配、不做正则、不做向量嵌入、不做意图分类。背后的设计哲学是：任何需要做模糊语义判断的地方都交给大模型，客户端做得越轻越通用。但这也带来了不可靠性：Skills 有可能没被自动激活，模型会直接跳过它们。对此的工程对策是：对必须使用的 Skill 用 /skill-name 强制调用、把关键规则放进始终在上下文里的 CLAUDE.md、或设置 disable-model-invocation: true 改为手动调用。

第二套机制面向的是海量技能场景，就是清华团队提出的 Skill-RAG（也叫 SRA，技能检索增强）。当技能数量超过几十个，全量塞进上下文不可行时，Skill-RAG 的思路非常直接：不再把所有技能都塞到上下文里，而是像搜索引擎一样，根据当前用户任务，从海量技能库里只检索出最相关的 2 到 3 个技能，再交给 Agent 使用。为了系统研究这个方案，清华团队还构建了行业首个技能检索评测基准 SRA-Bench：包含 5400 个测试实例、636 个人工标注的”黄金技能”（最优技能），并把它们混入 2.6 万个从网络收集的干扰技能中，模拟真实的大规模技能库场景，覆盖数学推理、形式逻辑、工具调用、医学计算、竞赛数学、代码生成六大领域。

这里要主动澄清一个常见的概念混淆：Skill-RAG 和我们熟悉的普通 RAG 不是一回事。可以用厨师做菜来类比：普通 RAG 是食材仓库，Skill 是菜谱。普通 RAG 检索的是静态的陈述性知识（文档、网页、知识库），目的是帮你”找知识”、减少生成幻觉；Skill-RAG 检索的是动态的可执行能力（技能、工具、API），目的是帮你”干事情”、扩展 Agent 的功能边界。两者的评价标准也不同：前者看检索结果与问题的语义相似度，后者看最终任务的完成质量与成功率。在企业系统里，它们是互补关系：知识类问题用普通 RAG，工具类需求用 Skill-RAG，常常混合使用。

最深刻的结论：瓶颈不在检索，在判断

如果只从这一节记住一个要点，应该是下面这个研究结论。

清华团队测试了 BM25、BGE、LLM 重排序等各种检索方案，发现 LLM 重排序效果最好，但一个反直觉的结论是：检索质量的提升，无法完全转化为最终任务的改进。原因在于：就算检索到了正确的技能，如果 Agent 选择不用，或者不该用的时候也在用，那么检索再准确也没用。研究者打了一个精妙的比方：这就像你把一本极好的烹饪书递给一个完全不看菜谱就瞎做的厨师，书再好，问题不在书上。

这个结论引出一个非常重要的工程判断：当前 Agent 的最大瓶颈不是检索技术，而是”需求感知”能力，也就是让 AI 准确判断”我什么时候自己就会，什么时候需要调用外部技能”。因此，做 Agent 系统时，不要盲目追求大而全的技能库，而应该优先把需求感知模块做扎实，先让 AI 能准确判断”什么时候该用工具”，再谈工具的丰富度。这个认知，是从”会用工具”到”会用对工具”的关键跃迁。

两个问题的内在联系：推理层的完整图景

把两个问题拼起来看，才能看到 Agent 推理层的完整设计。

意图识别和 Skill 命中并不是两个孤立的环节，而是一条连贯的推理流水线。当用户输入到达 Agent 时，先由意图识别判断”用户想干什么”，这决定了应该走哪条业务路径（查知识、查数据、执行操作、还是闲聊）。确定路径后，如果这条路径需要调用能力，就进入 Skill 命中环节，决定”用哪个具体技能去干”。在技能数量少时，命中可以靠模型直接判断（两阶段加载）；在技能数量多时，命中要靠 Skill-RAG 先检索再决策。而无论哪一步，都离不开查询重写和实体提取把用户的原始表达”翻译”成机器可执行的精准指令。

把这条流水线抽象一下，会得到一个清晰的”推理前处理层”架构：用户输入 → 意图识别（Router）→ 查询重写（Rewriter）→ 实体提取（Extractor）→ Skill 检索/命中 → 工具执行 → 结果整合 → 最终回答。这个架构在 Dify、Coze 等平台里都有对应的可视化节点，在代码里则对应 LangChain 的 RouterChain、LlamaIndex 的 Router Query Engine 等组件。理解了这张图，也就理解了为什么这两个问题总要放在一起看：它们本来就是同一件事的两面。

小结

回到开头的两个问题，我们可以给出这样完整的解答。

对于”Agent 意图识别怎么做”，答案是：先定性（本质是分类任务，要区分意图识别与槽位填充），再选方案（规则 → 传统 NLU → 大模型，Agent 场景下大模型方案因新增意图零成本而成为主流，配合大小模型分工控成本），然后抓 Prompt 设计（定义明确、边界清晰、输出结构化），最后补兜底（未知意图分类 + 澄清话术 + 转人工），并在此基础上叠加查询重写和实体提取，构成完整的 Router + Rewriter + Extractor 推理前处理层。

对于”Agent 如何精准命中 Skill”，答案是：先理清 Skill 的来龙去脉（Function Call → MCP → Skills 三阶段演进，本质是按需加载的提示词工程进化），正视规模带来的三大痛点（上下文过载、选择准确率骤降、技能加载幻觉），然后用两套机制解决（Skills 的两阶段加载机制 + 海量场景下的 Skill-RAG 检索），并牢记那个反直觉的结论：瓶颈不在检索技术，而在 Agent 的需求感知能力。

更宏观地看，这两个问题背后是同一个工程哲学：Agent 的智能，不来自于大模型本身有多强，而来自于我们在大模型和真实业务之间架设的那一层”认知中间件”有多精细。大模型负责模糊的自然语言理解与决策，中间件负责把它约束到正确的、可控的、可解释的业务路径上。把这一点想透了，就抓住了整个 Agent 工程的核心命题。
