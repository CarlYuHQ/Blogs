Nanobot 源码详解：4000 行代码如何打造一个超轻量个人 AI 智能体

2026-07-12
15.2k- 59m
课程笔记-学习Agent大模型开源源码

2026 年初，一个叫 Nanobot 🐈 的开源项目在几天内冲上数千 star，理由很有意思：它宣称用约 4000 行 Python 就复现了那个 43 万行体量的 OpenClaw 的核心智能体能力，代码量只剩原版的 1%。在”大模型应用 = 重型框架 + 海量依赖”几乎成为默认想象的今天，这种”小而可读”的反向工程美学格外吸引人。

这篇笔记要把 Nanobot 从架构到源码讲透。它的写法不是把模块拆开来一个个罗列，而是顺着一条消息在代码里的执行路径一路走下去：消息怎么进来的（run()）、怎么被处理的（_process_message()）、上下文是怎么拼出来的（build_messages()）、记忆和技能怎么挂进上下文、工具是怎么被调用和执行的（loop.py 的工具循环 + registry.execute()）。跟着这条路走一遍，你就能把 Nanobot 的核心彻底看懂。内容综合自官方仓库（HKUDS/nanobot）的 README 与架构文档，以及 loaf 和掘金上的两篇源码逐行解读，并对若干不准确的说法做了核对修正。

关于版本：Nanobot 迭代极快（2026 年 2 月 2 日发布，到 6 月已到 v0.2.2）。源码逐行解读类文章多基于 v0.1.x 早期的代码（那时 Agent 逻辑集中在一个 loop.py、Provider 用 litellm、记忆是按天的 Markdown 文件）。当前 v0.2.x 已经把循环拆成 AgentLoop + AgentRunner、移除了 litellm、记忆改成了 JSONL 会话加 Dream 巩固。本文会同时呈现两者：用早期可读性最强的代码讲清”一条消息的旅程”，再用官方当前架构标注演进后的差异，这样你既能看懂代码细节，也不会被过时信息误导。

Nanobot 是什么：定位与背景

先纠正一个流传很广的误解。不少中文介绍把它说成”香港大学数据智能实验室（HKUDS）集体用 4000 行代码复现了 OpenClaw”。查官方仓库的 README 会发现更准确的表述：这个项目由 Xubin Ren（GitHub: re-bin）作为个人开源项目发起，以个人身份维护，只是托管在 HKUDS 这个组织下，并接受社区贡献。把它理解成”一位港大研究者的个人开源项目，挂在 HKUDS 组织下”会比”实验室集体作品”更贴近事实。

它的官方一句话定位是：一个开源、超轻量、你可以真正拥有的个人 AI 智能体（an open-source, ultra-lightweight personal AI agent you can truly own）。关键词是”拥有”，它强调自托管、可检查、可定制、不被某个巨型平台绑死。它和 OpenClaw / Clawdbot 的关系是：把后者的核心智能体思路（一个常驻后台、能跨聊天平台对话、能用工具、有记忆的个人助手）用极简的方式重新实现一遍。

关于”4000 行”这个最出圈的口号，需要如实说明：它是 2026 年 2 月初发布时（v0.1.x 早期）核心代码的真实规模，也是它能在 3 天冲到 5k star 的最大噱头。但随着功能快速扩张，到本文写作时的 v0.2.2，核心已经增长了不少。仓库里那个 core_agent_lines.sh 脚本会按目录如实统计：agent/ 顶层加 bus/、config/、cron/、session/ 等核心目录约 1.15 万行（core），tools/、channels/、cli/、utils/ 等扩展部分约 3.9 万行（extra）。也就是说”4000 行”是起点的口号，不是当前规模；但即便按 1.15 万核心行算，它仍不到 OpenClaw 43 万行的 3%，”小而可读”的定位没有变，这个脚本也把”小”做成了一个可量化、可追溯的承诺。MIT 协议，Python 3.11+。

理解 Nanobot，绕不开两种编程范式的对照。OpenClaw 常被当作 Vibe Coding（凭感觉大量生成代码）的典型：功能强大、迭代飞快、成本低廉，但代码冗余、架构臃肿、人类很难读懂。Nanobot 走的是相反的路：手工精制、对开发者要求更高，但源码极利于人类理解。这两种风格没有绝对优劣，OpenClaw 用”量”换”广度与速度”，Nanobot 用”克制”换”可读与可改”。对一个想真正吃透 Agent 内部原理的人来说，Nanobot 是极佳的学习样本，因为它的核心循环真的能在几十分钟内读完。下面我们就来读。

项目整体结构：一张目录图看懂全貌

先看一张整体流程图建立印象：用户从对话界面或 IM 平台发消息，NanoBot 调用大模型、加载技能、执行工具，再把结果返回。

接着把整个项目的实际目录结构摆出来（对应 v0.2.2，注释标明了每个子系统的职责）。这张图本身就是一张架构地图，下一节会沿着 agent/loop.py 这一行一路钻进去：

几个要点先记住：agent/ 目录下是实现核心智能体功能的代码，它通过 loop.py 监听请求、调用大模型、加载各项技能执行任务；providers/ 提供大模型接口；channels/ 管理通信渠道；skills/ 里内置了 12 个常用功能；gateway/ 是长跑守护进程，把通道、cron、心跳、Dream 等后台服务组装起来。相比早期 v0.1.x 只有 agent/providers/channels/bus/session/config/cron/skills/cli 这几个目录，当前版本把网关、API、SDK、安全、触发器、配对、音频、WebUI 都做成了独立的子系统，但核心循环仍然是 agent/loop.py + agent/runner.py 那一条主干，其余子系统都是挂在这条主干上的旁支。

围绕这个小核心，整个运行时可以用一条单向主干来表述（来自官方架构文档）。请把这条主干和后面 loop.py 那一节的代码走读对照着看，你会发现每一行代码都能落回这条主干上：

这是官方给出的架构图，和上面这条主干一一对应：围绕一个很小的 Agent 循环，消息从聊天平台进来，由 LLM 决定何时需要工具，记忆和技能只作为上下文被按需拉入，而不变成沉重的编排层。

它提供了多种进入方式。最简单的是 nanobot agent -m "Hello!"，一条入站消息走一遍 Agent 循环，在终端打印回复。长跑模式是 nanobot gateway，各聊天渠道从 IM 或 WebUI 收到消息，发布到同一条 Agent 循环，回复再送回来源渠道。此外还有兼容 OpenAI API 的 nanobot serve，以及浏览器工作台 nanobot webui。无论消息从 CLI、WebUI 还是 Telegram 进来，走的都是同一条循环逻辑，这正是它”一个核心、多个入口”思想的体现。

通过 agent/loop.py 理解 ReAct：一条消息的完整旅程

这一节是整篇文章的核心。我们会跟着一条消息，从它进入循环开始，一路追到工具执行完毕、回复发出去为止。中间每到一处调用别的模块（context.py 构建上下文、memory 取记忆、skills 加载技能、registry.py 执行工具），就就地展开讲清楚。

先交代版本：本节讲解的是 v0.2.x 的结构。当前版本把 Agent 的核心循环拆成了两个职责清晰的对象：

  • AgentLoop（agent/loop.py，面向通道）：负责一个”轮次”（turn）对外的全部事务：接收入站消息、确定本次对话归属的会话和工作区、构建上下文、挂载钩子与进度回调、维护通道元数据、把回复发布成出站消息。一句话：管”消息从哪个渠道来、写进哪个会话、回复怎么送回去”。
  • AgentRunner（agent/runner.py，面向模型）：负责真正和 LLM 打交道的内层循环：把消息发给 Provider、处理流式增量与推理块、执行工具调用、把工具结果喂回模型，直到给出最终回答或触及运行时上限。一句话：管”调 LLM、跑工具、流式、迭代次数”。

两者一外一内，合起来正是早期 v0.1.x 那个大而全的 loop.py 干的全部事。源码解读时把两者合在一起看更容易看清”一条消息的完整旅程”，所以下面我会用一个统一的 _process_message() 伪代码把两者的逻辑串起来讲，并在每一步明确标注它实际属于 AgentLoop 还是 AgentRunner。这样你既看懂了流程，也清楚每段在 v0.2.x 里落在哪个文件。

读完这一节，Nanobot 的核心运行机制就完全没有黑盒了。

先弄懂 ReAct：推理与行动交替的循环

ReAct 是 Reasoning（推理）+ Acting（行动）的缩写，它是一种让大语言模型能够”推理”和”行动”交替进行的架构模式。它的核心思想可以用四个动作概括：

  • Reason（推理）：模型分析问题，制定行动计划
  • Act（行动）：模型执行具体的工具或函数
  • Observation（观察）：获取工具执行的结果
  • Thought（思考）：基于观察结果更新推理

这个循环持续进行，直到任务完成或达到最大迭代次数。之所以要有”最大迭代次数”这个上限，是因为模型有可能陷入”不停调工具却给不出答案”的死循环，必须有个硬刹车。理解了这个循环，下面每一行代码都是在实现它。

_process_message()：处理一条消息的四步

_process_message() 把一条消息从”收到”到”回复”的全过程串起来。先用一张总流程图建立整体印象（这张图基于早期 v0.1.x 的统一 loop.py 绘制；v0.2.x 已把模型循环拆进 AgentRunner，但下面的流程逻辑完全不变）：

把它的核心逻辑提炼出来，可以清楚地分成四步：

第一步，创建或取出会话（AgentLoop）。每个会话由 msg.session_key 标识，一般是 channel:chat_id，这样不同人、不同群、不同渠道各自维护独立的历史。get_or_create 的语义是”有就取出来，没有就新建一个”。

第二步，更新工具上下文（AgentLoop）。这里有个很容易被忽略、但很关键的细节：消息有三种来源，分别是 message（普通消息发送）、spawn（子 Agent 生成）、cron（定时任务调度）。代码里会根据消息来源给相关工具挂上当前上下文，比如 spawn_tool.set_context(msg.channel, msg.chat_id) 就是告诉 spawn 工具”你生成的子 Agent 要把结果发回这个渠道的这个会话”。没有这一步，子 Agent 就不知道往哪里回话。

第三步，构建上下文（AgentLoop），调用 self.context.build_messages()。这一步会把系统提示、历史、记忆、技能、当前消息全部拼成一个消息列表，准备喂给 LLM。它背后牵出 context.py、记忆、技能三个模块，是这一节后面三个小节（构建上下文、记忆、技能）要展开的重点。

第四步，ReAct 循环（AgentRunner）。前三步都属于 AgentLoop 的”面向通道”职责；到这一步，AgentLoop 把拼好的消息交给 AgentRunner，由它跑真正的 ReAct 内循环：while iteration < self.max_iterations 就是 前面讲 ReAct 时说的那个”最大迭代次数”刹车（早期版本默认 20 次）。循环里先调用 LLM 推理（Reason），如果模型返回里有工具调用（has_tool_calls），就遍历每个工具调用去执行（Act），把结果塞回历史（Observe），进入下一轮；一旦模型不再要求工具、直接给出文本回答，就 break 跳出循环，把最终答案交回 AgentLoop 去发布。

注意第四步里那个工具执行的细节，它牵出 loop.py 末尾的工具调用循环和 registry.py，我们在后文“调用工具”和“工具注册表”两小节展开。下面先跟着第三步，去看看上下文是怎么拼出来的。

它内部最关键的是 build_system_prompt()，负责组装系统提示词。系统提示词不是一段写死的文字，而是由若干块模板拼接而成，模板就放在 nanobot/templates/ 下。下面把每一块的真实内容拿出来看，而不是只给一句概括。

第一块：核心身份（agent/identity.md）。它注入运行时信息和工作区路径，还会按当前渠道切换格式提示。比如同一套回答，在 Telegram/QQ/Discord 上会要求”用短段落、少用标题、不用表格”，在 WhatsApp 上要求”纯文本、不渲染 markdown”，在邮件上要求”分清章节、格式从简”，在 CLI 上要求”避免标题和表格”。这是个很巧妙的设计：一份人格，按渠道自适应表达。它还告诉模型工作区里有什么：

第二块：灵魂（SOUL.md）。这是 Agent 的自我认知和行事准则，默认内容相当有人格魅力（原文是英文）：

第三块：用户画像（USER.md）。记录用户的基本信息和偏好，是个勾选式模板：姓名、时区、语言、沟通风格（随意/专业/技术）、回答长度偏好、技术水平、工作背景。模型据此调整说话的口吻和深浅。

第四块：行为准则（AGENTS.md）。这是给 Agent 的工作流规范，比如”安排提醒前先看可用技能、用内置 cron 工具而不是 exec 调命令””不要把提醒直接写进 MEMORY.md（那样不会触发通知）””HEARTBEAT.md 由网关的心跳任务周期性检查，别重复造一个”等。它把”怎么做才对”的工程经验固化成了提示词。

第五块：工具契约。agent/tool_contract.md 告诉模型工具使用的通用规范，原文要点包括”用最窄的工具匹配任务””先只读探测再写””不要把 exec 当万能兜底””工具失败要换思路重试而不是重复同样的调用”。

第六块：技能（分两部分进上下文）。v0.2.2 的 build_system_prompt（见 agent/context.py）对技能是分两段处理的，不是一锅端：

也就是说，标记了 always: true 的技能会被加载全文作为”# Active Skills”块；其余技能只把摘要放进 agent/skills_section.md，模型据此判断该不该去读某个技能的 SKILL.md 正文。

此外还有两块：记忆上下文（从长期记忆 MEMORY.md 取出与当前相关的片段）和平台策略（platform_policy.md，按渠道约束 Agent 行为）。

看到这里就很清楚了：Nanobot 的人设不是代码写死的，而是这几份 Markdown 拼出来的。你不需要改任何代码，只要编辑工作区里的 SOUL.md / USER.md / AGENTS.md，就能把同一个 Nanobot 实例调教成风格迥异的角色；而 identity.md 里的渠道格式提示，让一份人格能在 Telegram、邮件、CLI 上各自得体地表达。系统提示词里的”记忆上下文”和”技能清单”两块，分别牵出记忆系统和技能系统，下面两节就地展开。

记忆在上下文里的角色：短期与长期

系统提示词第三块”记忆上下文”的数据，来自 Nanobot 的记忆系统。v0.2.x 的记忆分成相互独立又彼此配合的两层存储。

短期一层是会话（Session），存近期对话的逐条记录，作用是”回放进上下文”。它以 JSONL（JSON Lines）格式存放在 <workspace>/sessions/*.jsonl，每个会话一个文件。选 JSONL 而不是普通 JSON 是有讲究的：JSON 是一个整体结构（一个大数组 [{},{},{}]），而 JSONL 是一行一条（{}\n{}\n{}），这样在取”最近 N 条消息”时直接读文件末尾即可，非常高效。get_history() 的实现就体现了这一点：

注意它取出历史时主动剥离了 timestamp 这类内部字段，只保留 role 和 content，纯粹是为了减少喂给模型的 token。长期记忆则仍是 MEMORY.md（外加一个 history.jsonl 作为沉淀用的原始历史）。

把短期和长期串起来的是 Dream，一个周期性的记忆巩固任务。Dream 会定期读取累积的 history，提炼出有价值的内容，更新工作区的长期记忆，让有用的上下文能活得比短期会话回放更久。这有点像人类睡眠时整理记忆：白天聊的内容（短期）在”做梦”时被筛选、压缩、写进长期记忆。无论版本怎么变，记忆系统的内核始终是”短期负责当下连贯、长期负责跨会话沉淀、Dream 负责定期整理“这三层分工。

技能在上下文里的角色：SKILL.md

系统提示词第六块（技能）的两部分”# Active Skills / 可用技能摘要”，数据来自技能系统。和记忆一样，技能也是用纯文本保存的：每个技能在自己对应子目录下的 SKILL.md 文件里描述。

这里要先把一个最容易混淆的概念讲清楚：技能（Skill）和工具（Tool）不是一回事。技能是给模型看的”使用说明书”，用自然语言告诉模型”什么情况下用哪个工具、参数怎么填、有哪些示例”；工具才是实际干活的可执行代码。模型读了技能说明，生成一条命令，再交给对应工具去执行。

技能怎么挂进上下文？当一个用户消息到来、build_system_prompt 被调用时，会先通过 SkillsLoader.build_skills_summary() 扫描技能目录，读出每个技能的 name 和 description，把摘要塞进系统提示词的”可用技能”清单。模型据此判断”这次任务该不该用某个技能”。一旦决定用，再通过 skills.py 的 load_skill 加载该技能 SKILL.md 的正文，照正文里的示例生成命令。技能的写法和加载流程在后文详细展开，这里只要记住：技能以摘要形式进上下文，模型据此决定要不要加载正文。

对应关系小结：四步落在 v0.2.x 的哪个文件

把上面这条旅程的四个步骤，明确对应回 v0.2.x 的 AgentLoop / AgentRunner 两个文件：

  • 第一、二、三步（取会话、更新工具上下文、build_messages 构建上下文）加上最后的收发消息，都落在 **agent/loop.py 的 AgentLoop**，也就是”面向通道”的那一半。
  • 第四步（while ReAct 循环、调 Provider、执行工具、结果回写）落在 **agent/runner.py 的 AgentRunner**，也就是”面向模型”的那一半。

这样拆的意义是关注点分离：通道路由和模型调用是两类完全不同的问题，分开后调试方向非常明确：渠道路由、会话、出站投递相关的报错翻 loop.py；Provider 调用、工具调用、流式、迭代次数相关的报错翻 runner.py。但无论怎么拆，本节讲的那条”消息旅程”逻辑是不变的，所以用统一的 _process_message() 伪代码串起来讲并不会失真。

内置工具逐个看：Agent 的手脚

前面 loop.py 那一节里我们只追到了”工具怎么被调用和执行”（registry.execute() 的四步），但还没看每个工具内部具体干什么。这一节就把内置工具逐个拆开。当前 agent/tools/ 下已经有 20 多个工具文件，大致分成几类：文件类（filesystem.py 的读写编辑列目录、apply_patch.py 的补丁、file_state.py 的状态追踪、path_utils.py）、Shell 与执行类（shell.py、exec_session.py、sandbox.py 沙箱、long_task.py 长任务）、网络类（web.py 抓取、search.py 搜索）、通信类（message.py 发消息、spawn.py 生成子 Agent）、调度类（cron.py）、扩展能力类（mcp.py 接入 MCP 服务器、cli_apps.py 本地 CLI Apps、image_generation.py 图像生成）、运行时类（self.py 自检、runtime_state.py、registry.py 注册表、base.py/schema.py 基类与参数 schema）。工具从 agent/tools/ 目录和插件入口点自动发现，每个工具实现统一的基类和 schema，工具行为被当作”模型契约”的一部分（工具名、参数 schema、错误信息都要尽量稳定）。下面挑最核心的几个展开。

消息总线：用 asyncio.Queue 解耦通道与 Agent

前面 loop.py 那一节里 run() 只用 consume_inbound 和 publish_outbound 两个方法就和消息总线打交道了，完全不用管渠道细节。这一节就来看消息总线是怎么做到这一点的。先用一张图建立整体印象：两个异步队列把通道层和 Agent 层彻底隔开，两边只认 InboundMessage / OutboundMessage 两个标准结构。

消息总线定义在 bus/events.py（事件类型）和 bus/queue.py（队列实现）里，是整个系统的通信中枢，基于 Python 的 asyncio.Queue 实现。先看它的核心结构：

总线标准化了两类消息：InboundMessage（入站消息） 由各个 Channel（Telegram、WebUI 等）产生，流入 Agent，如果是系统通知则 channel 字段为 system；OutboundMessage（出站消息） 由 Agent 产生，发往 Channel。它在内部维护两个独立的异步队列（一个入站、一个出站），外加一份”按渠道分组的出站订阅者”列表。

它的核心方法分工清晰：publish_inbound 让渠道把消息推进入站队列（由 Channel 的 _handle_message 调用）；consume_inbound 让 AgentLoop 阻塞等待下一条入站消息（就是外层 run() 里那一行）；publish_outbound 让 Agent 把回复推进出站队列；subscribe_outbound 让某个渠道订阅属于自己的出站消息。出站分发是一个后台运行的 Task，不断从 outbound 队列取消息、按渠道分发给订阅者：

这样设计的好处是，通道层和 Agent 层彻底解耦：通道只管把外部平台的协议翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API；Agent 只管消费消息、产出回复。两边都只认识 InboundMessage/OutboundMessage 这两个数据结构，互不依赖对方的实现细节。这也是为什么 Nanobot 能轻松接入十几个不同的聊天平台，而不需要改动核心循环。

定时任务 Cron：自己实现的 asyncio 调度器

Nanobot 自己实现了一套定时任务系统，没有依赖系统自带的 crontab。先用一张图看清全链路：模型调 CronTool.execute 只是转发 add_job，真正干活的是 CronService（算下次时间 → asyncio 设定时器 → 到点 _on_timer 执行回调 → 持久化到 cron.json → 通过 MessageBus 发消息）。

核心文件是 cron/types.py（数据模型）和 cron/service.py（调度服务）。types.py 里定义了五个数据模型，分工明确：CronSchedule 定义调度规则、CronPayload 定义触发时执行什么、CronJobState 定义任务状态、CronJob 表示一个完整的定时任务、CronStore 负责把任务内容持久化到本地 JSON 文件（<workspace>/cron/jobs.json）。

CronService 是整个模块的大脑，负责任务生命周期和调度执行。它完全依赖 Python asyncio 事件循环（Event Loop）的内部定时器来唤醒任务，而不是依赖系统 cron。机制是这样的：CronService 算出离下一个任务还有多久（比如 delay_s 秒），然后创建一个内部任务 tick()：

心跳 Heartbeat：主动巡检的另一种思路

除了 Cron，Nanobot 还有第二种主动唤醒机制，即 Heartbeat 心跳。但这个”心跳”和传统网络保活的心跳不是一回事，它其实是一个定期唤醒代理去检查预定义任务的主动式服务。

需要先说明一个版本变化：在早期的 v0.1.x 里，心跳是独立的 heartbeat/service.py，里面硬编码了 DEFAULT_HEARTBEAT_INTERVAL_S = 30*60、HEARTBEAT_PROMPT、HEARTBEAT_OK_TOKEN 这些常量。到了 v0.2.x，独立的 heartbeat/ 目录已经没有了，心跳被改造成网关（gateway）注册的一个受保护的系统任务（和 Dream 同级），由 gateway.heartbeat.enabled 开关控制，复用 cron/ 与 agent/automation_turns.py 这套自动化轮次基础设施来调度，而不再是一个单独的服务文件。这种”把心跳收进网关的系统任务”的做法，让它和 Cron、Dream 共用同一套调度与持久化机制，整体更统一。

但它的核心行为没变：定期唤醒 Agent，让它去读工作区里的 HEARTBEAT.md，检查 ## Active Tasks 下有没有要执行的任务；如果有就执行，并把有用的、可操作的结果发给最近活跃的聊天目标；如果没什么值得汇报的，就静默不发（而不是每次都打扰你）。这一点和早期版本的”回一句 HEARTBEAT_OK 才静默”相比，更省 token、更不扰民。

心跳和 Cron 看起来都在”定时唤醒”，但适用场景完全不同，对照着看最清楚：

特性 Heartbeat（心跳） Cron（定时任务）
触发源 文件内容（HEARTBEAT.md 的 Active Tasks） 时间规则（Cron 表达式 / 间隔）
执行内容 不确定，取决于文件里临时写了什么 确定，Payload 里预定义的提示词
持久化 无需持久化，实时读文件 需持久化（jobs.json）
结果投递 只在”有可操作结果”时发，否则静默 时间到就执行并按配置投递
交互性 人机异步协作（你随时往文件里加任务） 机器自动化
成本控制 高，文件空就不浪费 LLM 低，时间到必须唤醒 LLM 执行
代码位置 gateway 的系统任务 + automation_turns cron/（CronService）

你可以把 HEARTBEAT.md 当成”给 Agent 的临时交办清单”：运行中任何时候都能往里加任务，下一次心跳时它就会被捡起来执行。而 Cron 更像”定死的闹钟”。理解了它们的差异，才能在对的场景选对的工具。

技能系统：SKILL.md 的写法与加载流程

前面讲“技能在上下文里的角色”时我们说了技能是怎么以摘要形式进上下文的，这一节展开技能本身的写法和完整加载流程。每个技能是一个目录，里面必须有一个 SKILL.md。这个文件头部是 YAML frontmatter（前置元数据），声明技能的元信息。以内置的 GitHub 技能为例：

frontmatter 各字段的含义：name（必填）是技能名（目录名）；description（必填）是技能描述，模型就是根据它判断何时使用这个技能的；metadata.nanobot.emoji 是图标；requires.bins 是依赖的可执行程序；requires.env 是依赖的环境变量；always 表示是否始终加载到上下文；install 是依赖未满足时的安装提示。frontmatter 之后是正文，用大量”想做什么 → 用什么命令”的示例告诉模型操作范式，模型据此举一反三。

把前面的摘要加载和这里的正文加载串起来，技能的完整加载流程是这样的。当用户说”列出某仓库的 issue”：第一步，build_system_prompt 调用 SkillsLoader.build_skills_summary() 扫描技能目录，读出每个技能的 name 和 description，把摘要塞进系统提示词；第二步，因为 GitHub 技能的 description 里提到了 issue，模型判断它能满足需求；第三步，通过 skills.py 的 load_skill 加载 SKILLS.md 的正文；第四步，照正文里的示例生成一条 gh issue list 命令；第五步，交给 shell.py 的 execute() 执行；第六步，命令返回 JSON 或文本结果，Agent 整理后反馈给用户。

技能有两套来源：内置的 nanobot/skills/（v0.2.2 自带 12 个：github、weather、tmux、cron、summarize、skill-creator、memory（即 Dream 记忆整理）、long-goal（即 /goal 长期目标）、image-generation、clawhub（从 ClawHub 技能市场搜索安装技能）、update-setup、my）和用户工作区 <workspace>/skills/ 下的自定义技能，工作区技能优先级更高。这意味着你可以不改源码、只往工作区扔一个写好 SKILL.md 的目录，就给 Agent 加了新能力：

Provider 抽象：模型自由，且已移除 litellm

这里必须做一个重要纠正。早期的源码解读文章（写于 2026 年 2 月中旬）会说 Nanobot 用 litellm 库做统一 Provider 接口，对应的实现是 litellm_provider.py。这在当时是对的：litellm 是一个能把不同 LLM 提供商的 API 标准化成 OpenAI 格式的统一接口库，好处是不用为每个模型写单独的类，坏处是支持的模型受第三方限制。下面这张就是 litellm 官方文档的”Getting Started”页，它自我介绍的核心卖点就是”用 OpenAI 输入输出格式调用 100+ 个 LLM”——这也正是早期 Nanobot 选用它的原因：

但这已经过时了：2026 年 3 月 21 日，项目把 litellm 移除，改用原生的 openai 和 anthropic SDK（官方 News 里有明确记录 “Replace litellm with native openai + anthropic SDKs”）。原因是 litellm 这类统一层虽然省事，但对厂商特有能力（如 Anthropic 的 prompt caching、OpenAI 的 reasoning）支持滞后。改成原生 SDK 后，既减少了依赖，又能更好地利用每家模型的新特性。

不过源码里还保留了一段判断 self.is_vllm = bool(api_base) and not self.is_openrouter，说明它本就支持本地模型和自建模型。想用本地模型（如 Ollama）很简单：配置一个 OpenAI 兼容的 Provider，把 apiBase 指向本地服务地址（如 http://localhost:11434/v1，这里 v1 后缀是 OpenAI 模式的历史习惯，与版本号无关），apiKey 随便填一个非空字符串即可。当前 providers/ 目录下的实现已经清晰地体现了”原生 SDK”路线：openai_compat_provider.py 是通用 OpenAI 兼容实现（覆盖 DeepSeek、Qwen、Kimi、MiniMax、vLLM 等绝大多数厂商），anthropic_provider.py、azure_openai_provider.py、bedrock_provider.py、openai_codex_provider.py、github_copilot_provider.py 是各自的原生专门路径，再由 factory.py 统一组装、registry.py 集中维护元数据、fallback_provider.py 实现主模型挂了自动切备用模型。

模型选择方面，当前推荐用**命名的 modelPresets**（预设）来管理。一个预设包含 label、provider、model、maxTokens、contextWindowTokens、temperature 等，由 agents.defaults.modelPreset 指定当前激活的，还支持 fallbackModels（备用模型，主模型挂了自动切）。这种”预设 + 兜底”的设计比早期直接配 provider/model 更适合生产环境。

通道集成：一个 Agent，19 个聊天平台

前面消息总线那一节里把渠道细节屏蔽了，这一节看渠道层自己怎么工作。通道（Channel）负责把外部平台翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API。先看通道层的整体结构：ChannelManager 持有所有启用的通道、负责出站消息的分发调度，各具体通道都继承自抽象基类 BaseChannel。

BaseChannel 这个抽象基类定义了所有通道必须实现的契约（start / stop / send，外加 _handle_message 把外部消息转成 InboundMessage 推给 MessageBus），以及 InboundMessage / OutboundMessage 这两个标准消息结构：

值得一提的是 WebUI。它直接打包在发布的 wheel 里，不需要额外构建步骤。启动 nanobot webui 会准备好本地 WebSocket 通道、启动网关、打开 http://127.0.0.1:8765。这里有个容易混淆的端口区分：WebUI 走的是 WebSocket 通道的 8765 端口，而不是网关健康检查的 18790 端口，18790 只是个 /health 端点，不服务浏览器。WebUI 是个完整的浏览器工作台，支持多会话、工作区控制、Apps、Skills、Automations 和设置面板。

CLI 命令：六个入口

cli/commands.py 是 Nanobot 的命令行入口，用 typer 库定义命令、用 rich 库美化输出。它有六个主要命令，理解它们就理解了 Nanobot 的全部使用方式。先用一张图看清这六个命令各自做什么：

其中 gateway 是最复杂的一个，它是所有后台服务的中枢。nanobot gateway 启动时会先加载配置，然后并行创建并启动一组服务：MessageBus、LLMProvider、CronService、AgentLoop、HeartbeatService、ChannelManager，再让 Agent.run() / Cron.start() / Channels.start_all() / Heartbeat.start() 同时跑起来：

**1. onboard**：初始化配置和工作空间。它的作用是从零构建 Agent 的运行环境（配置 + 目录）和认知基础（人设 + 记忆），执行完后一个”白板”Agent 就诞生了。它会在用户目录下创建配置文件（如 ~/.nanobot/config.json），再调用 _create_workspace_templates() 在工作区内创建初始文件，包括三个 Markdown 模板和一个记忆系统：

再加上工作区下的 memory/ 子目录和默认的 MEMORY.md（长期记忆）。这就相当于定义了这个代理的人设和记忆。注意 Nanobot 的设计是单工作空间的（即默认只能有一个”人设”），但通过命令行的 --config 和 --workspace 参数可以指定不同工作区，从而运行多个实例模拟多个人格。

**2. gateway**：启动后台服务网关（核心守护进程）。它是系统的核心，组装并启动所有后台服务：启用的聊天通道、WebSocket 通道、workspace 级 cron 服务、Dream 和心跳等系统任务、健康端点。

**3. agent**：直接与 Agent 对话，支持单次模式（-m）或交互模式。它和 gateway 在代码上有许多共同之处（都要创建消息、初始化 Provider、构建 AgentLoop），区别在于 gateway 由网关进行消息路由，agent 则是直接交互。

**4. channels**：通道命令。status 查看各渠道连接状态，login 启动 Bridge 服务并显示二维码（用于 WhatsApp 登录，这个 Bridge 是一个 Node.js 端，模拟网页版 WhatsApp，所以需要扫码）。

**5. cron**：定时任务命令，list/add/remove/enable/disable，详见后文 Cron 一节。

安全边界：让 Agent 自由但不失控

让一个能执行 Shell、能读写文件、能联网的 Agent 常驻后台，安全是不可回避的话题。Nanobot 把安全相关代码集中在 security/ 和若干工具文件里，划出了几条明确的边界。

第一条是工作区作用域（security/workspace_access.py、workspace_policy.py）：文件工具的读写被限制在工作区范围内，防止 Agent 随意碰工作区之外的文件。第二条是 Shell 沙箱（agent/tools/shell.py）：除了前面 Shell 工具讲的正则黑名单 deny_patterns，后续版本引入了更正式的沙箱机制和可配置的命令允许列表，让 Shell 执行更可控。第三条是 SSRF/网络检查（security/network.py、agent/tools/web.py）：Web 工具在抓取时会做 SSRF 防护，避免被诱导去访问内网地址（如云元数据服务 169.254.169.254、127.0.0.1、内网网段）。第四条是通道访问控制：对生产环境或共享聊天 App，要配置 allowFrom、配对（pairing）或 WebSocket token，防止陌生人通过 Telegram 把你的 Agent 当成免费算力或攻击跳板。

需要强调的是，这些边界是”层层设防”而不是”绝对安全”。前面提到的正则命令过滤的脆弱性就是一个例子，它挡得住误操作，挡不住有意绕过。所以 Nanobot 的安全设计哲学更接近”默认安全 + 可审计 + 可收紧”，而不是”绝对不可能出错”。在共享或生产环境部署时，务必要把通道访问控制和工作区限制配置好。

配置与工作区：实例的两半

最后讲一个贯穿全局却容易被忽略的概念：配置（config）和工作区（workspace）的分离。一个 Nanobot 实例默认住在 ~/.nanobot/ 下，分成两半。~/.nanobot/config.json 是实例配置，声明这个实例”可以用什么”，包括 Provider、模型默认值、通道、工具、网关、API、运行时选项。~/.nanobot/workspace/ 是Agent 工作区，存放这个实例”为自己保留的状态”，包括记忆、会话、心跳任务、Cron 任务、技能、生成的产物。一句话，配置控制”能力边界”，工作区存放”自我认知与记忆”。

路径 含义
~/.nanobot/config.json 实例配置：providers、模型默认值、channels、tools、gateway、API、运行时
~/.nanobot/workspace/ Agent 工作区：memory、sessions、heartbeat、cron、skills、生成产物

这种分离有一个直接好处：你可以用命令行参数指定不同的配置和工作区，从而运行多个完全独立的 Nanobot 实例（多个人格）：

两个实例有各自的人设（SOUL.md）、各自的记忆（MEMORY.md）、各自的会话、各自的端口，互不干扰。配合 unifiedSession 选项（多个通道共享一个会话），还能实现”一个用户、多个设备、同一份记忆”的体验。

小结：小核心的胜利

回到开头那个数字：4000 行，是 Nanobot 2026 年 2 月发布时核心代码的规模，也是它最出圈的口号；到 v0.2.2，核心已经长到约 1.15 万行（仍不到 OpenClaw 43 万行的 3%）。但即便规模翻了几倍，Nanobot 的意义仍然不在于”功能比 OpenClaw 多”，恰恰相反，它的功能比 OpenClaw 少得多。它的真正价值在于证明了 Agent 的核心其实很小：一个 asyncio 编排的循环、一对入站/出站消息队列、一个 ReAct 内循环、几个可插拔的模块（通道、Provider、工具、技能、记忆），就足以支撑一个能跨平台对话、会用工具、有记忆、能主动唤醒的个人助手。

如果你记住了前面 loop.py 那一节那条”一条消息的旅程”，就等于抓住了它的全部主干：run() 等消息 → _process_message() 取会话、设上下文、build_messages() 拼提示词（身份 / 引导文件 / 记忆 / 技能）→ 进入 ReAct while 循环调 LLM → 有工具调用就交给 registry.execute() 的”查找/校验/执行/兜底”四步、结果塞回历史 → 直到模型给出最终答案 → 回复发回消息总线 → 渠道投递给用户。围绕这条主干，Nanobot 的设计原则可以浓缩成几条：面向通道和面向模型的循环要分开（AgentLoop 与 AgentRunner）；用消息总线解耦通道和 Agent；工具执行做异常兜底，让模型有自我修正的机会；记忆分短期/长期/Dream 三层；技能是给模型看的说明书、工具是干活的代码；Cron 管确定的时间规则、Heartbeat 管不确定的文件巡检；配置控制能力边界、工作区存放自我。

更宏观地看，Nanobot 是对”Vibe Coding”范式的一种反思。当 AI 生成的代码越堆越多、系统越来越难被人类理解时，一个”手工精制、小而可读”的实现就有了独特价值：它不只是工具，更是教材。读它的源码，你能看清一个真实 Agent 从”消息进来”到”回复出去”之间发生的每一件事，而没有被层层抽象和海量依赖淹没。这种”看得懂、改得动”的可控性，正是它官方那句”an AI agent you can truly own”的底气所在。
