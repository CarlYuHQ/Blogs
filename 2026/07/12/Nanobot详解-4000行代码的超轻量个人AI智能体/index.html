Nanobot 详解：4000 行代码的超轻量个人 AI 智能体

2026-07-12
7.7k- 28m
课程笔记-学习Agent大模型开源源码

2026 年初，一个叫 Nanobot 🐈 的开源项目在几天内冲上数千 star，理由很有意思：它宣称用约 4000 行 Python 就复现了那个 43 万行体量的 OpenClaw 的核心智能体能力，代码量只剩原版的 1%。在”大模型应用 = 重型框架 + 海量依赖”几乎成为默认想象的今天，这种”小而可读”的反向工程美学格外吸引人。

这篇笔记要把 Nanobot 讲透：它到底是什么、整体架构如何、核心的 Agent 循环是怎么运转的，以及记忆、工具、技能、定时任务、通道集成这些关键模块的设计思路。内容综合自 Nanobot 官方仓库（HKUDS/nanobot）的 README 与架构文档，以及两篇公开的源码逐行解读，并对其中若干不够准确的说法做了核对修正。

关于版本：Nanobot 迭代极快（2026 年 2 月 2 日发布，到 6 月已到 v0.2.2）。本文描述的是 v0.2.x 时期的设计，文末会特别指出几个早期解读文章里已经过时、需要修正的点。

一、Nanobot 是什么：定位与背景

先纠正一个流传很广的误解。不少中文介绍把它说成”香港大学数据智能实验室（HKUDS）集体用 4000 行代码复现了 OpenClaw”。查官方仓库的 README 会发现更准确的表述：这个项目由 Xubin Ren（GitHub: re-bin）作为个人开源项目发起，以个人身份维护，只是托管在 HKUDS 这个组织下，并接受社区贡献。把它理解成”一位港大研究者的个人开源项目，挂在 HKUDS 组织下”会比”实验室集体作品”更贴近事实。

它的官方一句话定位是：一个开源、超轻量、你可以真正拥有的个人 AI 智能体（an open-source, ultra-lightweight personal AI agent you can truly own）。关键词是”拥有”，它强调自托管、可检查、可定制、不被某个巨型平台绑死。它和 OpenClaw / Clawdbot 的关系是：把后者的核心智能体思路（一个常驻后台、能跨聊天平台对话、能用工具、有记忆的个人助手）用极简的方式重新实现一遍，体积只有对方的 1% 左右（仓库里甚至有一个 core_agent_lines.sh 脚本专门用来统计核心代码行数）。MIT 协议，Python 3.11+，发布几天就冲到 5k star，此后一路涨到数万。

理解 Nanobot，绕不开两种编程范式的对照。OpenClaw 常被当作 Vibe Coding（凭感觉大量生成代码）的典型：功能强大、迭代飞快、成本低廉，但代码冗余、架构臃肿、人类很难读懂。Nanobot 走的是相反的路：手工精制、对开发者要求更高，但源码极利于人类理解。这两种风格没有绝对优劣，OpenClaw 用”量”换”广度与速度”，Nanobot 用”克制”换”可读与可改”。对一个想真正吃透 Agent 内部原理的人来说，Nanobot 是极佳的学习样本，因为它的核心循环真的能在几十分钟内读完。

二、整体架构：一个小核心加多个入口

Nanobot 的设计哲学浓缩在官方架构图里：围绕一个很小的 Agent 循环组织一切，消息从聊天平台进来，由 LLM 决定何时需要工具，记忆和技能只作为上下文被按需拉入，而不变成沉重的编排层。这样既让核心路径可读、易扩展，又能不断叠加通道、工具、记忆和部署选项，而不至于演化成单体怪兽。

用官方文档里的核心数据流来表述，整个运行时可以画成一条单向主干，外加一个状态读写旁路：

其中 AgentLoop 还会旁路地读写一组状态：Session（会话历史）、Memory（长期记忆）、Hooks（生命周期钩子）、Skills（技能）、Templates（模板）。这条主干几乎就是 Nanobot 的全部：没有调度框架、没有消息中间件、没有插件总线，就是一个 asyncio 编排的循环加几个队列。

围绕这个小核心，它提供了多种进入方式。最简单的是 nanobot agent -m "Hello!"：一条入站消息走一遍 Agent 循环，在终端打印回复。长跑模式是 nanobot gateway：各聊天渠道从 IM 或 WebUI 收到消息，发布到同一条 Agent 循环，回复再送回来源渠道。此外还有兼容 OpenAI API 的 nanobot serve，以及浏览器工作台 nanobot webui。无论消息从 CLI、WebUI 还是 Telegram 进来，走的都是同一条循环逻辑，这正是它”一个核心、多个入口”思想的体现。

入口 命令 用途
CLI 单次 nanobot agent -m "..." 首次验证、脚本、快速本地问答
CLI 交互 nanobot agent 终端对话，带持久会话历史
网关 nanobot gateway 聊天 App、WebUI、心跳、Dream、长跑服务
OpenAI 兼容 API nanobot serve 通过 /v1/chat/completions 编程访问
WebUI nanobot gateway 加 WebSocket 通道 浏览器工作台，端口 8765

理解了这条主干和入口，下面逐层拆开看每个关键模块。

三、AgentLoop 与 AgentRunner：双循环设计

这是理解 Nanobot 运行时最关键的一点，也是早期源码解读文章里最容易讲偏的地方。在 Nanobot 较早的版本里，整个 Agent 逻辑确实集中在一个 agent/loop.py 里；但当前的 v0.2.x 已经把它拆成了两个职责清晰的循环：AgentLoop 和 AgentRunner。

AgentLoop（agent/loop.py）面向通道，负责一个完整的”轮次”（turn）对外的全部事务：接收入站消息、确定本次对话归属的会话和工作区、构建上下文、挂载钩子（hooks）与进度回调、维护通道元数据，最后把回复发布成出站消息。换句话说，但凡涉及”这条消息从哪个渠道来、要写进哪个会话、回复怎么送回去”的问题，都归 AgentLoop 管。

AgentRunner（agent/runner.py）面向模型，负责真正和 LLM 打交道的内层循环：把消息发给选定的 Provider、处理流式增量（streaming delta）和推理块（reasoning blocks）、执行模型请求的工具调用、把工具结果喂回模型，直到模型给出最终回答或触及运行时上限才停止。但凡涉及”Provider 调用、工具调用、流式、迭代次数”的问题，都归 AgentRunner 管。

为什么要这样拆？因为这是两种完全不同的关注点。通道路由、会话键、工作区选择、出站投递是一类问题；Provider 调用、工具执行、流式、迭代上限是另一类问题。把它们分到两个对象里，调试时方向非常明确：报错如果是渠道路由或会话相关的，去翻 agent/loop.py；如果是模型调用或工具相关的，去翻 agent/runner.py。这种”面向通道的轮次”和”面向模型的循环”的分离，是从早期单文件 loop.py 演进而来的重要重构，也是 Nanobot 在保持小体积的同时不被功能淹没的一个关键设计。

四、消息总线：用 asyncio.Queue 解耦通道与 Agent

AgentLoop 之所以能”面向通道”而不管通道细节，靠的是中间一层消息总线（MessageBus）。它定义在 nanobot/bus/events.py 和 nanobot/bus/queue.py 里，是整个系统的通信中枢，基于 Python 的 asyncio.Queue 实现。

总线标准化了两类消息：InboundMessage（入站消息） 由各个 Channel（Telegram、WebUI 等）产生，流入 Agent；OutboundMessage（出站消息） 由 Agent 产生，发往 Channel。MessageBus 在内部维护两个独立的异步队列，一个入站、一个出站，并维护一份”按渠道分组的出站订阅者”列表。

它的核心方法分工很清晰：publish_inbound 让渠道把消息推进入站队列；consume_inbound 让 AgentLoop 阻塞等待下一条入站消息；publish_outbound 让 Agent 把回复推进出站队列；subscribe_outbound 让某个渠道订阅属于自己的出站消息。这样设计的好处是，通道层和 Agent 层彻底解耦：通道只管把外部平台的协议翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API；Agent 只管消费消息、产出回复。两边都只认识 InboundMessage/OutboundMessage 这两个数据结构，互不依赖对方的实现细节。这也是为什么 Nanobot 能轻松接入十几个不同的聊天平台，而不需要改动核心循环。

五、ReAct 模式：推理与行动的循环

AgentRunner 内层循环的本质，就是经典的 ReAct（Reasoning + Acting）模式。ReAct 让大语言模型能够”推理”和”行动”交替进行：模型先分析问题、制定行动计划（Reason），再执行具体工具（Act），观察工具返回的结果（Observe），基于观察更新推理（Thought），如此循环，直到任务完成或达到最大迭代次数。

把这套模式映射到代码上，AgentRunner 的一次迭代大致是这样的：先调用 Provider 把当前消息序列发给 LLM（推理）；拿到响应后判断其中是否包含工具调用（tool_calls）；如果有，就遍历每个工具调用，通过工具注册表（registry.execute）执行，再把执行结果作为新的消息追加回历史；然后进入下一轮迭代，再次调用 LLM。这个过程有一个”最大迭代次数”的硬上限（早期版本默认 20 次），防止模型陷入无限调用工具的死循环。

工具执行的链路本身也很值得看。当 AgentRunner 决定调用某个工具时，请求会进入 agent/tools/registry.py 的 execute 方法。这个方法做四件事：先在注册表里按名字查找工具（找不到就返回友好错误）；再调用工具自己的 validate_params 做参数校验（schema 不匹配就提前拒绝）；接着才真正 await tool.execute(**params) 执行；最后用 try/except 兜住所有异常，把任何崩溃都转成字符串错误返回给模型，而不是让整个 Agent 崩掉。这种”工具失败也是模型可读的反馈”的设计，让模型有机会在工具报错后自我修正、换一种方式再试。

六、上下文构建：把”我是谁、我知道什么、我能做什么”拼成提示词

每一次调用 LLM 之前，都要把上下文组装好。这件事在 agent/context.py 的 build_messages（以及内部的 build_system_prompt）里完成。理解这一段，基本就理解了 Nanobot 是怎么”塑造”一个 Agent 的。

系统提示词由若干块拼成。第一块是核心身份：当前时间、工作区路径等基础信息。第二块是引导文件，也就是工作区里那几个 Markdown 文件，它们共同定义了这个 Agent 的人设：SOUL.md 定义自我认知（个性、价值观）、USER.md 记录用户画像（语言偏好、习惯）、AGENTS.md 是行为准则（”解释你的操作””善用工具”）、TOOLS.md 描述可用工具的使用要点，此外还有 IDENTITY.md 等。第三块是记忆上下文，从长期记忆里取出与当前相关的片段。第四块是始终加载的技能和可用的技能清单（技能系统后面单独讲）。最后是用户当前这条消息：如果带了图片或语音之类的多媒体，会被转成 base64 编码后一起塞进消息列表。

这种”用一堆 Markdown 文件定义人格和记忆”的设计是 Nanobot 最有人情味的地方。它意味着你不需要改任何代码，只要编辑工作区里的几个 .md 文件，就能把同一个 Nanobot 实例调教成风格迥异的角色。配合记忆系统，随着交互增加，Agent 对你的了解会越来越细，个性也会越来越稳定。

七、记忆与会话：短期回放 + 长期沉淀 + Dream 巩固

Nanobot 的”记忆”其实分成两个相互独立又彼此配合的存储。理解它们的分工，是理解 Agent 怎么”记住事”的关键。

第一个是会话（Session），存的是近期对话的逐条记录，作用是”回放进上下文”。它以 JSONL（JSON Lines）格式存放在 <workspace>/sessions/*.jsonl，每个会话一个文件。选 JSONL 而不是普通 JSON 是有讲究的：JSON 是一个整体结构（一个大数组），而 JSONL 是一行一条（{}\n{}\n{}），这样在取”最近 N 条消息”时直接读文件末尾即可，非常高效。会话还支持压缩（compaction）：当历史太长时，系统会把早期消息折叠成摘要，控制喂给模型的 token 量。每个通道会把入站消息映射到一个会话键（一般是 channel:chat_id），让不同人、不同群、不同渠道各自维护独立的历史。

第二个是长期记忆（Memory），存的是跨会话的事实和总结。主文件是 <workspace>/memory/MEMORY.md，另有一个 <workspace>/memory/history.jsonl 作为沉淀用的原始历史。和会话的”近期回放”不同，记忆是”长期沉淀”，保留那些值得跨会话记住的东西。

把这两者串起来的是 Dream，一个周期性的记忆巩固任务。Dream 会定期读取累积的 history，提炼出有价值的内容，更新工作区的长期记忆，让有用的上下文能活得比短期会话回放更久。这有点像人类睡眠时整理记忆：白天聊的内容（会话）在”做梦”时被筛选、压缩、写进长期记忆。Dream 在 agent/memory.py 里实现，由网关在开启时调度。

会话、记忆、Dream 三者一起，构成了一个从”秒级回放”到”长期人格”的连续光谱。会话让你在一段对话里保持连贯，记忆让你跨对话不忘事，Dream 让记忆不至于无限膨胀又能留下精华。

八、工具系统：Agent 的手脚

工具是 Agent 真正”做事”的能力。Nanobot 的工具从 agent/tools/ 目录和插件入口点自动发现，每个工具实现统一的基类和 schema。工具行为被当作”模型契约”的一部分：工具名、参数 schema、错误信息都要尽量稳定，否则模型会因为接口漂移而频繁出错。

内置的工具覆盖了一个个人助手最常用的几类能力。文件系统工具（filesystem.py）提供读、写、编辑、列目录等操作，主要用 pathlib.Path 做路径处理。Shell 执行工具（shell.py）用 asyncio.create_subprocess_shell 异步起子进程，用 communicate() 取 stdout/stderr。它有一个值得专门说的安全设计：_guard_command 方法用正则表达式维护一份危险命令黑名单（deny_patterns），拦截 rm -rf、dd if=、mkfs、fork bomb、关机重启等高危操作，也支持白名单（allow_patterns）。

不过必须指出这个设计的局限：基于正则的命令过滤本质上是脆弱的，攻击者（或一个足够”聪明”的模型）完全可以通过字符串拼接、变量替换、编码、alias 等方式绕过正则。所以这个黑名单只能算”防误触”而不是”防恶意”。真正要安全地让 LLM 执行 Shell，需要更强的隔离，这也是为什么后来的版本引入了更正式的沙箱机制（后面安全一节会讲）。

Web 工具（web.py）提供网络搜索和网页抓取，用 httpx 做异步 HTTP、readability-lxml 抽正文。早期版本封装的是 Brave Search API（需要 BRAVE_API_KEY），现在已扩展为多搜索引擎可切换。MCP 工具（mcp.py）让 Nanobot 能接入任何遵循 Model Context Protocol 的外部能力服务器。Cron 工具（cron.py）让模型自己能安排定时提醒和重复任务。此外还有图像生成、子 Agent 生成（spawn）、运行时自检（self） 等工具。子 Agent 工具特别有意思：它本身不干复杂逻辑，而是调用 SubagentManager 在后台拉起一个新的 AgentLoop 跑一个独立子任务，实现”Agent 套 Agent”。

九、技能系统：给模型看的”使用说明书”

技能（Skill）和工具（Tool）的关系，是初学者最容易混淆的概念。一句话区分：技能是给模型看的使用说明书，工具是实际干活的可执行代码。一个技能在 SKILL.md 文件里用自然语言告诉模型”什么情况下用哪个工具、参数怎么填、有哪些示例”，而真正的执行是模型根据这些说明生成一条命令、再交给对应工具去跑。

每个技能是一个目录，里面必须有一个 SKILL.md。这个文件头部是 YAML frontmatter（前置元数据），声明技能的 name、description（模型据此判断何时用它）、可选的图标、依赖（requires.bins 如 ["gh"]、requires.env）、是否始终加载（always）、以及安装指令。正文部分则是详细的使用指南，通常是一堆”想做什么 → 用什么命令”的示例，模型靠这些示例举一反三。

以内置的 GitHub 技能为例：它的 frontmatter 声明需要 gh CLI，正文分章节讲 Pull Requests、Advanced API Queries、JSON Output，每一节给出具体的 gh 命令范式。当用户说”列出某仓库的 issue”，Agent 的 context.build_system_prompt 会先调用 SkillsLoader.build_skills_summary 扫描所有技能目录，读出每个技能的 name 和 description；因为 GitHub 技能的 description 里提到了 issue，模型判断它能满足这个需求，于是加载该技能的正文，照里面的示例生成一条 gh issue list 命令，交给 Shell 工具执行，再把结果整理后回复用户。

技能有两套来源：内置的 nanobot/skills/（github、weather、tmux、cron、summarize、skill-creator 等）和用户工作区 <workspace>/skills/ 下的自定义技能，工作区技能优先级更高。这意味着你可以不改源码、只往工作区扔一个写好 SKILL.md 的目录，就给 Agent 加了新能力。这种”用 Markdown 扩展能力”的方式，把上一代”Skill 精准命中”问题（如何从海量技能里选对）部分化解了：技能数量不大时全部摘要进上下文，让模型自己判断就够了。

十、定时任务与心跳：两种”主动唤醒”机制

一个真正常驻的个人助手，不能只被动应答，还得能主动做事。Nanobot 提供了两种唤醒机制：Cron 定时任务和Heartbeat 心跳。它们看起来像，但适用场景不同，对照着看最清楚。

Cron 是 Nanobot 自己实现的一套定时任务系统，没有依赖系统自带的 crontab。任务持久化成本地 JSON 文件（<workspace>/cron/jobs.json），调度完全依赖 Python asyncio 事件循环的内部定时器：CronService 算出离下一个任务还有多久（比如 delay_s 秒），创建一个内部 tick() 任务 await asyncio.sleep(delay_s)；时间一到事件循环恢复执行，调用 _on_timer() 执行任务。数据模型分得很细：CronSchedule 定义调度规则、CronPayload 定义触发时执行什么、CronJobState 定义状态、CronJob 是一个完整任务、CronStore 负责持久化。用户可以用 CLI 的 cron add/list/remove/enable 管理任务，也可以让模型通过 Cron 工具自己加任务。

Heartbeat 则是另一种思路。它默认每 30 分钟主动唤醒一次 Agent，让它去读工作区里的 HEARTBEAT.md 文件，检查里面有没有 ## Active Tasks 下要执行的任务；如果没有，就回一句 HEARTBEAT_OK，不浪费 token；如果有，就执行并把有用的结果发给最近活跃的聊天目标。关键区别在于：心跳的触发源是文件内容，任务内容是不确定的（取决于你临时往文件里写了什么），而 Cron 的触发源是时间规则，任务内容是预先确定的。

特性 Heartbeat（心跳） Cron（定时任务）
触发源 文件内容（HEARTBEAT.md） 时间规则（Cron 表达式 / 间隔）
执行内容 不确定，取决于文件里写了什么 确定，Payload 里预定义的提示词
持久化 无需持久化，实时读文件 需持久化（jobs.json）
交互性 人机异步协作 机器自动化
成本控制 高，文件空就不唤醒 LLM 低，时间到必须唤醒 LLM 执行

另一个容易忽略的细节是心跳的任务执行期间计时器是暂停的：if self._running: await self._tick() 会停在这一行，等任务跑完才重新计 30 分钟。这避免了”上一次任务还没跑完，下一次又触发”的重复执行混乱。这恰恰是用心跳而非 Cron 来做”周期性巡检”的好处；如果硬用 Cron 做，任务一旦超过周期就会重叠。当然，你完全可以只用其中一种，但理解了它们的差异，才能在对的场景选对的工具。

十一、Provider 抽象：模型自由，且已移除 litellm

Nanobot 本身不是 AI，它只负责把上下文组织好发给后端大模型。这一层抽象在 nanobot/providers/ 下，元数据集中在 registry.py，配置字段在 config/schema.py。它支持 11 家以上的供应商：OpenAI 兼容的、Anthropic、Azure OpenAI、AWS Bedrock、OpenAI Codex（OAuth 登录）、GitHub Copilot、DeepSeek、Qwen、Kimi/Moonshot、MiniMax，以及本地的 Ollama、vLLM 等。

这里必须做一个重要纠正。早期的一些源码解读文章（写于 2026 年 2 月中旬）会说 Nanobot 用 litellm 库做统一 Provider 接口，对应的实现是 litellm_provider.py。这在当时是对的，但已经过时了：2026 年 3 月 21 日，项目把 litellm 移除，改用原生的 openai 和 anthropic SDK（官方 News 里有明确记录 “Replace litellm with native openai + anthropic SDKs”）。原因是 litellm 这类统一层虽然省事，但它支持的模型受第三方限制，且对厂商特有能力（如 Anthropic 的 prompt caching、OpenAI 的 reasoning）支持滞后。改成原生 SDK 后，既减少了依赖，又能更好地利用每家模型的新特性。当前大多数托管 Provider 走 OpenAI 兼容实现，而 Anthropic、Azure、Bedrock、Codex、Copilot 有专门路径。

模型选择方面，当前推荐用**命名的 modelPresets**（预设）来管理，而不是直接写 provider/model。一个预设包含 label、provider、model、maxTokens、contextWindowTokens、temperature 等，由 agents.defaults.modelPreset 指定当前激活的。预设还支持 fallbackModels（备用模型），主模型挂了自动切。这种”预设 + 兜底”的设计，比早期直接配 provider/model 更适合生产环境。想用本地模型也很简单：配置一个 OpenAI 兼容的 Provider，把 apiBase 指向本地服务地址（如 Ollama 的 http://localhost:11434/v1），apiKey 随便填一个非空字符串即可。

十二、通道集成：一个 Agent，十几个聊天平台

通道（Channel）层负责把外部平台翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API。基类契约在 channels/base.py，生命周期和发现机制在 channels/manager.py，各平台实现在 channels/*.py。通道通过内置模块扫描和插件入口点自动发现，所以加新通道不需要改核心代码。

当前支持的平台覆盖面相当广：CLI、WebUI/WebSocket、Telegram、Discord、Slack、飞书（Feishu/Lark）、企业微信/微信（WeChat/WeCom）、Email、Mattermost、DingTalk、QQ、Matrix、Microsoft Teams、Signal、WhatsApp 等。每个通道是否启用由配置文件决定；一旦实例跑起来，状态就是确定的，运行中改配置不会自动打开新通道，需要重启进程。

加一个新通道（比如微信）的标准流程是：写一个 channels/wechat.py，继承 BaseChannel 并实现三个抽象方法 start/stop/send；在 ChannelManager._init_channels 里注册它；并在配置 schema 里加上对应字段。这种”继承基类 + 三个方法”的极简扩展模型，正是 Nanobot”小核心 + 可插拔”哲学的体现。

值得一提的是WebUI。它直接打包在发布的 wheel 里，不需要额外构建步骤。启动 nanobot webui 会准备好本地 WebSocket 通道、启动网关、打开 http://127.0.0.1:8765。这里有个容易混淆的端口区分：WebUI 走的是 WebSocket 通道的 8765 端口，而不是网关健康检查的 18790 端口。18790 只是个 /health 端点，不服务浏览器。WebUI 是个完整的浏览器工作台，支持多会话、工作区控制、Apps、Skills、Automations 和设置面板。

十三、安全边界：让 Agent 自由但不失控

让一个能执行 Shell、能读写文件、能联网的 Agent 常驻后台，安全是不可回避的话题。Nanobot 把安全相关的代码集中在 nanobot/security/ 和若干工具文件里，划出了几条明确的边界。

第一条是工作区作用域（security/workspace_access.py、security/workspace_policy.py）：文件工具的读写被限制在工作区范围内，防止 Agent 随意碰工作区之外的文件。第二条是 Shell 沙箱（agent/tools/shell.py）：除了前面讲的正则黑名单，后续版本引入了更正式的沙箱机制和可配置的命令允许列表，让 Shell 执行更可控。第三条是 SSRF/网络检查（security/network.py、agent/tools/web.py）：Web 工具在抓取时会做 SSRF 防护，避免被诱导去访问内网地址（如 169.254.169.254 元数据服务、127.0.0.1、内网网段）。第四条是通道访问控制：对于生产环境或共享的聊天 App，要配置 allowFrom、配对（pairing）或 WebSocket token，防止陌生人通过 Telegram 把你的 Agent 当成免费算力或攻击跳板。

需要强调的是，这些边界是”层层设防”而不是”绝对安全”。前面提到的正则命令过滤的脆弱性就是一个例子，它挡得住误操作，挡不住有意绕过。所以 Nanobot 的安全设计哲学更接近”默认安全 + 可审计 + 可收紧”，而不是”绝对不可能出错”。在共享或生产环境部署时，务必要把通道访问控制和工作区限制配置好。

十四、配置与工作区：实例的两半

最后讲一个贯穿全局却容易被忽略的概念：配置（config）和工作区（workspace）的分离。一个 Nanobot 实例默认住在 ~/.nanobot/ 下，分成两半。

~/.nanobot/config.json 是实例配置：声明这个实例”可以用什么”，包括 Provider、模型默认值、通道、工具、网关、API、运行时选项。~/.nanobot/workspace/ 是Agent 工作区：存放这个实例”为自己保留的状态”，包括记忆、会话、心跳任务、Cron 任务、技能、生成的产物。一句话，配置控制”能力边界”，工作区存放”自我认知与记忆”。

这种分离有一个直接好处：你可以用命令行参数指定不同的配置和工作区，从而运行多个完全独立的 Nanobot 实例（即多个人格）：

两个实例有各自的人设（SOUL.md）、各自的记忆（MEMORY.md）、各自的会话、各自的端口，互不干扰。这正是早期”单工作区”设计被社区反复追问后，逐步在配置层补齐的能力。配合 unifiedSession 选项（多个通道共享一个会话），还能实现”一个用户、多个设备、同一份记忆”的体验。

十五、快速上手

把上面的设计落到实处，跑起来一个 Nanobot 只需要几步（Python 3.11+）。

安装（任选一种）：

初始化与配置：nanobot onboard 会创建 ~/.nanobot/config.json 和工作区，并在工作区里生成 SOUL.md、USER.md、AGENTS.md 和 memory/MEMORY.md 等模板文件。然后在 config.json 里配两块：Provider（apiKey、apiBase）和模型预设（modelPresets.primary 指定 provider/model，再用 agents.defaults.modelPreset 激活）。

运行：nanobot webui 打开浏览器工作台；nanobot agent 进交互式终端对话；nanobot agent -m "你好" 单次问答；nanobot gateway 起长跑服务接入聊天平台。nanobot status 可以看配置和工作区是否就绪。

十六、小结：小核心的胜利

回到开头那个数字：4000 行核心代码，复现 43 万行 OpenClaw 的核心智能体能力。Nanobot 的意义不在于”功能比 OpenClaw 多”；恰恰相反，它的功能比 OpenClaw 少得多。它的真正价值在于证明了 Agent 的核心其实很小：一个 asyncio 编排的循环、一对入站/出站消息队列、一个 ReAct 内循环、几个可插拔的模块（通道、Provider、工具、技能、记忆），就足以支撑一个能跨平台对话、会用工具、有记忆、能主动唤醒的个人助手。

它的设计可以被浓缩成几条原则。第一，面向通道和面向模型的循环要分开（AgentLoop 与 AgentRunner），让关注点不混杂。第二，用消息总线解耦通道和 Agent，两边只认统一的消息结构。第三，记忆分层：会话负责短期回放、MEMORY.md 负责长期沉淀、Dream 负责定期巩固，各司其职。第四，技能是给模型看的说明书、工具是干活的代码，用 Markdown 就能扩展能力。第五，两种主动唤醒机制分工：Cron 管确定的时间规则，Heartbeat 管不确定的文件巡检。第六，配置控制能力边界、工作区存放自我，两者分离才能支持多实例。

更宏观地看，Nanobot 是对”Vibe Coding”范式的一种反思。当 AI 生成的代码越堆越多、系统越来越难被人类理解时，一个”手工精制、小而可读”的实现就有了独特价值：它不只是工具，更是教材。读它的源码，你能看清一个真实 Agent 从”消息进来”到”回复出去”之间发生的每一件事，而没有被层层抽象和海量依赖淹没。这种”看得懂、改得动”的可控性，正是它官方那句”an AI agent you can truly own”的底气所在。
