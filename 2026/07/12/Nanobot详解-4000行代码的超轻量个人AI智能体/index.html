Nanobot 源码详解：4000 行代码如何打造一个超轻量个人 AI 智能体

2026-07-12
12.3k- 47m
课程笔记-学习Agent大模型开源源码

2026 年初，一个叫 Nanobot 🐈 的开源项目在几天内冲上数千 star，理由很有意思：它宣称用约 4000 行 Python 就复现了那个 43 万行体量的 OpenClaw 的核心智能体能力，代码量只剩原版的 1%。在”大模型应用 = 重型框架 + 海量依赖”几乎成为默认想象的今天，这种”小而可读”的反向工程美学格外吸引人。

这篇笔记要把 Nanobot 从架构到源码讲透。它的写法不是把模块拆开来一个个罗列，而是顺着一条消息在代码里的执行路径一路走下去：消息怎么进来的（run()）、怎么被处理的（_process_message()）、上下文是怎么拼出来的（build_messages()）、记忆和技能怎么挂进上下文、工具是怎么被调用和执行的（loop.py 的工具循环 + registry.execute()）。跟着这条路走一遍，你就能把 Nanobot 的核心彻底看懂。内容综合自官方仓库（HKUDS/nanobot）的 README 与架构文档，以及 loaf 和掘金上的两篇源码逐行解读，并对若干不准确的说法做了核对修正。

关于版本：Nanobot 迭代极快（2026 年 2 月 2 日发布，到 6 月已到 v0.2.2）。源码逐行解读类文章多基于 v0.1.x 早期的代码（那时 Agent 逻辑集中在一个 loop.py、Provider 用 litellm、记忆是按天的 Markdown 文件）。当前 v0.2.x 已经把循环拆成 AgentLoop + AgentRunner、移除了 litellm、记忆改成了 JSONL 会话加 Dream 巩固。本文会同时呈现两者：用早期可读性最强的代码讲清”一条消息的旅程”，再用官方当前架构标注演进后的差异，这样你既能看懂代码细节，也不会被过时信息误导。

一、Nanobot 是什么：定位与背景

先纠正一个流传很广的误解。不少中文介绍把它说成”香港大学数据智能实验室（HKUDS）集体用 4000 行代码复现了 OpenClaw”。查官方仓库的 README 会发现更准确的表述：这个项目由 Xubin Ren（GitHub: re-bin）作为个人开源项目发起，以个人身份维护，只是托管在 HKUDS 这个组织下，并接受社区贡献。把它理解成”一位港大研究者的个人开源项目，挂在 HKUDS 组织下”会比”实验室集体作品”更贴近事实。

它的官方一句话定位是：一个开源、超轻量、你可以真正拥有的个人 AI 智能体（an open-source, ultra-lightweight personal AI agent you can truly own）。关键词是”拥有”，它强调自托管、可检查、可定制、不被某个巨型平台绑死。它和 OpenClaw / Clawdbot 的关系是：把后者的核心智能体思路（一个常驻后台、能跨聊天平台对话、能用工具、有记忆的个人助手）用极简的方式重新实现一遍，体积只有对方的 1% 左右。仓库里甚至有一个 core_agent_lines.sh 脚本专门用来统计核心代码行数，把这个”小”做成了一个可量化、可追溯的承诺。MIT 协议，Python 3.11+，发布几天就冲到 5k star，此后一路涨到数万。

理解 Nanobot，绕不开两种编程范式的对照。OpenClaw 常被当作 Vibe Coding（凭感觉大量生成代码）的典型：功能强大、迭代飞快、成本低廉，但代码冗余、架构臃肿、人类很难读懂。Nanobot 走的是相反的路：手工精制、对开发者要求更高，但源码极利于人类理解。这两种风格没有绝对优劣，OpenClaw 用”量”换”广度与速度”，Nanobot 用”克制”换”可读与可改”。对一个想真正吃透 Agent 内部原理的人来说，Nanobot 是极佳的学习样本，因为它的核心循环真的能在几十分钟内读完。下面我们就来读。

三、通过 agent/loop.py 理解 ReAct：一条消息的完整旅程

这一节是整篇文章的核心。我们会跟着一条消息，从它进入 loop.py 开始，一路追到工具执行完毕、回复发出去为止。中间每到一处调用别的模块（context.py 构建上下文、memory 取记忆、skills 加载技能、registry.py 执行工具），就就地展开讲清楚。读完这一节，Nanobot 的核心运行机制就完全没有黑盒了。

3.1 先弄懂 ReAct：推理与行动交替的循环

ReAct 是 Reasoning（推理）+ Acting（行动）的缩写，它是一种让大语言模型能够”推理”和”行动”交替进行的架构模式。它的核心思想可以用四个动作概括：

  • Reason（推理）：模型分析问题，制定行动计划
  • Act（行动）：模型执行具体的工具或函数
  • Observation（观察）：获取工具执行的结果
  • Thought（思考）：基于观察结果更新推理

这个循环持续进行，直到任务完成或达到最大迭代次数。之所以要有”最大迭代次数”这个上限，是因为模型有可能陷入”不停调工具却给不出答案”的死循环，必须有个硬刹车。理解了这个循环，下面每一行代码都是在实现它。

真正干活的是下一行的 self._process_message(msg)，我们跟进去。

3.3 _process_message()：处理一条消息的四步

_process_message() 把一条消息从”收到”到”回复”的全过程串起来。把它的核心逻辑提炼出来，可以清楚地分成四步：

第一步，创建或取出会话。每个会话由 msg.session_key 标识，一般是 channel:chat_id，这样不同人、不同群、不同渠道各自维护独立的历史。get_or_create 的语义是”有就取出来，没有就新建一个”。

第二步，更新工具上下文。这里有个很容易被忽略、但很关键的细节：消息有三种来源，分别是 message（普通消息发送）、spawn（子 Agent 生成）、cron（定时任务调度）。代码里会根据消息来源给相关工具挂上当前上下文，比如 spawn_tool.set_context(msg.channel, msg.chat_id) 就是告诉 spawn 工具”你生成的子 Agent 要把结果发回这个渠道的这个会话”。没有这一步，子 Agent 就不知道往哪里回话。

第三步，构建上下文，调用 self.context.build_messages()。这一步会把系统提示、历史、记忆、技能、当前消息全部拼成一个消息列表，准备喂给 LLM。它背后牵出 context.py、记忆、技能三个模块，是这一节后面三个小节（3.4、3.5、3.6）要展开的重点。

第四步，Agent 循环，也就是 ReAct 的本体。while iteration < self.max_iterations 就是 3.1 节说的那个”最大迭代次数”刹车（早期版本默认 20 次）。循环里先调用 LLM 推理（Reason），如果模型返回里有工具调用（has_tool_calls），就遍历每个工具调用去执行（Act），把结果塞回历史（Observe），进入下一轮；一旦模型不再要求工具、直接给出文本回答，就 break 跳出循环。

注意第四步里那个工具执行的细节，它牵出 loop.py 末尾的工具调用循环和 registry.py，我们在 3.7 和 3.8 节展开。下面先跟着第三步，去看看上下文是怎么拼出来的。

3.4 构建上下文：build_messages() 与 build_system_prompt()

第三步调用的 build_messages()，源码在 agent/context.py。先看它的函数签名，参数本身就说明了它要拼接哪些东西：

它内部最关键的是 build_system_prompt()，负责组装系统提示词。系统提示词由五大块拼成：

  • 核心身份：身份、当前时间、工作空间路径等基础信息
  • 引导文件：AGENTS.md（行为准则）、SOUL.md（灵魂与个性）、USER.md（用户画像）、TOOLS.md（工具要点）、IDENTITY.md（身份信息）。这几个 Markdown 文件共同定义了这个 Agent 的人设
  • 记忆上下文：从长期记忆里取出与当前相关的片段
  • 始终加载的 Skills：标记了 always: true、每次都要带上的技能
  • 可用的技能：其余可选技能的清单

而在生成用户的当前消息时，如果带了多媒体（图片、语音），会被转成 base64 编码后加入消息列表，让多模态模型能”看见”。

这种”用一堆 Markdown 文件定义人格和记忆”的设计是 Nanobot 最有人情味的地方。它意味着你不需要改任何代码，只要编辑工作区里的几个 .md 文件，就能把同一个 Nanobot 实例调教成风格迥异的角色。系统提示词里的”记忆上下文”和”技能”两块，分别牵出记忆系统和技能系统，下面两节就地展开。

3.5 记忆在上下文里的角色：短期与长期

系统提示词第三块”记忆上下文”的数据，来自 Nanobot 的记忆系统。记忆用纯文本保存，分短期和长期两层。

早期 v0.1.x 的设计很直观：短期记忆是工作区 memory/ 目录下按天命名的 YYYY-MM-DD.md 文件（比如 2026-02-13.md），记录当天的交互要点；长期记忆则是工作区下那个直接叫 MEMORY.md 的文件，存跨天、跨会话沉淀下来的事实和总结。

到了 v0.2.x，这套机制变得更工程化，但思路没变：短期对话回放改成了会话（Session），以 JSONL 格式存放在 <workspace>/sessions/*.jsonl，每个会话一个文件。选 JSONL 而不是普通 JSON 是有讲究的：JSON 是一个整体结构（一个大数组 [{},{},{}]），而 JSONL 是一行一条（{}\n{}\n{}），这样在取”最近 N 条消息”时直接读文件末尾即可，非常高效。get_history() 的实现就体现了这一点：

注意它取出历史时主动剥离了 timestamp 这类内部字段，只保留 role 和 content，纯粹是为了减少喂给模型的 token。长期记忆则仍是 MEMORY.md（外加一个 history.jsonl 作为沉淀用的原始历史）。

把短期和长期串起来的是 Dream，一个周期性的记忆巩固任务。Dream 会定期读取累积的 history，提炼出有价值的内容，更新工作区的长期记忆，让有用的上下文能活得比短期会话回放更久。这有点像人类睡眠时整理记忆：白天聊的内容（短期）在”做梦”时被筛选、压缩、写进长期记忆。无论版本怎么变，记忆系统的内核始终是”短期负责当下连贯、长期负责跨会话沉淀、Dream 负责定期整理“这三层分工。

3.6 技能在上下文里的角色：SKILL.md

系统提示词第四、五块”始终加载的技能 / 可用的技能”，数据来自技能系统。和记忆一样，技能也是用纯文本保存的：每个技能在自己对应子目录下的 SKILL.md 文件里描述。

这里要先把一个最容易混淆的概念讲清楚：技能（Skill）和工具（Tool）不是一回事。技能是给模型看的”使用说明书”，用自然语言告诉模型”什么情况下用哪个工具、参数怎么填、有哪些示例”；工具才是实际干活的可执行代码。模型读了技能说明，生成一条命令，再交给对应工具去执行。

技能怎么挂进上下文？当一个用户消息到来、build_system_prompt 被调用时，会先通过 SkillsLoader.build_skills_summary() 扫描技能目录，读出每个技能的 name 和 description，把摘要塞进系统提示词的”可用技能”清单。模型据此判断”这次任务该不该用某个技能”。一旦决定用，再通过 skills.py 的 load_skill 加载该技能 SKILL.md 的正文，照正文里的示例生成命令。技能的写法和加载流程在第七节详细展开，这里只要记住：技能以摘要形式进上下文，模型据此决定要不要加载正文。

那 self.tools.execute(...) 内部又做了什么？跟进去看 registry.py。

3.8 工具注册表 registry.execute()：查找、校验、执行、兜底

工具注册表的 execute() 方法在 agent/tools/registry.py（约 L38-60），非常短，但做了一件结构极其清晰的事，值得逐行看：

四步：查找、校验、执行、异常兜底。最值得注意的是第 4 步的设计哲学：任何工具崩溃都被 try/except 兜住，转成字符串错误返回给模型，而不是让整个 Agent 崩掉。这意味着”工具失败”对模型来说是一种可读的反馈，模型有机会在工具报错后自我修正、换一种参数或换一个工具再试。这是 Agent 系统鲁棒性的关键一环，也呼应了 3.3 节那个 ReAct 循环为什么要”最多迭代 N 次”：模型可能要在几次失败后才能找到正确的工具用法。

执行完工具、把结果塞回历史后，控制权回到 3.3 节第四步的 while 循环，进入下一轮迭代，再次调用 LLM。如此往复，直到模型给出不再包含工具调用的最终答案，循环结束，_process_message() 返回一个 OutboundMessage，再被 3.2 节的 run() 发到消息总线的出站队列，最后由对应渠道把回复送到用户眼前。

3.9 一张图回顾全程

把上面这八个小节串起来，就是一条消息在 Nanobot 里的完整旅程：

用一句话回顾：**run() 等消息 → _process_message() 取会话、设上下文、build_messages() 拼提示词（含身份/引导文件/记忆/技能）→ 进入 ReAct while 循环调 LLM → 有工具调用就交给 registry.execute() 执行、结果塞回历史 → 直到模型给出最终答案 → 回复发回消息总线 → 渠道投递给用户**。这条路径之外的东西（Cron、Heartbeat、Provider 细节、通道协议），都是挂在这条主干上的旁支。

3.10 演进补注：AgentLoop 与 AgentRunner 的拆分

最后必须说明一个版本演进。上面这套 _process_message() 是 v0.1.x 早期、所有逻辑集中在一个 loop.py 时的形态，也是各大源码解读文章讲解的对象。当前 v0.2.x 把它拆成了两个职责清晰的循环：

AgentLoop（agent/loop.py）面向通道，负责一个完整”轮次”（turn）对外的全部事务：接收入站消息、确定本次对话归属的会话和工作区、构建上下文、挂载钩子与进度回调、维护通道元数据，最后把回复发布成出站消息，也就是 3.3 节里”第一、二、三步”加上收发消息那部分。

AgentRunner（agent/runner.py）面向模型，负责真正和 LLM 打交道的内层循环：把消息发给 Provider、处理流式增量和推理块、执行工具调用、把工具结果喂回模型，直到给出最终回答或触及运行时上限，也就是 3.3 节里”第四步的 ReAct while 循环”那部分。

换句话说，早期 _process_message() 的”会话与上下文”这部分变成了 AgentLoop，”ReAct 循环调 Provider 调工具”这部分变成了 AgentRunner。这样拆的意义是关注点分离：通道路由和模型调用是两类完全不同的问题，分开后调试方向非常明确，渠道路由相关报错翻 loop.py，模型/工具相关报错翻 runner.py。但无论怎么拆，3.1 到 3.9 讲的那条”消息旅程”逻辑是不变的。

这样设计的好处是，通道层和 Agent 层彻底解耦：通道只管把外部平台的协议翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API；Agent 只管消费消息、产出回复。两边都只认识 InboundMessage/OutboundMessage 这两个数据结构，互不依赖对方的实现细节。这也是为什么 Nanobot 能轻松接入十几个不同的聊天平台，而不需要改动核心循环。

六、定时任务 Cron：自己实现的 asyncio 调度器

Nanobot 自己实现了一套定时任务系统，没有依赖系统自带的 crontab。核心文件是 cron/types.py（数据模型）和 cron/service.py（调度服务）。types.py 里定义了五个数据模型，分工明确：CronSchedule 定义调度规则、CronPayload 定义触发时执行什么、CronJobState 定义任务状态、CronJob 表示一个完整的定时任务、CronStore 负责把任务内容持久化到本地 JSON 文件（<workspace>/cron/jobs.json）。

除了 Cron，Nanobot 还有第二种主动唤醒机制，即 Heartbeat 心跳。但这个”心跳”和传统网络保活的心跳不是一回事，它其实是一个定期唤醒代理去检查预定义任务的主动式服务。代码在 heartbeat/service.py，几个关键常量很能说明它的设计：

它默认每 30 分钟主动唤醒一次 Agent，让它去读工作区里的 HEARTBEAT.md 文件，检查里面有没有 ## Active Tasks 下要执行的任务。如果文件里没有可执行内容，Agent 直接回一句 HEARTBEAT_OK，不浪费 token 去做后续动作；如果有，就执行并把有用的结果发给最近活跃的聊天目标。

心跳和 Cron 看起来都在”定时唤醒”，但适用场景完全不同，对照着看最清楚：

特性 Heartbeat（心跳） Cron（定时任务）
触发源 文件内容（HEARTBEAT.md） 时间规则（Cron 表达式 / 间隔）
执行内容 不确定，取决于文件里写了什么 确定，Payload 里预定义的提示词
持久化 无需持久化，实时读文件 需持久化（jobs.json）
交互性 人机异步协作 机器自动化
成本控制 高，文件空就不唤醒 LLM 低，时间到必须唤醒 LLM 执行

另一个容易忽略的细节是心跳的任务执行期间计时器是暂停的：

它会停在 await self._tick() 这一行，等任务跑完才重新计 30 分钟。这避免了”上一次任务还没跑完，下一次又触发”的重复执行混乱。这恰恰是用心跳而非 Cron 来做”周期性巡检”的好处；如果硬用 Cron 做，任务一旦超过周期就会重叠。当然，你完全可以只用其中一种，但理解了它们的差异，才能在对的场景选对的工具。

frontmatter 各字段的含义：name（必填）是技能名（目录名）；description（必填）是技能描述，模型就是根据它判断何时使用这个技能的；metadata.nanobot.emoji 是图标；requires.bins 是依赖的可执行程序；requires.env 是依赖的环境变量；always 表示是否始终加载到上下文；install 是依赖未满足时的安装提示。frontmatter 之后是正文，用大量”想做什么 → 用什么命令”的示例告诉模型操作范式，模型据此举一反三。

把 3.6 节的摘要加载和这里的正文加载串起来，技能的完整加载流程是这样的。当用户说”列出某仓库的 issue”：第一步，build_system_prompt 调用 SkillsLoader.build_skills_summary() 扫描技能目录，读出每个技能的 name 和 description，把摘要塞进系统提示词；第二步，因为 GitHub 技能的 description 里提到了 issue，模型判断它能满足需求；第三步，通过 skills.py 的 load_skill 加载 SKILLS.md 的正文；第四步，照正文里的示例生成一条 gh issue list 命令；第五步，交给 shell.py 的 execute() 执行；第六步，命令返回 JSON 或文本结果，Agent 整理后反馈给用户。

第三节里 ReAct 循环每轮都要 await self.provider.chat(...) 调一次 LLM，这一节就看 Provider 这层抽象。Nanobot 本身不是 AI，它只负责把上下文组织好发给后端大模型。providers/base.py 定义了标准接口，用 Python 的 abc 模块实现抽象基类，用 dataclasses 的 @dataclass 来组织数据：

这里必须做一个重要纠正。早期的源码解读文章（写于 2026 年 2 月中旬）会说 Nanobot 用 litellm 库做统一 Provider 接口，对应的实现是 litellm_provider.py。这在当时是对的：litellm 是一个能把不同 LLM 提供商的 API 标准化成 OpenAI 格式的统一接口库，好处是不用为每个模型写单独的类，坏处是支持的模型受第三方限制。但这已经过时了：2026 年 3 月 21 日，项目把 litellm 移除，改用原生的 openai 和 anthropic SDK（官方 News 里有明确记录 “Replace litellm with native openai + anthropic SDKs”）。原因是 litellm 这类统一层虽然省事，但对厂商特有能力（如 Anthropic 的 prompt caching、OpenAI 的 reasoning）支持滞后。改成原生 SDK 后，既减少了依赖，又能更好地利用每家模型的新特性。

不过源码里还保留了一段判断 self.is_vllm = bool(api_base) and not self.is_openrouter，说明它本就支持本地模型和自建模型。想用本地模型（如 Ollama）很简单：配置一个 OpenAI 兼容的 Provider，把 apiBase 指向本地服务地址（如 http://localhost:11434/v1，这里 v1 后缀是 OpenAI 模式的历史习惯，与版本号无关），apiKey 随便填一个非空字符串即可。当前大多数托管 Provider 走 OpenAI 兼容实现，而 Anthropic、Azure、Bedrock、Codex、Copilot 有专门路径。

模型选择方面，当前推荐用**命名的 modelPresets**（预设）来管理。一个预设包含 label、provider、model、maxTokens、contextWindowTokens、temperature 等，由 agents.defaults.modelPreset 指定当前激活的，还支持 fallbackModels（备用模型，主模型挂了自动切）。这种”预设 + 兜底”的设计比早期直接配 provider/model 更适合生产环境。

十、通道集成：一个 Agent，十几个聊天平台

第三节里消息总线把渠道细节屏蔽了，这一节看渠道层自己怎么工作。通道（Channel）负责把外部平台翻译成 InboundMessage、把 OutboundMessage 翻译回平台 API。基类契约在 channels/base.py，生命周期和发现机制在 channels/manager.py，ChannelManager 负责根据配置初始化启用的通道、启停所有通道、把出站消息路由到正确通道。当前支持的平台覆盖面相当广：CLI、WebUI/WebSocket、Telegram、Discord、Slack、飞书、企业微信/微信、Email、Mattermost、DingTalk、QQ、Matrix、Microsoft Teams、Signal、WhatsApp 等。

十二、安全边界：让 Agent 自由但不失控

让一个能执行 Shell、能读写文件、能联网的 Agent 常驻后台，安全是不可回避的话题。Nanobot 把安全相关代码集中在 security/ 和若干工具文件里，划出了几条明确的边界。

第一条是工作区作用域（security/workspace_access.py、workspace_policy.py）：文件工具的读写被限制在工作区范围内，防止 Agent 随意碰工作区之外的文件。第二条是 Shell 沙箱（agent/tools/shell.py）：除了第四节讲的正则黑名单 deny_patterns，后续版本引入了更正式的沙箱机制和可配置的命令允许列表，让 Shell 执行更可控。第三条是 SSRF/网络检查（security/network.py、agent/tools/web.py）：Web 工具在抓取时会做 SSRF 防护，避免被诱导去访问内网地址（如云元数据服务 169.254.169.254、127.0.0.1、内网网段）。第四条是通道访问控制：对生产环境或共享聊天 App，要配置 allowFrom、配对（pairing）或 WebSocket token，防止陌生人通过 Telegram 把你的 Agent 当成免费算力或攻击跳板。

需要强调的是，这些边界是”层层设防”而不是”绝对安全”。前面提到的正则命令过滤的脆弱性就是一个例子，它挡得住误操作，挡不住有意绕过。所以 Nanobot 的安全设计哲学更接近”默认安全 + 可审计 + 可收紧”，而不是”绝对不可能出错”。在共享或生产环境部署时，务必要把通道访问控制和工作区限制配置好。

十三、配置与工作区：实例的两半

最后讲一个贯穿全局却容易被忽略的概念：配置（config）和工作区（workspace）的分离。一个 Nanobot 实例默认住在 ~/.nanobot/ 下，分成两半。~/.nanobot/config.json 是实例配置，声明这个实例”可以用什么”，包括 Provider、模型默认值、通道、工具、网关、API、运行时选项。~/.nanobot/workspace/ 是Agent 工作区，存放这个实例”为自己保留的状态”，包括记忆、会话、心跳任务、Cron 任务、技能、生成的产物。一句话，配置控制”能力边界”，工作区存放”自我认知与记忆”。

路径 含义
~/.nanobot/config.json 实例配置：providers、模型默认值、channels、tools、gateway、API、运行时

十五、小结：小核心的胜利

回到开头那个数字：4000 行核心代码，复现 43 万行 OpenClaw 的核心智能体能力。Nanobot 的意义不在于”功能比 OpenClaw 多”，恰恰相反，它的功能比 OpenClaw 少得多。它的真正价值在于证明了 Agent 的核心其实很小：一个 asyncio 编排的循环、一对入站/出站消息队列、一个 ReAct 内循环、几个可插拔的模块（通道、Provider、工具、技能、记忆），就足以支撑一个能跨平台对话、会用工具、有记忆、能主动唤醒的个人助手。

如果你记住了第三节那条”一条消息的旅程”，就等于抓住了它的全部主干：run() 等消息 → _process_message() 取会话、设上下文、build_messages() 拼提示词（身份 / 引导文件 / 记忆 / 技能）→ 进入 ReAct while 循环调 LLM → 有工具调用就交给 registry.execute() 的”查找/校验/执行/兜底”四步、结果塞回历史 → 直到模型给出最终答案 → 回复发回消息总线 → 渠道投递给用户。围绕这条主干，Nanobot 的设计原则可以浓缩成几条：面向通道和面向模型的循环要分开（AgentLoop 与 AgentRunner）；用消息总线解耦通道和 Agent；工具执行做异常兜底，让模型有自我修正的机会；记忆分短期/长期/Dream 三层；技能是给模型看的说明书、工具是干活的代码；Cron 管确定的时间规则、Heartbeat 管不确定的文件巡检；配置控制能力边界、工作区存放自我。

更宏观地看，Nanobot 是对”Vibe Coding”范式的一种反思。当 AI 生成的代码越堆越多、系统越来越难被人类理解时，一个”手工精制、小而可读”的实现就有了独特价值：它不只是工具，更是教材。读它的源码，你能看清一个真实 Agent 从”消息进来”到”回复出去”之间发生的每一件事，而没有被层层抽象和海量依赖淹没。这种”看得懂、改得动”的可控性，正是它官方那句”an AI agent you can truly own”的底气所在。
