智能体通信协议详解：MCP、A2A 与 ANP

2026-06-17- 2026-06-18
6k- 22m
课程笔记-学习AI Agent机器学习分享

本文主要整理自 DataWhale Hello-Agents 第十章，并结合 MCP 官方规范、A2A 协议说明 以及社区实践资料写成。目标不是背概念，而是把「Agent 怎么连外部世界、怎么连其它 Agent」这件事讲清楚。

引言：单体 Agent 跑通之后，真正难的是「连接」

如果你已经用 ReAct 写过带工具调用的 Agent，大概会有这样的体验：前几小时很兴奋，接下来几周都在和集成打架。每接一个 GitHub、数据库、文件系统或 Slack，你都要重新写认证、错误处理、参数 schema，还要适配 OpenAI、Claude、开源模型各自不同的 function call 格式。更麻烦的是，这些工具往往只能服务你当前这一个项目，别的团队写的 Tool 很难直接拿来用。

当任务进一步升级到「研究员 + 撰写员 + 编辑」这类多角色协作时，问题又从「连工具」变成「连 Agent」。很多人第一反应是写一个中央 orchestrator，所有消息都经过它转发。短期可行，长期通常会遇到三类麻烦：协调器本身成为单点故障；所有流量挤在一个节点上形成瓶颈；每新增一个 Agent，你都要改 orchestrator 的路由逻辑。

智能体通信协议想解决的，正是这类「连接层」的标准化问题。本文重点讨论三套协议：MCP 负责 Agent 与工具/资源，A2A 负责 Agent 与 Agent，ANP 负责更大规模网络中的发现与信任。它们不是互相替代，而是分层互补。

一、为什么需要协议，而不是继续手写 Tool？

在没有协议时，一个典型 Agent 项目的工具层往往长这样：你为 GitHub 写一个 search_repositories，为 PostgreSQL 写一个 run_query，为本地文件系统写一个 read_file。每个函数都要自己处理 HTTP、鉴权 token、超时重试、日志和异常包装。与此同时，你还要把同一套能力翻译成 OpenAI tools、Claude tools、Gemini tools 三种 schema。模型一换，适配层就要重写。

这种做法的问题并不在于「写几个函数麻烦」，而在于它无法形成可复用的生态。A 团队做的数据库工具，B 团队的 IDE 很难直接接入；社区里明明已经有成熟的 GitHub 集成，你仍可能在项目里再实现一遍。工具描述格式不统一，模型也很难在运行时做可靠的能力发现。

通信协议的价值，在于把「连接」变成公共基础设施。用互联网做类比：没有 TCP/IP 时，每台设备都要为彼此定制通信方式；有了标准协议后，应用层只需要关心业务逻辑。Agent 领域正在经历类似过程。MCP 试图统一「模型如何访问外部能力」，A2A 试图统一「Agent 之间如何委托任务」，ANP 则更关注「开放网络里如何发现与信任一个 Agent」。

二、三种协议分别解决什么层次的问题？

先把三者放在同一张图里理解。MCP 面向的是能力接入：我的 Agent 需要读文件、查库、调 API。A2A 面向的是协作过程：多个 Agent 如何围绕一个任务分工、同步状态、交换产物。ANP 面向的是网络治理：当 Agent 数量从 3 个涨到 300 个时，如何发现、路由、验证彼此。

维度 MCP A2A ANP
主要通信对象 工具、数据源、外部服务 其它 Agent 开放网络中的 Agent 服务
核心抽象 Tools / Resources / Prompts Task / Message / Artifact 服务描述、DID、发现与路由
拓扑 Host-Client-Server Client-Remote（P2P 协作） 去中心化服务发现
成熟度 生态最快，社区 Server 多 Google 主推，规范迭代快 偏早期，偏基础设施设想
典型问题 工具重复接入、模型耦合 多 Agent 编排复杂、协调器瓶颈 大规模发现、身份与信任

选择时不必纠结「哪个最先进」，而应问「当前瓶颈在哪一层」。如果你只是想让一个 coding agent 稳定读写仓库和数据库，先把 MCP 做好通常就够了。如果你在做「研究助手 + 写作助手 + 审稿助手」流水线，A2A 的任务抽象会更自然。如果你设想的是开放 Agent 市场，让陌生 Agent 在互联网上互相发现并调用，才需要认真看 ANP 或类似方案。

工程上比较常见的组合是：单体 Agent 用 MCP 接工具；跨 Agent 协作用 A2A；当 Agent 数量和来源继续增长，再引入 ANP 风格的服务发现与身份机制。

三、MCP：把「工具接入」做成标准层

3.1 MCP 到底是什么？

Model Context Protocol（MCP） 是 Anthropic 推动的开放协议，目标是在 LLM 应用与外部数据源、工具之间建立统一连接方式。根据 MCP 官方规范，它使用 JSON-RPC 2.0 作为消息格式，采用有状态连接，并支持客户端与服务端在初始化阶段进行能力协商（capability negotiation）。

这句话里其实有三个关键词。第一，JSON-RPC 意味着它不是某种模型厂商私有 SDK，而是可跨语言实现的协议。第二，有状态连接意味着 Client 和 Server 之间会维持会话，能处理订阅、通知、进度更新等连续交互，而不是一次性 HTTP 调用那么简单。第三，能力协商意味着连接建立时双方会声明自己支持哪些 primitives，后续只能使用已声明的能力，这为安全边界提供了基础。

很多人第一次接触 MCP 时会问：我已经有 Function Calling 了，为什么还要 MCP？这两个概念处在不同层次。Function Calling 是模型能力：模型根据上下文判断要不要调用函数、参数填什么。MCP 是连接协议：工具如何被发现、如何描述、如何执行、结果如何回传。可以把它类比成「会打电话」和「电话网标准」——前者是能力，后者是基础设施。

3.2 Host / Client / Server 架构

MCP 采用三层结构，这在 官方架构文档 中有明确说明。

Host 是用户直接交互的应用，例如 Claude Desktop、Cursor、某个自研 chat UI。Host 负责管理用户体验、权限同意、以及内部可能存在的多个 Client 实例。一个 Host 可以同时连接多个 MCP Server，比如文件系统、GitHub、数据库，但用户通常只面对一个统一界面。

Client 是 Host 内部负责「说 MCP 语言」的连接器。每个 Client 通常对应一个 Server 连接，负责协议握手、能力交换、消息路由、订阅通知等。官方文档特别强调，Client 需要维护 Server 之间的安全边界，避免一个 Server 越权访问另一个 Server 的上下文。

Server 才是具体能力的提供者。它可以是一个本地进程，也可以是一个远程服务。Server 通过 MCP primitives 暴露资源与工具，并在需要时通过 Client 提供的接口请求 sampling（让 Host 侧模型代为推理）或向用户发起补充信息请求（elicitation）。

用一个具体场景理解整条链路：用户问「我桌面上有哪些文档？」Host 把问题交给模型；模型判断需要访问文件系统；Host 内的 MCP Client 连接 filesystem server；Server 扫描目录并返回结果；模型再基于结果组织自然语言回答。开发者通常只需要实现 Server 或复用社区 Server，而不必重复实现 Host 和 Client。

3.3 三大 primitives：Tools、Resources、Prompts

MCP 的能力暴露围绕三个 primitive 展开，这在官方规范中被明确定义。

Tools 是模型可以执行的操作，通常带有副作用，例如创建文件、执行 SQL、发送消息。它们是可调用的函数，关键不只是 API 包装，而是其描述必须让 LLM 能可靠理解用途与参数约束。社区经验表明，工具描述写得是否清楚，往往比模型大小更能影响调用成功率。

Resources 是只读上下文，通常通过 URI 标识，例如 file:///docs/spec.md、postgres://db/schema。Resources 的设计目的是让模型获取信息，而不是直接执行操作。官方文档也提醒，Host 应控制哪些 URI 可被暴露，避免模型任意读取敏感数据；同时要考虑大文件分块、摘要和订阅更新，避免上下文爆炸。

Prompts 则是可复用的模板化工作流，可以带参数，也可以引用 tools 或 resources。它更像「标准化操作说明」，适合把常见任务固化下来，例如代码审查模板、日志分析模板。

三者关系可以这样记：Tools 负责「做事」，Resources 负责「读信息」，Prompts 负责「规定怎么做」。Hello-Agents 教程中强调，很多初学者会把所有东西都做成 Tool，其实大量只读上下文更适合 Resource。

3.4 一次完整的 MCP 调用是如何发生的？

从工程视角看，一次工具调用通常经历五个阶段。

首先是连接与能力协商。Client 与 Server 建立会话后，双方交换各自支持的能力，例如 Server 是否支持 tools、resources、prompts，是否支持资源订阅，Client 是否支持 sampling 等。只有被声明过的能力才能在后续使用。

然后是发现阶段。Client 调用 tools/list、resources/list 或 prompts/list 获取当前可用能力及 schema。注意，这不是一次性静态配置，而是运行时发现；Server 也可以根据权限动态返回不同工具集。

接着是上下文构建。Host/Client 会把工具描述注入模型上下文。不同框架实现方式不同，有的转成 OpenAI tools，有的转成 Claude tools，但底层信息来自 MCP Server 的统一描述。

再之后是模型决策与执行。模型基于用户问题和工具描述，决定是否调用某个 tool，并生成参数。Client 收到 tool call 后，通过 MCP Server 真正执行，并把结构化结果返回。

最后是结果整合。模型读取 tool output，再继续生成面向用户的回答。如果结果很大，Host 还可能做摘要、截断或分步追问。

这里有一个常被忽略的工程点：MCP 并没有替你做「模型一定会选对工具」这件事。它只是把工具接法标准化了。真正影响稳定性的是工具粒度、描述质量、权限边界和错误返回格式。

3.5 MCP 与 Function Calling：互补，不是替代

很多人以为上了 MCP 就不用 Function Calling，实际上几乎所有 MCP Host 仍然依赖模型的 tool use 能力。区别只是工具定义和执行不再绑死在某个模型 SDK 上。

以访问 GitHub 和本地文件系统为例。纯 Function Calling 方案下，你至少要维护多套 schema，自己实现 API 调用，并处理不同厂商响应结构差异：

3.6 传输层：为什么 MCP 能在本地和远程都跑？

MCP 的另一个重要特性是 Transport Agnostic。协议本身不绑定某种网络栈，因此同一套 Server 逻辑可以通过不同传输方式暴露。

本地开发最常见的是 Stdio，也就是把 MCP Server 作为子进程启动，通过标准输入输出通信。这种方式配置简单、延迟低，非常适合个人 IDE 插件或本地脚本式 Server。Hello-Agents 和很多教程里的示例都采用这种模式。

生产环境更常见的是 HTTP 或基于 HTTP 的流式传输。远程 Server 作为独立服务部署，多个 Host 通过网络连接。对于需要实时进度回传的场景，还可以使用 SSE 或 Streamable HTTP 一类更适合流式更新的传输方式。测试场景则常用 Memory Transport，在进程内直接调用，便于单元测试和快速原型。

选择传输方式时，不必追求「最先进」，而应看部署位置和交互形态。个人工具优先 Stdio；团队共享服务用 HTTP；需要长任务进度推送时考虑流式传输。

四、A2A：从「调工具」到「协作完成任务」

4.1 为什么 MCP 不够用了？

MCP 解决的是「我这个 Agent 能访问什么外部能力」。但当你有多个 Agent，每个 Agent 本身就是一个具备专长、状态和任务队列的服务时，问题就不再是简单的 function call，而是协作。

比如一个内容生产系统可能有：

  • 研究 Agent：负责检索资料、整理要点
  • 写作 Agent：根据要点生成初稿
  • 编辑 Agent：做结构和语言润色
  • 审核 Agent：检查事实与风格

它们之间传递的不只是「一个函数返回值」，而是任务状态、上下文历史、中间产物、补充提问。你用 MCP 也能硬做，但会把本属于协作层的语义硬塞进 tool schema，后期维护会非常痛苦。

A2A（Agent-to-Agent Protocol） 由 Google 推动，目标就是给这种协作提供标准语言。根据 A2A 规范，协议的核心单位不是 Tool，而是 Task。

4.2 Task：A2A 中最重要的抽象

在 A2A 里，Task 是一个可追踪、有状态、可异步的工作单元。客户端发起一个需要后台处理的请求后，远程 Agent 可能返回一个 Task 对象，而不是立即返回最终结果。随后 Task 会经历明确的生命周期。

根据 Google 开发者社区对 A2A 的解释 以及 Atlan 的协议梳理，常见状态包括：

  • submitted：任务已被接收
  • working：远程 Agent 正在处理
  • input-required：需要客户端补充信息
  • completed：任务成功结束，并附带 artifacts
  • failed：执行失败
  • canceled：客户端取消
  • rejected：远程 Agent 拒绝受理

这套状态机看起来比普通消息队列复杂，但它解决的是真实协作中的几个痛点。第一，长任务不必阻塞客户端，Agent 可以异步执行并通过事件汇报进度。第二，多轮协作可以挂在同一个 Task 上，而不是每次重新开对话。第三，失败、取消、拒绝都有明确语义，便于上层编排系统做恢复。

4.3 Message 与 Artifact 为什么要分开？

A2A 另一个关键设计，是把沟通过程和任务产物分开。

Message 用于 Agent 与客户端之间的交流，例如澄清需求、汇报进度、请求补充输入。它承载的是交互，而不是最终交付物。

Artifact 是任务执行过程中产生的输出，例如报告、表格、图片、结构化 JSON。规范明确要求：任务结果应通过 Artifact 返回，而不是塞进普通 Message。这样做的好处是，上层系统可以把 Artifact 当作版本化、可存储、可引用的工作产品，而 Message 只承担协作通信。

这个区分非常重要。很多早期 multi-agent demo 把一切都当聊天记录，导致后续无法做审计、版本管理、部分结果复用。A2A 的设计是在补这块工程短板。

4.4 Agent 如何被发现与调用？

A2A 并不假设所有 Agent 都是硬编码在配置文件里的。协议支持通过 Agent Card 暴露能力，通常位于 /.well-known/agent-card.json。客户端可以先读取这张「名片」，了解远程 Agent 能做什么、如何通信、支持哪些任务类型，再决定是否委托任务。

从请求流程看，A2A 大致包括：发现目标 Agent、完成身份验证、发送消息或任务请求、通过轮询或 SSE 接收状态更新。对于短任务，整个生命周期可以在一次同步响应中完成；对于长任务，则通过 message/stream 持续接收 TaskStatusUpdateEvent 和 TaskArtifactUpdateEvent。

transport 层面，A2A 使用 HTTP + SSE + JSON-RPC 2.0，而不是另起一套全新网络协议。这一点降低了 adoption 成本，也让它更容易融入现有后端体系。

4.5 为什么 A2A 倾向 P2P，而不是中心化编排？

Hello-Agents 教程里用星型拓扑对比网状拓扑，本质是在说「协调器模式」的扩展性问题。中心化 orchestrator 并非不能用，很多早期系统都从这里起步。但当 Agent 数量变多、角色变复杂后，所有消息都经过一个中心节点，会带来三个现实问题：协调器宕机导致全系统停摆；热门协调器成为性能瓶颈；每新增一种 Agent 角色都要改中心路由逻辑。

A2A 更鼓励 Agent 作为对等参与者直接通信、委托和回报结果。上层当然仍可以存在「项目经理 Agent」，但它本身也只是 A2A 网络中的一个节点，而不是协议层硬绑定的唯一中心。这样更符合真实组织协作：团队里有负责人，但成员之间也可以直接沟通。

五、ANP：当 Agent 网络大到需要「互联网层」

5.1 ANP 想解决的不是单次调用，而是网络扩张

MCP 解决工具接入，A2A 解决团队协作。可如果把场景再放大，问题会变成：互联网上有成百上千个彼此陌生的 Agent，它们来自不同组织、不同框架、不同安全域，你如何知道该找谁、能否信任、如何路由？

ANP（Agent Network Protocol） 目前仍偏早期，更像一套开放网络基础设施设想，但它提出的问题非常真实。Hello-Agents 教程将其概括为三个核心挑战：服务发现、智能路由、动态扩展。

服务发现问的是：来了一个新任务，网络里谁能做？智能路由问的是：如果多个 Agent 都能做，选哪个？依据可能是延迟、价格、成功率、负载。动态扩展问的是：新 Agent 上线后，如何被其他节点自动发现，而不是手工改配置。

5.2 DID、服务描述与 .well-known 端点

ANP 的一个核心思路，是结合 DID（Decentralized Identifier） 与标准化服务描述，建立去中心化信任与发现机制。典型流程如下：

  1. Agent 对外暴露标准描述端点，例如 .well-known/agent-descriptions
  2. 发现服务爬取或索引这些描述，形成能力目录
  3. 请求方根据任务语义检索候选 Agent
  4. 请求方使用私钥对含 DID 的消息签名
  5. 被调用方通过 DID 解析公钥并验证身份
  6. 双方按标准接口执行任务

这套机制想解决的是开放网络里最麻烦的两件事：一是身份与信任，不能靠中心化账号系统管所有 Agent；二是动态发现，不能让每个应用写死一份 Agent 名单。ANP 用 DID 解决前者，用标准描述协议解决后者。

当然，必须诚实地说：ANP 目前生态成熟度明显弱于 MCP 和 A2A。它更像方向正确的底层设想，真正落地时你仍需要处理大量工程细节，例如服务描述质量、声誉系统、计费、防刷、权限隔离等。

六、把三种协议放进同一个工程视角

学到这里，最重要的不是记住三个缩写，而是建立分层意识。

如果你的系统只有一个主 Agent，要接 GitHub、数据库、文件系统，那么 MCP 就是当下性价比最高的切入点。社区已经有 @modelcontextprotocol/server-github、server-filesystem 等 Server，IDE 和桌面应用也在快速接入 MCP Host 能力。

如果你的系统已经是多个专职 Agent 协作，比如研究、写作、审稿、发布，那么 A2A 提供的 Task / Artifact / 状态机抽象，会比「把别的 Agent 伪装成一个 tool」更清晰。你能明确知道一个任务进行到哪一步、失败在哪、产物是什么。

如果你设想的是开放 Agent 网络，让第三方 Agent 互相发现与调用，那么 ANP 代表的发现、身份、路由机制就必须提前设计。哪怕最终不完全照搬 ANP，实现里也很可能需要 Agent Card、能力描述、签名验证、服务目录等组件。

Hello-Agents 框架把三种协议统一抽象为 Tool，是为了降低学习门槛：

这种统一入口适合教学，但在真实架构里，你仍应清楚它们各自承担的层次不同，不要误把「能调用」当成「应该用同一种方式调用」。

七、落地时我目前最认可的几个经验

第一，优先复用成熟 MCP Server。尤其是文件系统、GitHub、Postgres 这类常见能力，社区实现通常比个人手写更完整，也更愿意持续跟进 API 变化。

第二，工具描述要当成产品文档来维护。名称、参数解释、边界条件、失败示例，这些都会直接影响模型调用质量。MCP 并没有自动替你解决「模型理解工具」的问题。

第三，多 Agent 不要一上来就中心化编排。先用 A2A 思路把任务状态和产物模型立住，再决定是否需要上层 orchestrator。

第四，协议层与业务层解耦。MCP Server 里只放通用能力，不要把某个具体业务流程硬编码进去；业务编排放在 Agent 或 workflow 层。

第五，安全边界默认最小授权。尤其是 Resources 和远程 Server，Host 必须决定哪些 URI、哪些工具、哪些 Agent 可以被暴露给模型。

八、小结

智能体通信协议不是锦上添花，而是 Agent 系统从 demo 走向工程化的分水岭。MCP 把工具接入标准化，A2A 把多 Agent 协作语义标准化，ANP 则试图为大规模开放网络补齐发现与信任机制。

如果只记住一句话，可以是：

MCP 让 Agent 会用手边的工具，A2A 让多个 Agent 会协作，ANP 让陌生 Agent 能在网络中被发现与信任。
