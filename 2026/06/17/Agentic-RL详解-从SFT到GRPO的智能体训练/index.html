Agentic-RL 详解：从 SFT 到 GRPO 的智能体训练

2026-06-17- 2026-06-18
5.8k- 21m
课程笔记-学习分享AI Agent强化学习机器学习

本文主要整理自 DataWhale Hello-Agents 的 Agentic-RL 章节，并结合 DeepSeekMath 论文、GRPO 相关解读 以及我此前写的强化学习笔记，尝试把「LLM 后训练」和「Agent 多步决策」放在同一条脉络里理解。

引言：会调用工具的 Agent，为什么还会「不会做」？

当你已经能搭 ReAct、接上 MCP、甚至做多 Agent 协作后，仍会遇到一类更棘手的问题：模型看起来会推理，但一换题型就崩；工具调用时好时坏，靠改 prompt 只能缓解，无法系统性提升；多步任务里，中间某一步错了，后面很难自我纠正。

这类问题的根源，往往不在「Agent 框架怎么写」，而在「策略本身没有被针对任务目标训练」。Prompt、RAG、记忆、工具协议，解决的是系统怎么连；而 Agentic-RL 关心的是策略怎么学——把 LLM 当作可学习策略，放进感知-决策-执行循环，通过奖励信号优化多步任务表现。

一、从监督学习到 Agentic-RL：问题为什么变难了？

1.1 SFT 很强，但天然有三个短板

监督微调（SFT）是后训练最常见的第一步。你给它高质量输入输出对，它就能学会某种回答风格、输出格式和基本解题套路。但对 Agent 任务来说，SFT 有三个很难回避的局限。

第一，数据质量就是能力上限。模型主要在模仿标注者给出的解法，很难稳定超越示范。标注者没走过的推理路径，模型通常也不会自己探索出来。

第二，缺乏主动探索。SFT 学的是「看到这种状态，输出那种动作」，而不是「在多种可能路径里试错并比较」。对于数学推理、工具调用、多步规划这类任务，只靠模仿往往不够。

第三，难优化长期目标。一个 Agent 任务是否成功，常常不取决于某一句回答是否漂亮，而取决于整条行动链是否最终达成目标。SFT 对中间步骤的约束很弱，即使最终答案错了，只要整体文本看起来像那么回事，损失也未必大。

考虑一个 GSM8K 风格问题：

这里的关键不是「把 LLM 硬叫成 RL」，而是看到 Agentic-RL 真正想优化的是什么：不是单句文本像不像训练集，而是整条决策过程是否导向正确结果。模型可以先采样多条不同推理路径，再根据最终正确性回传信号，逐渐学会哪些中间步骤更有价值。这正是监督学习不擅长、而 RL 特别擅长的地方。

二、LLM 训练全景：预训练、后训练，再到 Agentic-RL

要把 Agentic-RL 放对位置，必须先理解 LLM 是怎么一步步从「会续写」变成「会解题」的。

2.1 预训练：学会语言，不等于学会听话

预训练阶段的目标，是在海量无标注文本上学习语言规律和世界知识。最常见任务是 Causal Language Modeling，也就是根据上文预测下一个 token。这个阶段的数据规模通常是 TB 级，计算成本极高，产出的是「懂语言」的基座模型。

但预训练模型并不天然具备几个能力：遵循用户指令、以对话方式交互、拒绝不当请求、按指定格式输出答案。它更像一个读过很多书的语言天才，而不是一个可靠助手。因此几乎所有实用 LLM 都要进入后训练阶段。

2.2 后训练：从「会预测」到「会帮忙」

后训练的核心目标，是对齐人类偏好，让模型从续写机器变成可用助手。常见路径包括：

SFT（Supervised Fine-Tuning）：用高质量指令数据教模型如何回答、如何组织输出。

RLHF（Reinforcement Learning from Human Feedback）：让人类对多个回答排序，训练奖励模型，再用 PPO 等算法优化策略，使模型更符合人类偏好。

RLAIF（Reinforcement Learning from AI Feedback）：用强模型代替人类做偏好标注，显著降低成本。流程通常是：SFT 模型生成多个候选 → 强模型打分排序 → 训练奖励模型 → 用 RL 优化策略。经验上，RLAIF 在不少任务上可以接近 RLHF，但标注成本更低。

这一阶段解决的是「单轮回答是否像一个合格助手」。它非常重要，但对 Agent 任务仍不够，因为 Agent 面对的是多步决策，而不是一次说完。

2.3 从 PBRFT 到 Agentic-RL：优化对象变了

Hello-Agents 教程里把传统偏好强化微调称为 PBRFT（Preference-Based Reinforcement Fine-Tuning）。它关注的是：给定一个问题，模型给出一个回答，这个回答好不好。

Agentic-RL 关注的是另一件事：一个复杂目标能否通过多步行动完成。

比较两个场景会更直观。

在 PBRFT 场景里，用户问「请解释什么是强化学习」，模型一次性生成长回答，系统根据回答质量给分。任务边界清晰，状态几乎不变，奖励只看最终文本。

在 Agentic-RL 场景里，用户说「帮我分析某个 GitHub 仓库的代码质量」。Agent 可能需要：

  1. 调用 GitHub API 获取仓库结构，得到 +0.1
  2. 读取核心代码文件，得到 +0.1
  3. 分析代码质量，得到 +0.2
  4. 输出高质量报告，得到 +0.6

总奖励是过程与结果的累积，而不是某一句回答是否好听。此时状态会随着工具调用不断变化，动作空间也不只是生成文本，还包括调用工具、读取环境、更新上下文。

维度 PBRFT Agentic-RL
任务类型 单轮问答 多步序贯决策
状态 静态 prompt 动态演化状态
动作 文本生成 文本 + 工具 + 环境操作
奖励 单步质量 长期任务成功

这不是细节差异，而是问题定义变了。也正因为如此，DeepSeek-R1、OpenAI o 系列等推理模型背后，才需要把 RL 从「偏好对齐」进一步推进到「推理与行动策略优化」。

三、Agentic-RL 想提升的，不只是「答对题」

Hello-Agents 将 Agentic-RL 的目标概括为六项核心能力。它们彼此关联，但训练难点不同。

推理（Reasoning） 不只是会写 CoT，而是学会在复杂问题上选择有效推理路径。SFT 可以教格式，RL 更适合教「哪种拆法更稳」。

工具使用（Tool Use） 要求模型知道何时调用、调用哪个、参数怎么填、失败后如何改。纯 prompt 只能给规则，RL 可以让模型从成功/失败经验中学习策略。

记忆（Memory） 在 Agent 任务里不是简单 RAG，而是决定记什么、忘什么、何时检索。静态检索策略很难针对任务优化，RL 有机会学到任务相关的记忆管理。

规划（Planning） 指为长期目标制定行动序列。CoT 常见问题是线性推进、难以回溯；静态规划模板又难适应新环境。RL 允许模型通过试错发现「先收集信息再下结论」这类绕路但必要的策略。

自我改进（Self-Improvement） 意味着模型能识别自己的错误并调整策略，而不是每次从零犯错。RL 奖励长期表现，天然更适合塑造这种能力。

感知（Perception） 则把多模态理解纳入决策，例如结合图像、图表、文档做推理。

如果只把 Agentic-RL 理解成「用 RL 刷数学题」，会低估它的范围。数学题只是最容易验证、最适合入门的试验场。

四、数据与奖励：Agentic-RL 的两块地基

4.1 为什么 GSM8K 适合作为入门任务？

GSM8K（Grade School Math 8K） 是 Agentic-RL 教程里常用的入门数据集，原因很现实。

第一，答案可以自动判定。数学题有明确 ground truth，不需要昂贵的人工偏好标注，也不需要先训一个复杂奖励模型。对于第一次跑通 RL pipeline 来说，这至关重要。

第二，题目需要多步推理。很多题需要 2 到 8 步推导，天然适合训练「过程 + 结果」的策略，而不是只会背最终答案。

第三，推理能力可迁移。数学只是验证场，学会分步推导、检查中间结果、组织答案格式之后，这类能力可以部分迁移到其它结构化推理任务。

4.3 奖励函数决定模型到底学成什么样

奖励函数是 Agentic-RL 的灵魂。模型最终学到什么，很大程度上不是由网络结构决定，而是由「什么行为被奖励」决定。

Hello-Agents 教程里常见三类奖励。

准确率奖励（Accuracy Reward） 最直接：答案完全正确给高分，错误给低分或零分。优点是实现简单、目标明确；缺点是奖励稀疏，训练初期大量样本都得 0 分，策略梯度噪声很大。

长度惩罚（Length Penalty） 用来抑制废话。推理模型很容易学会「说很多看起来正确但其实冗余的话」，长度惩罚可以控制 token 成本。但如果惩罚过重，又可能压制必要的中间推理。

步骤奖励（Step Reward） 鼓励输出结构化步骤，例如显式写 Step 1、Step 2、Final Answer。这能提升可解释性，也方便调试；但模型也可能为了拿步骤分而故意拆碎推理，出现「步骤很多但逻辑很差」的奖励投机。

好的奖励函数通常有几个特征：成功标准清晰、能提供稳定梯度、方差可控、多个目标之间不互相打架。坏的奖励函数则常见几种问题：只有最后一步给奖励、中间过程全黑箱；存在明显漏洞，模型学会「作弊」拿高分；多个奖励项彼此冲突；方差过大导致训练不收敛。

五、SFT：Agentic-RL 的起跑线，而不是可有可无的前奏

5.1 为什么不能跳过 SFT 直接 RL？

很多初学者会想：既然 RL 更高级，能不能直接从基座模型做 GRPO？实践中这往往很难成功，因为模型连最基本的行为模式都没有。

一个没做过 SFT 的小模型，常见表现包括：不会按模板回答；不会写 Step 1 / Final Answer；输出格式混乱；同一 prompt 采样出的候选几乎全是垃圾。此时奖励几乎全为 0，RL 没有有效学习信号，训练看起来在跑，实际上策略不会变好。

SFT 的作用，是为 RL 提供一个「能开始探索」的策略起点。它教会模型：

  • 任务的基本输出格式
  • 初步的推理骨架
  • 对话模板和停止条件
  • 一个合理的探索空间

可以把它理解成：SFT 教会模型「怎么开始解题」，RL 再教会模型「怎样解得更好，甚至超越示范」。

5.2 LoRA：让消费级 GPU 也能做后训练

全量微调成本高得惊人。以 Qwen3-0.6B 为例，FP16 全量微调就可能需要约 12GB 显存；7B、13B 模型在消费级 GPU 上更难承受。对于个人学习者，LoRA（Low-Rank Adaptation） 几乎是默认选项。

LoRA 的核心思想是：微调时的权重更新矩阵往往是低秩的，因此不必更新全部参数，只需在原模型旁边插入两个小矩阵，训练这两个小矩阵即可。原模型权重保持冻结。

这带来几个直接好处。显存占用显著下降，训练速度更快，部署时可以保留基座模型并外挂 LoRA 权重，也方便做不同任务的 adapter 切换。代价是，效果通常略弱于全量微调，但在资源受限场景下性价比极高。

对 Agentic-RL 学习来说，这意味着你完全可以在单卡环境里先跑通 SFT + GRPO 小实验，而不必一上来就追求全参训练。

六、GRPO：为什么它比 PPO 更适合 LLM 推理训练？

SFT 之后，模型已经「像样地解题」了，但很多时候只是模仿训练集。要突破数据上限，需要 RL。问题在于：经典 PPO 对 LLM 来说太重了。

6.1 PPO 在 LLM 场景里的负担

PPO（Proximal Policy Optimization）是 RLHF 时代最常见的算法之一，但它依赖 critic / value model 来估计优势函数。对于 LLM 训练，这往往意味着你至少同时维护：

  • Policy Model
  • Reference Model
  • Value Model
  • Reward Model（或奖励函数）

其中 Value Model 通常和 Policy Model 一样大。也就是说，你本来只想优化一个 7B 模型，结果却要为它再养一个 7B critic。显存、通信、检查点管理都会变得复杂，训练稳定性也更难调。

6.2 GRPO 的核心思想：用「组内相对表现」代替 critic

GRPO（Group Relative Policy Optimization） 最早在 DeepSeekMath 论文 中提出，并在 DeepSeek-R1 等后续工作中进一步受到关注。它的关键改动是：去掉 value model，改为对同一 prompt 采样一组输出，在组内做相对比较。

直觉上，PPO 问的是：「这个回答绝对有多好？」
GRPO 问的是：「在同一道题的几种回答里，这个回答相对更好还是更差？」

根据 Cameron R. Wolfe 的 GRPO 解读 和 Yuge Shi 的 PPO/GRPO 对照文章，GRPO 的优势主要来自两点：

第一，不再需要 critic，显存和工程复杂度显著下降，常见估计是可节省约一半相关开销。

第二，组内相对比较更贴合奖励模型训练方式。奖励模型本来就是在「同一问题的多个回答」之间学偏好，GRPO 直接在策略优化阶段沿用这种相对比较逻辑，往往更自然。

DeepSeekMath 论文中，GRPO 对同一问题采样一组输出 {o1, o2, ..., oG}，根据组内奖励计算相对优势，再优化策略。论文还提到，实践中常对每个问题采样 64 个输出，并使用 KL 正则约束策略不要偏离 reference model 太远，以避免训练过程中模型「漂移」到奇怪分布。

6.3 GRPO 训练过程，逐步看发生了什么

可以把 GRPO 的一次更新理解成下面几步。

第一步，对同一个 prompt 采样一组候选回答，而不是只生成一个。组大小通常是 8、16、32、64 之类，越大越稳定，但计算成本也越高。

第二步，用奖励函数给每个候选打分。数学题里可能是最終答案是否正确，也可能叠加步骤奖励、长度惩罚。

第三步，在组内做归一化，计算每个候选的相对优势。表现优于组内平均的候选被鼓励，劣于平均的被抑制。这样即使所有候选都一般，也仍能产生相对梯度。

第四步，使用 clipped surrogate objective 更新策略，避免一次更新走太远。

第五步，加入 reference model 的 KL 约束，防止策略为了刷奖励而偏离 SFT 模型太远，出现语言质量崩坏或奖励投机。

DeepSeek-R1-Zero 的一个更有启发性的点是：在某些数学任务上，甚至可以不用神经网络奖励模型，而用规则化奖励，例如答案字符串匹配、格式检查。这大幅降低了训练门槛，也说明「可验证任务」对 Agentic-RL 有多重要。任务越能自动判分，RL 越容易落地。

七、训练完之后，别只盯着一个准确率

7.1 四类错误，比总分更重要

准确率当然要看，但它掩盖了太多信息。Hello-Agents 教程把错误大致分为四类，这个分类在实战中很有用。

计算错误：推理思路对，但算术做错，例如把 48/2 算成 25。这通常说明基础计算能力或数字敏感性不足。

推理错误：步骤顺序或逻辑关系错了，例如该先除后加却反过来。这指向策略问题，而不是单纯格式问题。

理解错误：没抓住题意，例如题目问「总共」却只算了一部分。这往往是语言理解或目标抽取失败。

格式错误：答案其实对，但没有按 Final Answer: 或 #### 格式输出，导致自动判分失败。这类问题在 RL 初期非常常见，说明 SFT 和奖励函数仍需加强。

7.2 真正有效的迭代，是「训练—分析—改奖励/改数据—再训练」

Agentic-RL 很少是一键成功。更真实的路径是：

  1. 准备数据并做 SFT
  2. 看模型是否学会基本格式
  3. 上 GRPO，观察奖励曲线是否稳定
  4. 抽样错误案例，判断是数据、奖励还是超参问题
  5. 调整奖励权重、采样数、学习率、KL 系数
  6. 再训一轮

如果训练过程中奖励突然暴涨但人工查看答案质量变差，要高度警惕奖励投机。比如模型学会输出很长但空洞的步骤，或者学会某种格式漏洞骗过正则匹配。此时不是「继续训就好」，而是要回头改奖励函数。

7.3 端到端 pipeline 长什么样？

一个完整的 Agentic-RL 实验通常包括：

  1. 数据准备：原始题集 → SFT 格式 / RL 格式
  2. SFT 训练：先用 LoRA 建立基线策略
  3. SFT 评估：看格式、初步准确率、推理可读性
  4. GRPO 训练：在可验证奖励下优化策略
  5. GRPO 评估：比较前后准确率、错误类型分布
  6. 部署与监控：上线后持续看真实失败样本

对个人学习来说，哪怕只是在 GSM8K 子集上跑通这条链路，也比单纯读概念更有价值。你会真切看到：SFT 决定了能不能开始学，奖励决定了往哪里学，GRPO 决定了学得更稳不稳。

八、和传统强化学习课程怎么对齐？

如果你学过 MDP、贝尔曼方程、蒙特卡洛、时序差分、策略梯度，那么 Agentic-RL 并不是另一套平行宇宙，而是把这些概念嵌进了 LLM 生成过程。

传统 RL 概念 Agentic-RL 中的对应
状态 (s) 题目、历史推理、工具返回、对话上下文
动作 (a) 下一个 token、tool call、环境操作
策略 (\pi(a \mid s)) 大模型参数化的条件分布 (\pi_\theta(a \mid s))：给定上下文 (s)，按概率逐 token（或 tool call）采样动作
奖励 (r) 正确性、步骤质量、长度惩罚等
回报 (G) 多步任务累积奖励
策略梯度 PPO / GRPO 中的优化目标

区别主要在于：动作空间不再是离散小集合，而是巨大词表；状态由上下文窗口承载；环境有时是可验证任务本身，有时是工具返回的外部世界。但「通过奖励塑造策略」这条主线没有变。

九、小结

Agentic-RL 不是给 LLM 再套一层花哨名词，而是明确承认：当任务变成多步决策后，光会做单轮回答已经不够，必须让策略在反馈中进化。

如果只记几句核心话，可以是：

  • SFT 解决「会不会开始做题」
  • RL（尤其是 GRPO） 解决「能不能做得更好」
  • 奖励函数 决定「到底学成什么样」
  • 错误分析 决定「下一轮该怎么改」

对个人学习者来说，最务实的路径通常不是一上来就追求最大模型，而是选一个可自动判分的任务（如 GSM8K），用 LoRA 跑通 SFT → 评估 → GRPO → 再评估 的最小闭环。闭环一旦跑通，你后面把任务换成工具调用、代码修复、检索问答，方法论都是可迁移的。
