智能体通信协议详解：MCP、A2A 与 ANP

2026-05-29- 2026-06-17
2.7k- 10m
课程笔记-学习AI Agent机器学习分享

本文基于 DataWhale Hello-Agents 第十章「智能体通信协议」整理与扩展，记录我在学习 Agent 工程化时，对 MCP / A2A / ANP 三层通信体系的理解。

前面几章我们已经在本地跑通了 ReAct、Plan-and-Solve、Reflection 等单体智能体范式。但当任务变复杂，问题很快从「模型会不会推理」变成「系统能不能连起来」：

  • 智能体如何稳定访问文件、数据库、GitHub、Slack？
  • 多个专业 Agent 如何像团队一样分工协作？
  • 成百上千个 Agent 如何在开放网络里被发现、被信任、被路由？

这就是智能体通信协议要解决的问题。目前业界比较主流的三套协议是：

  1. MCP（Model Context Protocol）：Agent ↔ 工具/资源
  2. A2A（Agent-to-Agent Protocol）：Agent ↔ Agent
  3. ANP（Agent Network Protocol）：大规模 Agent 网络基础设施

一、为什么单体 Agent 还不够？

一个功能完备的单体 Agent，通常已经具备：

  • 推理（ReAct / CoT）
  • 工具调用（Function Calling）
  • 记忆与 RAG
  • 上下文工程

但在真实工程里，你会很快撞上三类瓶颈：

1. 工具集成困境

每接入一个新服务（GitHub API、PostgreSQL、本地文件系统），往往都要写一套 Tool 适配器，处理：

  • 认证
  • 错误重试
  • 参数 schema
  • 不同模型提供商的 function call 格式差异

结果是：重复造轮子、难维护、难复用。

2. 能力扩展瓶颈

工具集通常是静态配置的。Agent 很难在运行时动态发现「网络里还有哪些能力可以用」。

3. 协作缺失

复杂任务往往需要「研究员 + 撰写员 + 编辑 + 审核员」协作。如果全靠你在代码里手写 orchestrator，系统会变成：

  • 强耦合
  • 难扩展
  • 单点故障

通信协议的价值，就是把「怎么连」这件事标准化，就像 TCP/IP 统一了设备互联方式。

二、三种协议怎么选？一张表先建立直觉

维度 MCP A2A ANP
核心问题 Agent 如何访问外部工具/资源 Agent 之间如何协作 大规模网络中如何发现与连接 Agent
通信对象 工具、数据源、服务能力 其他 Agent 开放网络中的 Agent 服务
拓扑结构 Host-Client-Server 点对点（P2P）/ 网状 去中心化服务发现
典型场景 读文件、查库、调 API 多 Agent 分工协作 Agent 互联网、开放生态
成熟度 生态最成熟 Google 推动，快速发展 概念框架，生态早期
设计哲学 上下文共享 对等通信 去中心化发现

选择建议（实战版）

  • 只需要增强单个 Agent 能力 → 优先 MCP
  • 需要多个 Agent 紧密协作 → A2A
  • 要构建开放、大规模 Agent 网络 → ANP（或 MCP + A2A + 自研注册中心）

目前工程上最稳妥的组合往往是：

MCP 负责接工具，A2A 负责 Agent 协作，ANP 负责更大规模的服务治理。

三、MCP：智能体与工具的「USB-C」

3.1 MCP 是什么？

MCP（Model Context Protocol） 由 Anthropic 提出，目标是标准化 Agent 与外部工具/资源之间的交互。

你可以把它理解成：

  • Function Calling = 模型「会不会打电话」
  • MCP = 电话系统的「统一通信标准」

两者不是竞争关系，而是上下层配合：

  • 上层：LLM 决定何时调用、传什么参数（Function Calling）
  • 下层：MCP 负责工具发现、连接、执行、返回结构化结果

3.2 三层架构

MCP 采用 Host → Client → Server 架构：

以「Claude Desktop 问：我桌面上有哪些文档？」为例：

  1. Host（宿主）：用户交互入口（如 Claude Desktop、IDE、Cursor）
  2. Client（客户端）：按 MCP 协议连接 Server，转发请求
  3. Server（服务端）：真正执行能力（文件系统、GitHub、数据库等）

完整链路：

这种分层的好处是关注点分离：

  • Host 专注体验
  • Client 专注协议
  • Server 专注业务能力

开发者通常只需要写 MCP Server，就能被多个 Host 复用。

3.3 MCP 的三大核心能力

能力 类型 作用
Tools 主动 执行操作，如读文件、发消息、查数据库
Resources 被动 提供上下文数据，如文档内容、配置、日志
Prompts 指导性 提供可复用提示模板，规范任务执行方式

关键区别：

  • Tools：会「做事」
  • Resources：会「给信息」
  • Prompts：会「给方法」

3.4 MCP 完整工作流程

  1. 工具发现：Client 调 list_tools() 获取工具名、描述、参数 schema
  2. 上下文构建：把工具描述注入 system prompt / tool schema
  3. 模型推理：LLM 决定是否调用、调用哪个
  4. 工具执行：Client 通过 Server 执行
  5. 结果整合：工具输出回到模型，生成最终回答

这里有个工程细节：工具描述质量直接决定调用成功率。描述含糊，模型就会乱调或不敢调。

3.5 MCP vs Function Calling：不是二选一

方式 A：纯 Function Calling（每个模型各写一套）

你还需要自己实现：

  • HTTP 请求
  • 鉴权
  • 错误处理
  • 不同模型响应解析逻辑

方式 B：MCP（连接社区 Server，模型无关）

优势：

  • 工具实现与模型解耦
  • 社区 Server 可直接复用
  • 换模型时工具层基本不用重写

3.6 MCP 传输层：五种模式怎么选？

MCP 是 Transport Agnostic（传输无关） 的，常见传输方式：

传输方式 典型场景
Memory 单元测试、快速原型
Stdio 本地开发、Python 脚本 Server
HTTP 生产部署、远程服务
SSE 流式输出、长连接
Streamable HTTP 需要双向流式通信的 HTTP 场景

工程建议：

  • 本地调试：Stdio
  • 团队内服务化：HTTP
  • 需要实时流：SSE / Streamable HTTP

四、A2A：智能体之间的「对等协作」

4.1 A2A 解决什么问题？

MCP 解决的是 Agent ↔ 工具。
A2A（Agent-to-Agent Protocol，Google 推动）解决的是 Agent ↔ Agent。

典型协作任务：

  • 研究 Agent 收集资料
  • 写作 Agent 生成初稿
  • 编辑 Agent 修改润色
  • 审核 Agent 做质量检查

它们需要：

  • 发消息
  • 委托任务
  • 同步状态
  • 交换中间产物（Artifact）

4.2 为什么不用中心化 Orchestrator？

传统「星型拓扑」协调器有三个痛点：

  1. 单点故障：协调器挂了，全系统瘫痪
  2. 性能瓶颈：所有消息都过中心节点
  3. 扩展困难：加一个新 Agent 往往要改 orchestrator 逻辑

A2A 采用 P2P / 网状拓扑，Agent 可以直接对话、委托、协商。

4.3 核心抽象：Task 与 Artifact

A2A 与 MCP 最大的概念差异：

协议 核心对象 关注点
MCP Tool / Resource 单次能力调用
A2A Task / Artifact 协作过程与产物管理

Task（任务） 有标准生命周期，例如：

  • 创建（Created）
  • 协商（Negotiating）
  • 代理/委派（Delegated）
  • 执行中（In Progress）
  • 完成（Completed）
  • 失败（Failed）

这让多 Agent 协作从「发消息」升级为「可管理的工作流」。

4.4 A2A 请求生命周期（四步）

  1. Agent 发现：找到能处理某类任务的 Agent
  2. 身份验证：建立可信通信
  3. 消息 API：发送任务请求、交换状态
  4. 消息流 API：支持流式协作过程

五、ANP：面向大规模 Agent 网络的「互联网层」

5.1 ANP 的定位

如果说：

  • MCP = Agent 的「USB 接口」
  • A2A = Agent 的「团队协作协议」

那么 ANP（Agent Network Protocol） 更像 Agent 的「互联网基础设施」。

它重点解决开放网络里的三个难题：

  1. 服务发现：新任务来了，找谁做？
  2. 智能路由：多个 Agent 都能做，选哪个？（负载、成本、时延）
  3. 动态扩展：新 Agent 加入后，如何被自动发现？

5.2 ANP 核心机制（简化理解）

典型流程：

  1. 服务发现与匹配

    • Agent A 通过发现服务按语义/能力查询 Agent B
    • 发现服务索引各 Agent 暴露的标准端点（如 .well-known/agent-descriptions）

  2. 基于 DID 的身份验证

    • Agent A 用私钥签名请求（携带 DID）
    • Agent B 通过 DID 解析公钥，验证身份与完整性

  3. 标准化服务执行

    • 双方按约定接口与数据格式交互（查询、预订、调用等）

核心思想：用 DID 建信任根 + 用标准描述协议做动态发现，从而无需中央协调器也能协作。

实战经验总结

  1. 优先复用社区 MCP Server，少写重复适配器
  2. 小团队协作用 A2A，别过早上复杂 orchestrator
  3. 开放生态再考虑 ANP，并提前设计服务描述与身份体系
  4. 工具描述要工程化维护，它直接影响模型调用成功率
  5. 协议层与业务层解耦，避免把业务逻辑写进协议适配代码

七、从「能跑」到「能上线」：我的理解

学到这里，我对 Agent 系统的分层更清晰了：

如果只做 demo，单体 ReAct + 几个本地 Tool 就够了。
如果要长期维护、可扩展、可协作，通信协议层几乎绕不过去。

九、小结

智能体通信协议不是「锦上添花」，而是 Agent 工程化的基础设施：

  • MCP 让 Agent 稳定连接外部世界（工具与资源）
  • A2A 让多个 Agent 像团队一样协作（任务与产物）
  • ANP 让大规模 Agent 网络可发现、可信任、可路由

下一步我打算继续把 Agentic-RL 和 评估基准（BFCL / GAIA） 串起来，形成「能连、能学、能评」的完整 Agent 学习路线。
